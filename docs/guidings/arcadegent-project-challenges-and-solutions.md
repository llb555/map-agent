# Arcadegent 项目难点与解决办法

这份文档专门整理 `Arcadegent` 在实现过程中遇到的核心难点，以及对应的解决思路。

适用场景：

- 项目复盘
- 面试中回答“项目里遇到过哪些难点”
- 写项目总结、周报、答辩材料

本文不是逐行代码说明，而是从工程问题的角度总结“为什么难、怎么解、最后学到了什么”。

---

## 1. 总览

这个项目表面上是一个“机厅搜索和导航系统”，但真正做起来，它同时碰到了几类典型难题：

- 大模型应用的 **Agent 编排问题**
- 前后端联动的 **流式状态同步问题**
- RAG 的 **检索工程化问题**
- 多 provider 的 **接口兼容问题**
- 地图与路线能力的 **外部依赖回退问题**
- 多工具接入的 **能力发现和治理问题**

如果要用一句话概括：

> 这个项目的难点不在单一算法，而在于把多轮对话、Agent、RAG、地图服务、前端流式展示和外部 provider 稳定地串成一个可用系统。

---

## 2. 难点一：Agent 不只是“调一次模型”

### 难点是什么

一开始最容易把项目想成：

- 用户发一句话
- 模型回一句话

但实际业务不是这样。

这个项目里，用户的问题可能是：

- 查某个城市的机厅
- 找某个地标附近的机厅
- 给一个店铺做路线规划
- 结合知识库回答补充问题

这些任务不适合都靠一个 prompt 硬做完，否则会出现：

- 意图理解和执行逻辑混在一起
- prompt 越堆越长
- 工具调用边界不清晰
- 后续扩展困难

### 为什么难

因为项目既要有“智能感”，又不能失控。

如果全靠一个 agent：

- 结构简单，但行为容易发散

如果全靠写死流程：

- 可控，但会失去大模型在复杂表达理解上的优势

### 解决办法

项目采用了 **主 Agent + worker + tool registry** 的结构：

- `main_agent` 负责理解用户意图和调度
- `search_worker` 负责机厅搜索相关任务
- `navigation_worker` 负责目标解析和路线规划
- `tool registry` 统一管理可调用工具

同时引入了 `working memory`：

- 保存搜索结果
- 保存路线结果
- 保存用户位置
- 保存知识检索命中
- 保存最近工具结果

这样就把“理解”“执行”“状态存储”拆开了。

### 最后的效果

- prompt 更聚焦
- worker 的职责更清晰
- 工具调用更容易管控
- 后续加新能力时不需要推翻整体结构

### 可以怎么讲给面试官

> 这个项目一个核心难点是不能把它做成“一个大 prompt 包打天下”。我最后采用的是 main agent + worker + tool registry 的结构，把意图理解、执行任务和工具调用拆开，同时用 working memory 串联中间状态，这样既保留了大模型的灵活性，也把系统行为控制在可维护范围内。

## 3. 难点二：如何让 Agent 执行过程对用户“可见”

### 难点是什么

如果系统只是等后端全跑完再一次性返回，用户会遇到两个问题：

- 看不到过程，只觉得慢
- 不知道系统到底卡在哪一步

对于普通 CRUD 页面问题不大，但对 Agent 系统来说体验会明显变差。

### 为什么难

因为 Agent 执行往往是多阶段的：

- 会话启动
- 子 agent 切换
- 工具调用
- 工具结果回填
- 最终回答生成

前端如果没有过程事件，只能盲等。

### 解决办法

项目没有采用“纯同步接口”，而是采用：

- `POST /api/chat/sessions` 先派发会话
- `GET /api/stream/{session_id}` 用 SSE 持续推送过程事件

后端会推送例如：

- `session.started`
- `subagent.changed`
- `tool.started`
- `tool.completed`
- `assistant.token`
- `assistant.completed`

前端再通过状态管理把这些过程展示出来。

### 最后的效果

- 用户知道系统正在做什么
- 前端可以边收边渲染
- 路线、地图、回答都能渐进展示
- 出问题时更容易排查

### 面试表达

> 另一个难点是 Agent 的执行过程比较长，如果做成同步返回，用户体验会很差。所以项目用了异步会话派发 + SSE 事件流，把 session started、tool 调用、token 输出和最终完成这些阶段实时推给前端，让 Agent 的执行过程变成用户可感知的产品能力。

## 4. 难点三：多轮对话里的上下文不能只靠聊天历史

### 难点是什么

多轮对话系统里，如果所有上下文都只靠自然语言历史传递，会有几个问题：

- 信息容易丢
- token 浪费
- 模型不稳定
- 工具结果难复用

比如第一轮查到 5 个机厅，第二轮说“帮我导航到第一个”，如果只是把上一轮回答文本拼回去，模型不一定稳。

### 为什么难

因为会话里真正重要的信息很多是结构化的：

- 店铺 id
- 地图坐标
- 路径信息
- 用户位置
- 知识命中结果

这些信息如果只放在自然语言里，后续推理会很脆弱。

### 解决办法

项目采用了两层方案：

1. `session_state` 持久化完整会话
2. `working_memory` 存放结构化中间结果

并且通过 `ContextBuilder` 把这些状态整理成模型更容易读的结构化上下文：

- `directory`
- `query`
- `search_catalog`
- `knowledge_hits`
- `shop_details`
- `route`

### 最后的效果

- 模型更容易抓住当前主轴
- 工具结果可以复用
- 多轮对话更稳定
- 上下文不会全靠自然语言“猜”

### 面试表达

> 多轮对话的难点不只是保留历史，而是保留“可计算、可复用的结构化状态”。这个项目通过 session state 和 working memory 保存搜索结果、路线、位置和知识命中，再由 ContextBuilder 按目录和细节块组织给模型，避免把所有信息都压成自然语言 history。

## 5. 难点四：RAG 从“能跑”到“工程可用”

### 难点是什么

RAG 一开始最容易做成：

- 文档切分
- embedding
- 内存里查一下

这样 demo 可以跑，但一旦想变成可持续使用的系统，就会遇到问题：

- 服务重启后索引丢失
- 知识库更新需要重建
- 检索链路缺少工程化形态

### 为什么难

因为 RAG 的难点不只是“能查到”，还包括：

- 索引怎么保存
- 元数据怎么管理
- 更新后怎么重建
- 前端怎么管理知识库

### 解决办法

项目做了两层处理：

1. 保留 `memory` backend，保证最轻量可跑
2. 新增 `FAISS` backend，用于本地向量索引持久化

核心思路是：

- 文档切分并向量化
- 将索引写入 `.faiss` 文件
- 将 metadata 写入 sidecar 文件
- 知识库上传 / 删除后重新构建索引

这让 RAG 从“内存演示版”向“可复用工程版”前进了一步。

### 最后的效果

- 重启后索引不会完全丢失
- 本地运行更稳定
- 面试里可以明确讲“做过 FAISS 向量索引”
- RAG 的工程含量更高

### 面试表达

> RAG 这块一个实际难点是从“纯内存 demo”演进到“可持续使用的系统”。为了解决索引重建和持久化问题，我引入了 FAISS 作为本地向量索引 backend，把 embedding 后的索引和 metadata 分开落盘，这样知识库检索就更接近真实工程系统。

## 6. 难点五：多 provider 兼容并不等于“都兼容 OpenAI”

### 难点是什么

理论上很多模型服务都说自己兼容 OpenAI 接口。  
实际上在这些方面经常有差异：

- tool calling 格式
- responses / chat completions 支持差异
- 多模态消息格式差异
- 是否支持 `image_url`

### 为什么难

因为如果只看“base_url + api_key + model”这几个参数，会误以为 provider 可以互换。  
但一旦涉及：

- 工具调用
- 图片输入
- reasoning 内容

兼容性问题就会暴露。

### 解决办法

项目引入了两层处理：

1. `ProviderAdapter`  
   负责兼容不同 provider 的消息格式、调用方式和解析方式

2. `ProviderRouter`  
   负责根据能力做请求分流，比如：
   - 纯文本请求走主 provider
   - 图片请求优先走视觉 provider

### 最后的效果

- 不同 provider 能更稳定地接入
- 文本链路和视觉链路不会互相污染
- 可以把 provider 差异吸收在应用层

### 面试表达

> 一个很真实的难点是，多 provider 虽然都说兼容 OpenAI，但实际在工具调用和多模态支持上差异很大。为了解决这个问题，项目里做了 ProviderAdapter 处理协议兼容，又做了 ProviderRouter 做能力分流，把差异尽量收敛在统一入口里。

## 7. 难点六：图片上传不等于模型真的“看到了图”

### 难点是什么

项目原先支持上传图片附件，但真正联调时发现：

- 前端能上传
- 后端能收到
- UI 也显示“图片已上传”

但模型其实并没有看到图片内容。

### 为什么难

因为“附件上传成功”和“多模态消息进入模型”是两件不同的事。

如果后端只是记录：

- 文件名
- mime type
- preview text

那模型实际上只拿到了“有一张图”，并没有看到图像内容。

### 解决办法

项目做了三步修复：

1. 让图片进入当前轮 user message，而不是只停留在附件元数据
2. 修改 provider 层，保留多模态消息结构，不再压成纯字符串
3. 增加 vision provider 分流和不支持场景下的友好降级

### 最后的效果

- 支持视觉的模型可以真正识别图片
- 不支持视觉的 provider 不再直接把底层错误甩给用户
- 应用层对多模态能力的处理更完整

### 面试表达

> 这个项目里一个很典型的坑是“上传图片成功”不等于“模型真的看到了图片”。我最后是把图片从附件元数据升级成真正进入模型消息的多模态内容，同时在 provider 层保留图片消息结构，并为不支持视觉的 provider 增加自动分流和友好降级。

---

## 8. 难点七：外部地图能力不稳定，系统必须能回退

### 难点是什么

路线规划和地理解析依赖外部地图服务时，问题通常不是“能不能调通”，而是：

- key 可能失效
- 配额可能耗尽
- MCP 服务可能变更
- 网络可能波动

如果完全依赖外部能力，一旦服务异常，系统核心链路就断了。

### 为什么难

地图服务看起来只是一个工具，但它实际上影响：

- 搜索附近
- 目标解析
- 路线规划
- 地图展示

属于高依赖能力。

### 解决办法

项目做了多层 fallback：

- 路线优先尝试高德 MCP 路线工具
- MCP 不可用时回退到内置 `route_plan_tool`
- REST 也失败时，再退化为离线直线距离和估算时间

同时在 `/health` 里暴露 MCP discovery 和路线工具状态，便于排查。

### 最后的效果

- 单点依赖失败不会直接让导航能力完全不可用
- 运维和调试更容易
- 面试里也能讲清楚“为什么做 fallback”

### 面试表达

> 地图能力这块的难点在于它是强依赖外部服务的，所以必须做回退设计。这个项目对路线规划做了 MCP -> REST -> 离线估算 的多级 fallback，这样即使高德某一层能力异常，系统仍然能给出一个可接受的结果，而不是直接失败。

## 9. 难点八：MCP 工具接入不是“配个地址就完了”

### 难点是什么

MCP 听起来像“接一个远程工具服务”，但真实问题包括：

- 工具 schema 会变
- 工具名称会变
- 某些 provider 的返回结构不稳定
- 路线工具不一定叫同一个名字

### 为什么难

如果本地系统把远端工具名写死：

- 对方一改名字，本地就挂
- 调试时很难知道到底是没发现工具，还是发现了但选错了

### 解决办法

项目对 MCP 做了：

- 启动时 discovery
- 工具投影为本地统一命名
- 支持手动指定 `route_tool_name`
- `/health` 输出当前发现到的工具状态

### 最后的效果

- 对第三方工具命名变化更有韧性
- 路由逻辑更容易调试
- 接第三方服务时更接近“可治理”而不是“硬耦合”

### 面试表达

> MCP 接入的难点不只是调用远程工具，而是对方的 schema 和命名并不总稳定。这个项目通过工具 discovery、本地命名投影、可配置 route tool name 和健康检查输出，把这种不稳定性控制在可观测、可调试的范围里。

## 10. 难点九：前后端状态一致性

### 难点是什么

聊天、地图、历史会话、流式过程展示这些能力混在一起时，前端很容易遇到状态问题：

- 用户已经切会话，但流还在回推旧消息
- 前端 optimistic update 和后端真实状态不一致
- 地图结果和文本回复不同步

### 为什么难

因为这不是一个普通表单页面，而是一个：

- 有异步派发
- 有 SSE 长连接
- 有历史恢复
- 有地图 artifacts

的复杂交互系统。

### 解决办法

项目前端通过：

- `useChatSessionController`
- `useStreamReply`
- `Zustand store`

统一管理：

- 当前 active session
- turns
- sending / awaiting assistant
- stream items
- active subagent
- map artifacts

这样状态更新有统一入口，不会完全散在组件里。

### 最后的效果

- 会话切换更稳定
- 流式回复和地图展示能同步推进
- 出错时更容易恢复

### 面试表达

> 前后端联动里的一个难点是状态一致性，因为这个项目不是简单请求返回，而是异步会话 + SSE + 地图 artifacts 的组合。前端最后通过统一的 controller hook 和 Zustand store 管理 session、stream、reply、map artifacts，尽量把状态更新收口，减少竞态和错位。

## 11. 最值得拿来讲的 3 个难点

如果面试时间有限，最推荐你优先讲这三个：

### 11.1 多 provider / 多模态兼容

因为这最像真实工作里的工程问题。

### 11.2 RAG 从内存检索演进到 FAISS 持久化

因为这能体现你不只是“会说 RAG”，而是做了检索工程化。

### 11.3 Agent 执行过程的 SSE 可视化

因为这能体现你考虑了系统体验，而不是只看模型输出。

## 12. 面试时的压缩版回答

如果面试官问：

> 你这个项目里遇到过哪些难点？

你可以直接这样答：

> 我觉得这个项目的难点主要有三类。  
> 第一类是 Agent 编排和状态管理，不能把所有任务都塞给一个 prompt，所以最后拆成了 main agent、worker 和 tool registry，并通过 working memory 保存结构化上下文。  
> 第二类是 RAG 工程化，早期内存检索能跑通，但不够稳定，所以后面引入了 FAISS 做本地向量索引持久化，让知识检索更接近真实系统。  
> 第三类是 provider 兼容和多模态输入，尤其是图片场景，最终通过多模态消息链路补齐、provider adapter 和 vision routing 才把问题闭环。  

---

## 13. 结论

这个项目真正难的地方，不是“会不会写接口”，也不是“会不会调模型”，而是：

- 怎么拆职责
- 怎么管状态
- 怎么做回退
- 怎么处理外部依赖不稳定
- 怎么把 demo 能力变成工程能力

这也是它最适合在面试里讲的价值所在。
