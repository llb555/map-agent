# Arcadegent 面试整合稿（基于最近一轮优化）

这份文档用于基于 `Arcadegent` 项目做面试表达、项目复盘和深度追问准备。

和已有题库相比，这份文档更强调最近一轮真实改动，包括：

- ReAct 运行时与主从 Agent 协作
- Query Rewrite 的工程化演进
- RAG 检索、分块、重排序、混合检索
- 缓存与持久化优化
- 会话状态与可恢复性设计

目标不是背源码，而是让你在面试里能把“为什么这样设计、怎么一步步演进、遇到什么问题、怎么权衡”讲清楚。

---

## 1. 30 秒项目介绍

Arcadegent 是一个面向音游机厅搜索、问答和路线建议的全栈 Agent 应用。前端用 React + TypeScript + Vite 负责聊天交互、地图展示和会话管理；后端用 FastAPI 承担 Agent 编排、工具调用、RAG 检索、地理解析和流式事件推送。项目的核心不是单纯“问一句答一句”，而是把搜索、地点理解、知识检索、路线规划和前端可视化串成一个完整闭环。

---

## 2. 2 分钟架构介绍

可以按下面顺序讲：

1. 这是一个前后端分离的全栈项目，前端负责 UI、地图和会话体验，后端负责 Agent、工具和数据能力。
2. 后端有一层组合根，统一装配仓储、RAG、工具注册表、地理服务、会话存储和 Agent runtime。
3. Agent 运行时采用 ReAct 风格，不是一次 prompt 出答案，而是主 Agent 调度 worker，worker 再调用搜索、知识检索、地理解析、路线规划等工具。
4. 对话过程通过 SSE 持续推送，前端可以看到 session started、worker started、tool completed、assistant token 这些过程事件。
5. 检索层既有结构化 DB 查询，也有 LangChain 风格的 RAG，后续又加了 query rewrite、hybrid search、reranker、query cache 和 snapshot 恢复。
6. 为了让系统更像真实工程，而不是 demo，我又补了缓存机制、RAG 持久化、会话单文件存储和 geo cache 批量刷盘这些优化。

## 3. 项目最值得讲的主线

如果面试官问“你觉得这个项目里你最有代表性的工作是什么”，建议优先讲下面四条主线。

### 3.1 把聊天做成可执行 Agent，而不是普通问答

可以这样讲：

这个项目一开始如果只做成“用户输入一句话，模型回一句话”，实现会很快，但无法稳定处理搜索、地点理解、路线规划、知识补充这些任务。所以我把它做成了 ReAct 风格的运行时：由主 Agent 负责理解意图和调度，由 search/navigation worker 执行具体任务，再通过统一工具层访问数据库、RAG 和地图服务。这样做的好处是职责更清晰、上下文污染更少、后续扩展新能力时也不用推翻整体结构。

### 3.2 把 Query Rewrite 做成可控的工程能力

可以这样讲：

用户真实输入会有大量口语、别名、简称和噪声，比如“魔都浦东哪里有舞萌”“人广附近有中二吗”。如果完全依赖模型临场理解，稳定性会比较差。所以我做的是一个可解释、规则化的 rewrite 层，而不是黑盒式 LLM 改写。它负责：

- 机种别名归一化
- 城市/区县 alias 识别
- 商圈/高校/地铁站 alias 归一化
- shop name 提取
- knowledge query 改写
- place query 提取并传给 location resolve

后面我又把 rewrite 结果写入 runtime memory，再注入结构化 context payload，让它不仅影响工具参数，还能被 prompt 显式消费和调试。

### 3.3 把 RAG 从 demo 提升到工程可用

可以这样讲：

RAG 最早很容易做成“文档切块 + embedding + 内存检索”，但这样只适合 demo。为了让它更可用，我做了几层演进：

- 支持普通字符切块和语义分块回退链路
- 支持 hybrid search，把向量检索和 BM25 结合
- 支持 reranker 做两阶段精排
- 支持 FAISS 持久化索引
- 后来又补了 query result cache 和 generic snapshot 恢复

这样 RAG 的能力就从“能查”升级成了“能持续运行、能恢复、能观察”。

### 3.4 把缓存和持久化做成可运营能力

可以这样讲：

后面我发现项目里真正影响体验和资源效率的，不只是模型效果，还有缓存和恢复机制。所以我又继续做了四类优化：

- RAG query cache，减少重复 embedding 和重复检索
- generic snapshot，memory backend 也能恢复 embedding 结果
- session store 从整文件 JSON 升级成单 session 单文件
- geo cache 改成延迟批量刷盘，减少频繁 I/O

再往后我又补了 cache hit/miss 统计、snapshot 模型签名校验，保证这些优化不是“黑盒提速”，而是可观测、可验证的。

## 4. ReAct 是怎么实现的

这是高频题，建议按“运行机制”来讲，不要陷入文件细节。

### 标准回答

这个项目实现的是一个工程化的 ReAct 运行时。它不是单次 prompt 生成最终答案，而是把一次会话拆成多个可观察的阶段：

1. 先根据用户输入推断意图，例如普通搜索、附近搜索、导航、知识补充。
2. 主 Agent 根据当前任务决定是否派发 worker。
3. worker 拿到更聚焦的 prompt 和 allowed tools，只做当前职责范围内的推理和工具调用。
4. 每次工具调用后，结果会写入 working memory，并进入结构化 context payload。
5. 模型可以基于“最新工具结果 + 当前 memory”继续下一步推理。
6. 最后由主 Agent 汇总 worker 产物，生成面向用户的最终回答。

### 为什么这算 ReAct

因为它符合 ReAct 的核心特征：

- 有 reasoning 和 action 的交替
- action 不是虚构的，而是真正调用工具
- tool result 会反过来影响下一轮推理
- 系统不是单轮答案生成，而是多步状态推进

### 面试追问：为什么不用一个大 prompt 直接做完

因为一个大 prompt 虽然简单，但工程上会有几个问题：

- 搜索、路线、知识问答的任务边界会混在一起
- 工具权限不好收敛
- prompt 越来越长，稳定性会下降
- 出错后很难定位是意图理解错、工具调用错还是结果整合错

用主 Agent + worker + tool registry 的结构之后，职责会更清晰，也更容易调试和扩展。

## 5. Query Rewrite 的演进过程

这是你这次最适合讲“持续优化能力”的部分。

### 第一阶段：基础改写

最开始做的是受控规则改写，主要解决：

- 噪声词清理
- 机种别名统一
- knowledge query 基础归一化

目标不是“改写得多聪明”，而是先把高频噪声去掉，降低工具参数误填概率。

### 第二阶段：区域与店名抽取

后面我发现很多查询本质上包含区域和店名约束，于是又加了：

- province/city/county hint 提取
- tighter 的 shop name 提取
- 搜索参数补全

这样用户哪怕没显式传结构化参数，runtime 也能从自然语言里推断出更稳定的工具输入。

### 第三阶段：地点 alias + place query

再往后，问题从“城市区县”扩展到了“商圈/高校/地铁站”。像：

- 人广
- 复旦
- 北大
- 徐家汇地铁站

这种表达不完全是区域，也不一定是店名，更像地标。所以我又新增了：

- place alias 词表
- place-only 不误判为 shop_name
- `place_query` 字段
- `location_resolve_tool` 自动优先消费 `place_query`

这样“口语地点 -> 地点归一化 -> 地点解析 -> 附近搜索”才真正闭环。

### 第四阶段：让 rewrite 进入结构化上下文

为了避免 rewrite 只存在 memory 里但 prompt 看不见，我又把它显式注入进结构化 `query` payload，分成：

- 当前真实 query 字段
- `rewrite_*` 辅助字段

这样模型能区分“这次实际执行用的参数”和“系统归一化得到的 hint”，避免误覆盖。

### 这个设计为什么适合面试讲

因为它体现了三个能力：

- 你不是只会写规则，而是会分阶段演进能力
- 你知道什么时候该做黑盒 LLM，什么时候该做可解释规则
- 你考虑了调试性、可观测性和下游消费方式

## 6. RAG 是怎么设计和优化的

### 6.1 基础设计

项目里的 RAG 是一个轻量但完整的检索服务，负责：

- 读取 md/txt/json/jsonl/pdf/docx/doc
- 提取文档内容和结构化 metadata
- 切块
- embedding
- 向量检索

### 6.2 为什么需要 hybrid search

纯向量检索擅长语义理解，但对专有名词、型号、ID、店名等精确匹配不够敏感。所以我又加了 BM25，做成 hybrid search：

- 向量分数反映语义相关性
- BM25 分数反映关键词精确匹配
- 两者做归一化后按 alpha 加权融合

这样在知识库规模扩大后，检索鲁棒性会更好。

### 6.3 为什么需要 reranker

向量检索更像“召回”，不能保证前几个结果排序一定最优。所以我又做了第二阶段 reranker：

- 第一阶段先召回更多候选
- 第二阶段用更精确的模型重新排序

这个设计本质上是“快召回 + 慢精排”的工业常见方案。

### 6.4 为什么语义分块不是默认对所有文档强开

这个项目里我判断语义分块是“按文档类型选择启用”更合理，而不是全量默认开启。

原因是：

- FAQ、攻略、长评论这种长文本，语义分块有明显收益
- 机厅名、地址、交通、区县这类结构化条目，固定分块反而更稳定

所以更合理的方向是“选择性启用”，而不是一刀切。

## 7. 缓存机制是怎么优化的

这部分是今天比较新的亮点，面试里很适合讲“性能工程和工程成熟度”。

### 7.1 RAG query cache

我给 RAG 查询增加了 `LRU + TTL` 风格的缓存，主要缓存：

- normalized query
- top_k
- source signature
- vector/hybrid/reranker 配置

目的很明确：减少重复 embedding、重复向量检索、重复 BM25 和重复 rerank。

### 7.2 cache hit/miss 统计

后来我又继续补了：

- `query_cache_hits`
- `query_cache_misses`
- `query_cache_hit_rate`

这样缓存不只是“加了”，而是能量化收益。

### 7.3 generic snapshot

之前只有 FAISS 模式能恢复持久化索引，memory backend 每次重启都要重新切块和 embedding。后来我补了 generic snapshot，让 memory backend 也能直接恢复 chunk + embedding。

### 7.4 snapshot 校验增强

为了避免恢复脏快照，我又加了这些校验：

- snapshot version
- embedding model signature
- chunk config
- source signature

意思是只要模型、切块配置或知识源有变化，就拒绝恢复旧快照，强制重建。

### 7.5 geo cache 和 session store

除了 RAG，我还做了两块更工程化的缓存/持久化：

- geo cache 从“频繁立即刷盘”改成“脏标记 + 延迟/批量 flush”
- session store 从“所有 session 写一个 JSON”升级成“一个 session 一个文件”

这样做的好处是：

- 减少写放大
- 降低并发下的整文件重写风险
- 单 session 读写和删除更自然
- 崩溃恢复粒度更细

## 9. 高频追问与建议回答

### 9.1 你为什么没有直接用 LLM 做 query rewrite

建议回答：

因为这个场景里 rewrite 更像“高频、低风险、可解释的参数归一化”，而不是开放生成。用规则做的好处是：

- 稳定
- 容易调试
- 成本低
- 不容易引入额外 hallucination

我保留了 LLM 在主推理链路里的灵活性，但把 rewrite 做成更受控的工程层。

### 9.2 你怎么判断一项优化有没有必要

建议回答：

我一般看三个维度：

- 它是不是当前链路里的高频热点
- 它是不是系统性瓶颈，而不是偶发问题
- 它能不能以比较低的复杂度换来可观收益

比如 query cache、session 单文件持久化、geo cache 批量刷盘，就都符合这个判断。

### 9.3 这项目如果继续做，下一个最值得推进的点是什么

建议回答：

我会优先做两类事情：

1. 可观测性
   例如把 cache hit rate、snapshot 恢复状态、rewrite 命中情况挂到健康检查或后台页面里。
2. 配置化和可维护性
   例如把地点 alias、机种 alias 从代码抽成配置，方便长期维护。

### 9.4 你为什么没直接上向量数据库服务，而是先做 FAISS + snapshot

建议回答：

因为这个项目当前的目标是把 RAG 做到“本地可运行 + 工程可恢复”，而不是先引入重基础设施。FAISS 更适合当前阶段：

- 接入轻
- 单机快
- 落盘简单
- 本地调试成本低

而 snapshot 进一步解决了 memory backend 每次重启都要重建 embedding 的问题。这条演进路径更符合项目当前规模。

### 9.5 如果面试官问：query cache 为什么不直接上 Redis

建议回答：

因为当前缓存对象主要是 RAG 查询结果，而且作用范围在单实例服务内部，优先目标是减少重复 embedding 和重复检索成本。这个阶段先做进程内 `LRU + TTL` 更合适，复杂度低、收益直接。只有在后面出现多实例部署、缓存共享或跨进程一致性需求时，Redis 这类外部缓存才更有必要。

### 9.6 你怎么保证 cache 不会返回过期错误结果

建议回答：

我不是只用 query 本身做 key，而是把 `source_signature`、backend 配置、top_k、hybrid/reranker 等信息一起纳入 key 或校验条件。这样只要知识源变了、模型变了、切块变了、配置变了，旧缓存或旧 snapshot 就不会继续复用。

### 9.7 session 单文件模式的好处除了性能还有什么

建议回答：

除了写放大更小，还有两个很实际的好处：

- 单个 session 的损坏不会拖垮全部会话
- 后续如果要做按 session 生命周期清理、归档、迁移，会更自然

所以它不仅是性能优化，也是恢复性和可运维性优化。

### 9.8 如果面试官问：这些优化有业务价值吗，还是只是技术自嗨

建议回答：

有明确业务价值。比如：

- query rewrite 提升用户自然语言输入的命中率
- location resolve 补参降低附近搜索失败率
- RAG query cache 降低重复查询延迟
- session 单文件提高会话恢复稳定性
- geo cache 批量 flush 降低热点路径的 I/O 开销

这些都会反映到检索成功率、响应速度和会话稳定性上，不是纯内部优化。

### 9.9 如果面试官问：这轮改动里你做过哪些取舍

建议回答：

我做过几个明确取舍：

- Query Rewrite 选择规则优先，而不是一开始就上 LLM 或分类模型
- 语义分块选择“按文档类型启用”，而不是一刀切
- cache 先做进程内缓存，而不是直接引入外部缓存系统
- session 持久化先做单文件目录化，而不是直接上数据库

这些选择的共同点是：优先用最小复杂度解决当前最明显的问题。

### 9.10 如果面试官问：你怎么证明你不是只会“修修补补”

建议回答：

我会强调这不是零散 patch，而是围绕一条完整链路做系统性演进。比如 Query Rewrite 从 alias 识别一路演进到 runtime memory、structured context、location resolve；RAG 从 basic retrieval 演进到 hybrid、reranker、cache、snapshot、签名校验；session store 从能存演进到更合理的存储组织。这个过程体现的是架构延展能力，而不是碎片化修补。

## 11. 更具体的问题库

这一节补的是更贴近一二面现场的问法，你可以把它理解成“加强版追问题库”。

### 11.1 请你讲一个你最近做的、最能体现工程判断的优化

建议回答：

我会讲 session store 从整文件 JSON 改成单 session 单文件这件事。因为它看起来不是最显眼的功能，但很能体现工程判断。表面问题是持久化，实质上涉及写放大、恢复粒度、兼容历史数据和后续扩展性。我最后不是简单改存储结构，而是同时保留了旧聚合格式读取兼容，这样升级不会把已有会话打断。

### 11.2 你最近做的优化里，哪个最像“性能优化”

建议回答：

最典型的是 RAG query cache 和 geo cache 批量刷盘。前者减少重复 embedding、重复检索和 rerank，后者减少热点批量路径上的频繁磁盘写入。这两者一个优化 CPU/网络/模型调用成本，一个优化 I/O 成本，都比较典型。

### 11.3 你最近做的优化里，哪个最像“稳定性优化”

建议回答：

我会讲 snapshot 签名校验和 session 单文件持久化。因为它们都不是直接让结果更快，而是减少“悄悄出错”的概率。尤其是 snapshot 校验，如果不做模型签名和 chunk config 校验，很容易恢复一个看似可用、实际不匹配的旧索引，这种问题线上最难排查。

### 11.4 你最近做的优化里，哪个最像“可维护性优化”

建议回答：

Query Rewrite 的 structured context 注入最像可维护性优化。因为它把 rewrite 从“某个工具调用前临时生效”提升成了运行时显式状态，后面无论是 prompt 消费、调试还是继续加规则，都有清晰落点。

### 11.5 面试官如果问：为什么你总在做“系统边缘层”的事情

建议回答：

因为很多真实系统问题都发生在边界层。模型、数据库、地图服务、会话存储、前端流式展示之间的接口缝隙，往往比单个模块内部更容易出问题。把这些边界做好，系统整体体验才会稳定。

### 11.6 面试官如果问：你怎么定义“今天这轮改动已经完成”

建议回答：

我的标准一般有三个：

1. 主功能链路已经闭环  
   比如 rewrite 不只是能识别 alias，而是能真正影响 location resolve 或 db query。

2. 关键行为有测试覆盖  
   不只是手工试一下，而是把预期行为锁进单测或 integration。

3. 新增能力是可观察的  
   比如 cache 统计、snapshot 校验、session 重启恢复结果都能被验证。

### 11.7 面试官如果问：你在这轮改动里最难的一点是什么

建议回答：

最难的不是实现某一段代码，而是保证一项改动不会破坏上下游。比如 Query Rewrite 不是一个独立模块，它会影响 db query、knowledge search、location resolve、prompt 消费和 context builder。所以每次看起来是小改动，实际上都要考虑整条链路。

### 11.8 面试官如果问：这轮改动里你最满意的一点是什么

建议回答：

我最满意的是把“今天加了什么功能”变成了“系统工程能力往前走了一步”。比如 query rewrite、cache、snapshot、session store 这些点单独看都不算炫技，但合起来之后，系统的可恢复性、可观测性和长期可维护性都明显提升了。

### 11.9 面试官如果问：如果让你重做一次，你会先做什么

建议回答：

如果重做一次，我会更早地把几个基础抽象定下来：

- query rewrite 的结构化输出
- RAG backend 与 snapshot 的边界
- session store 的存储抽象
- cache metrics 的统一出口

因为这些东西后期补虽然也能做，但如果更早有这些边界，后续迭代会更顺。

### 11.10 面试官如果问：你从这轮改动里学到了什么

建议回答：

我最大的体会是，一个系统从 demo 走向工程化，最重要的不是突然换一套大架构，而是不断把高频问题沉淀成稳定能力。今天这轮的 rewrite、cache、snapshot、session store，其实都属于这种“把一次次遇到的问题变成系统能力”的过程。

## 12. 二面深挖题

这一节不是给你背标准答案，而是帮你应对“面试官继续往下挖”的情况。建议你把每题的核心立场记住，现场再按语气展开。

### 12.1 你这个项目为什么没有做长期记忆

建议回答：

这个项目当前更偏任务型 Agent，而不是陪伴型或超长周期 persona 产品，所以我优先做的是 working memory，而不是长期记忆。因为这个场景里最关键的是把当前轮次的搜索条件、地点约束、工具结果和路线信息保留下来，支持本轮和短期多轮推理。长期记忆当然可以做，比如记住用户常去区域、偏好机种、常问问题，但那已经不是“把当前任务做对”这个阶段的第一优先级了。

如果面试官继续问“是不是能力不够所以没做”，可以补一句：

不是做不了，而是我有意把优先级放在任务闭环、检索质量和状态恢复上。长期记忆引入之后，马上会带来记忆抽取、冲突合并、过期策略、隐私边界这些新问题，所以要看业务价值再推进。

### 12.2 你怎么理解这个项目里的 memory

建议回答：

我把 memory 理解成运行时工作记忆，不是一个无限堆积的聊天历史仓库。它更像当前任务的状态容器，主要保存：

- 用户当前目标
- rewrite 后的查询线索
- 最近几次工具 observation
- 检索出的关键结果
- 路线或地点解析的中间状态

这样做的重点是让下一步推理有依据，而不是让模型背所有历史对话。

### 12.3 为什么 tool 要尽量确定性，agent 才负责生成

建议回答：

这是一个很典型的工程边界问题。tool 的职责应该是查、算、取、格式化，也就是确定性执行；agent 的职责才是理解上下文、做推理、组织自然语言。如果把 LLM 再塞进 tool 里，短期看像是更聪明，长期看会让链路很难调试，因为你分不清到底是工具错了、提示词错了，还是模型在工具内部又发散了。所以我更倾向于让 tool 保持可预测，让 agent 消费结构化结果去生成回答。

### 12.4 为什么 context payload 要做成结构化块，而不是把一堆 JSON 平铺给模型

建议回答：

因为模型也有注意力竞争问题。如果把搜索结果、店铺详情、路线、query hint 全平铺在一起，模型容易抓不住主轴。我这里更偏向“目录块 + 细节块”的组织方式，先告诉模型当前问题的焦点是什么、有哪些信息块可读，再把重字段放到 detail block 里。这样不只是为了美观，而是为了减少 prompt 歧义，让模型先回答主问题，再按需下钻细节。

### 12.5 这个项目为什么需要动态工具注册表

建议回答：

因为一旦 Agent 能力变多，工具管理很快会变成维护负担。动态工具注册表的价值，不只是“看起来更插件化”，而是把 schema、executor、provider 的职责拆开，让新增工具时不需要持续回到中心大文件里登记。这样做的直接收益是：

- 新工具更局部自描述
- 权限和可用工具更容易按场景收敛
- 后续做 bundle、插件化或多 provider 扩展时更顺

如果面试官追问“这是不是过度设计”，你可以回答：

如果项目只有两三个工具，的确没必要。但这个项目已经涉及搜索、地理解析、路线、RAG、摘要和 worker 选择，工具数量上来以后，注册方式本身就是工程问题。

### 12.6 你怎么理解主 Agent 和 worker 的边界

建议回答：

主 Agent 更像 orchestrator，负责理解任务、判断该走哪条链路、什么时候派发子任务、最后怎么汇总。worker 更像领域执行者，只拿更聚焦的上下文和更有限的工具去完成当前职责。这样拆分的好处是两个：第一，减少一个大 prompt 里所有任务互相打架；第二，tool 权限可以按 worker 收紧，稳定性会更好。

### 12.7 如果这个系统要支持多实例部署，你会先改哪几块

建议回答：

我会优先改三块：

1. 把进程内缓存逐步外移  
   例如 query cache、geo cache 从单实例内存转向 Redis 这类共享缓存。

2. 把 session store 从本地文件演进成外部存储  
   例如对象存储、KV 或数据库，这样实例切换时不会丢状态。

3. 补可观测性和实例无关的恢复机制  
   包括 cache 命中率、snapshot 恢复状态、tool error rate、SSE 中断率。

我不会一开始就全量推翻，而是先把“单机内隐状态”一块块外移。

### 12.8 现在的缓存如果以后升级成分布式缓存，最大的变化是什么

建议回答：

最大的变化不是把 `dict` 换成 Redis，而是要重新定义缓存一致性和失效策略。单机缓存主要关注命中率和延迟，分布式缓存还要考虑：

- 多实例之间是否共享同一份 key 设计
- 数据源变更后的广播失效
- 热点 key 的并发竞争
- 缓存穿透和雪崩

也就是说，缓存一旦跨实例，就从性能问题变成了系统一致性问题。

### 12.9 你怎么保证 session 持久化的一致性

建议回答：

当前阶段我的重点是“单 session 读写边界清晰、损坏隔离更细、恢复粒度更小”。如果面试官问强一致，我会坦诚说本地文件模式不是分布式强一致方案，它更适合单实例或轻量部署阶段。它的优势是实现简单、易恢复、易调试；如果业务再往上走，就要逐步过渡到外部存储和更清晰的写入协议。

### 12.10 这个项目的失败降级策略是什么

建议回答：

我会从链路角度回答。比如：

- rewrite 没命中时，回退原 query，不让系统直接失效
- location resolve 失败时，保留普通搜索或知识回答能力
- RAG snapshot 校验失败时，拒绝恢复旧快照并触发重建
- 某个 worker 或工具失败时，主 Agent 至少要给出可解释反馈，而不是无响应

也就是说，降级目标不是“每条链路都成功”，而是“局部失败不要拖垮整轮对话”。

### 12.11 你怎么做这个项目的可观测性

建议回答：

我会分三层讲：

1. 运行时事件层  
   比如 session started、worker started、tool completed、assistant token 这些流式事件。

2. 指标层  
   比如 query cache hit rate、snapshot 命中/重建次数、tool 调用失败率、location resolve 命中率。

3. 调试上下文层  
   比如 rewrite 结果显式进入 runtime memory 和 structured payload，方便定位“系统到底理解成了什么”。

面试里这样回答会显得你不是只盯功能，而是在想线上排障。

### 12.12 你会怎么评估 Query Rewrite 做得好不好

建议回答：

我会先做轻量评估，而不是一上来做很重的 ML eval。比如整理一批真实 query，覆盖机种别名、地区简称、商圈/高校/地铁站、店名混合表达几类样本，然后看：

- rewrite 后是否更接近目标参数
- location resolve 命中率是否提高
- 附近搜索失败率是否下降
- 最终用户回答是否更贴近意图

因为 rewrite 的价值不在“改写文本看起来多漂亮”，而在它有没有改善后续链路。

### 12.13 你会怎么评估 RAG 做得好不好

建议回答：

我会分成离线和在线两个层面。离线看召回质量，比如相关 chunk 是否进入 top-k、hybrid 和 reranker 是否真的改善排序；在线看链路效果，比如知识问答的可用率、重复问题的响应延迟、snapshot 恢复后的冷启动时间。因为 RAG 不是一个独立算法组件，它最终还是服务回答质量和系统效率。

### 12.14 为什么你说这项目不是模板 Agent

建议回答：

因为最终回答不是提前写死的模板拼接，而是 agent 基于结构化 context、skills 和 tool observation 去生成。模板在这里最多只负责确定性格式化，不负责主推理链路。如果系统的核心回答依赖一堆 if/else 模板，那它更像流程机，不像真正的 Agent。

### 12.15 如果面试官问：你这套 rewrite 会不会过拟合某个城市

建议回答：

这是个很好的追问。我现在的做法确实是从高频真实场景出发，比如城市、区县、商圈、高校、地铁站这些 alias，但抽象层面不是只服务某一个城市，而是“地点别名归一化 + place query 提取 + location resolve 消费”这条链路。也就是说，词表内容是城市相关的，但能力边界本身是可迁移的。后面如果扩城市，主要扩的是词表和数据，而不是推翻整套机制。

## 13. 高频 20 问速记版

这一节更适合临场前 10 分钟快速过一遍。尽量按口语背，不用追求逐字一致。

### 13.1 这个项目最核心的技术点是什么

建议回答：

我觉得核心不是某一个算法，而是把搜索、地点理解、RAG 和路线规划放进一个 ReAct 风格的 Agent runtime 里，让系统能多步调用工具、持续累积状态、最后再统一回答。

### 13.2 你在里面最有代表性的工作是什么

建议回答：

我做得最有代表性的是把这个系统从“能跑”继续推到“更工程化”，尤其是 Query Rewrite、RAG cache/snapshot、session 持久化这些连续优化。

### 13.3 为什么要做 ReAct

建议回答：

因为这个场景不是纯聊天，它天然需要先理解任务，再查数据、解地点、算路线，最后整合结果。单轮大 prompt 很难稳定完成这么多动作。

### 13.4 为什么不用一个大模型一次性回答

建议回答：

可以做，但稳定性和可调试性会差很多。你很难知道它到底是理解错了、调用错了，还是整合错了。拆成 Agent + tool 之后，问题更容易定位。

### 13.5 为什么 Query Rewrite 很重要

建议回答：

因为真实用户不会按数据库字段说话，他们会说简称、口语、别名、模糊地点。Rewrite 的价值就是把这些自然语言转成后端更容易消费的稳定信号。

### 13.6 为什么 Rewrite 先用规则

建议回答：

因为这是高频、低风险、讲究稳定性的链路。规则更便宜、更可控，也更方便调试。LLM 还是留给真正需要生成和推理的地方。

### 13.7 为什么还要把 rewrite 结果写进 memory

建议回答：

因为 rewrite 如果只在某个工具调用前生效，后面的 agent 和 prompt 看不见，调试也看不见。写进 memory 和结构化 payload 之后，整条链路都能消费。

### 13.8 这个项目实现记忆了吗

建议回答：

实现的是运行时工作记忆，不是长期记忆。重点是保留当前任务的状态、工具结果和约束，支持多步推理。

### 13.9 为什么没有长期记忆

建议回答：

因为当前阶段优先级更高的是把当轮任务做对。长期记忆会引入提取、更新、冲突、隐私等额外问题，要看业务价值再上。

### 13.10 RAG 为什么不是只做向量检索

建议回答：

因为纯向量检索对专有名词和精确词不够敏感，所以我又加了 BM25 做 hybrid，再加 reranker 做二阶段排序。

### 13.11 为什么要做 cache

建议回答：

因为很多查询会重复出现，尤其是热门问题或相近问题。cache 可以直接减少重复 embedding、重复检索和 rerank，收益很直观。

### 13.12 为什么还要做 snapshot

建议回答：

因为只靠内存索引的话，服务一重启就得重建 embedding。snapshot 让恢复更快，也更像真实工程服务。

### 13.13 为什么 snapshot 要做模型签名校验

建议回答：

因为最怕的是恢复一个“看起来能用、实际已经不匹配”的旧快照。模型、分块配置、知识源只要变了，就应该拒绝复用。

### 13.14 为什么 session store 要改成单文件

建议回答：

因为整文件 JSON 在会话变多时会带来写放大和恢复粒度过粗的问题。单 session 单文件更自然，也更利于清理和隔离损坏。

### 13.15 geo cache 批量刷盘的意义是什么

建议回答：

本质上是减少热点路径上的频繁 I/O。不是改业务逻辑，而是让缓存落盘方式更像长期可运行的服务。

### 13.16 你做这些优化的判断标准是什么

建议回答：

我主要看是不是高频链路、是不是系统性问题、以及能不能用适中的复杂度换到明确收益。

### 13.17 这轮最难的点是什么

建议回答：

最难的是每个小改动都会影响上下游，不是改完一个函数就结束。像 rewrite 会连带影响 location resolve、db query、prompt 和最终回答。

### 13.18 这轮你最满意的点是什么

建议回答：

我最满意的是把几条原本分散的小优化串成了一条完整的工程演进线，而不是零散 patch。

### 13.19 如果继续做，下一步你会做什么

建议回答：

我会继续补可观测性、配置化词表管理，以及根据部署形态决定是否把缓存和 session 外移。

### 13.20 你从这个项目学到了什么

建议回答：

我最大的体会是，工程化不是一下子换大架构，而是把高频问题持续沉淀成稳定能力。

## 14. 反问面试官时可以问的问题

如果面试到后半段让你反问，尽量别只问“团队氛围怎么样”。更好的问法是围绕你今天讲的这些点，反向了解对方团队成熟度。

### 14.1 如果岗位也是做 Agent / RAG

可以问：

- 你们现在的 Agent 更偏单 Agent 还是多 Agent / worker 协作？
- 你们在工具调用和模型推理之间，怎么划清职责边界？
- 你们现在更关注回答质量，还是更关注稳定性和成本？
- 你们内部有没有做 rewrite、retrieval、rerank 这类链路的离线评估？

### 14.2 如果岗位更偏工程平台

可以问：

- 你们线上最常见的 Agent 故障类型是什么？
- 现在可观测性做到什么程度，能看到 tool error、latency、cache hit rate 这些指标吗？
- 你们对 session state、memory、cache 的持久化是怎么设计的？
- 多实例部署时，你们是怎么处理缓存共享和状态恢复的？

### 14.3 如果岗位更偏业务落地

可以问：

- 你们判断一个 Agent 功能上线是否成功，主要看哪些业务指标？
- 现在用户最真实的痛点是在回答质量、响应速度，还是任务成功率？
- 团队更鼓励快速试错，还是先把基础设施做扎实再往前推？

这些问题的好处是，一方面显得你真懂系统，另一方面也能反向判断团队是不是只会讲概念。

## 15. 基础与低层知识问答

这一节专门补“偏基础、偏底层”的问题。很多面试官不会一上来问你架构，而是会先看你对底层原理是不是真的懂。

### 15.1 什么是 SSE，为什么这里适合用 SSE

建议回答：

SSE 是服务端单向推送的一种 HTTP 流式机制。它很适合这种“后端不断产生事件，前端持续接收展示”的聊天场景。这个项目里，后端会不断产出 session started、worker started、tool completed、assistant token 之类的事件，前端只需要稳定接收并按顺序渲染就行。  
如果面试官继续问为什么不是 WebSocket，你可以回答：这里主要是服务端往前端推，前端不需要高频双向实时通信，所以 SSE 更轻，协议也更简单。

### 15.2 SSE 和 WebSocket 的核心区别是什么

建议回答：

最核心的区别是通信模式。SSE 本质上还是 HTTP 连接上的单向流，服务器推给客户端；WebSocket 是升级后的全双工长连接，双方都可以主动发消息。  
在这个项目里，用户消息提交仍然可以走普通 HTTP，请求发出后，真正需要持续刷新的只有后端事件流，所以 SSE 更贴合场景。

### 15.3 为什么流式输出对 Agent 项目重要

建议回答：

因为 Agent 不是一下子就有最终答案，它会经历推理、调 worker、调工具、整合结果几个阶段。如果前端一直白屏，用户会觉得系统卡住了。流式输出的价值不只是“更炫”，而是把系统过程透明化，让用户知道现在执行到哪一步了。

### 15.4 HTTP 无状态是什么意思，那 session 为什么还能存在

建议回答：

HTTP 无状态的意思是单个请求天然不记住上一个请求发生过什么，不代表应用层不能自己维护状态。session 就是应用层人为引入的一层状态抽象。这个项目里，session 负责把一轮轮对话、运行时 memory、工具结果这些信息串起来，这样后续请求进来时，后端能知道是在续接哪段上下文。

### 15.5 session id 的作用是什么

建议回答：

session id 的本质是状态定位键。前端带着它发请求，后端就能找到对应的会话上下文、历史消息和运行时数据。没有这个 id，多轮对话就会退化成一堆彼此无关的单轮请求。

### 15.6 为什么 session store 改成单文件后更合理

建议回答：

因为整文件模式本质上是“所有 session 共享一个大对象”，任何一个会话更新都可能导致整份数据重写。单文件模式把存储粒度下沉到 session 级别后，读写范围更小，损坏隔离更清楚，也更适合后续做过期清理和迁移。

### 15.7 什么叫写放大

建议回答：

写放大就是“明明只改了一点点内容，底层却写了更多数据”。比如所有 session 都存在一个大 JSON 里，只改其中一个 session，也可能要把整个文件重新写一遍，这就是典型写放大。把 session 拆成单文件，本质上就是在减轻这个问题。

### 15.8 什么是缓存，为什么缓存会提速

建议回答：

缓存就是把高成本、可复用的结果先存起来，下次相同或等价请求直接复用，避免重复计算。这个项目里，RAG 查询如果每次都重新做 embedding、向量检索、BM25 和 rerank，开销会比较高；有了 cache 之后，重复查询可以直接命中结果，所以延迟会明显下降。

### 15.9 cache key 为什么很重要

建议回答：

因为 cache key 决定了“什么情况下算同一个请求”。如果 key 设计太粗，可能把本来不该复用的结果复用了，返回错误数据；如果太细，又会导致命中率很低。这个项目里不能只拿 query 文本做 key，还要把 source signature、top_k、检索模式、reranker 配置这些因素一起考虑进去。

### 15.10 TTL 是什么，为什么缓存通常要配 TTL

建议回答：

TTL 就是缓存项的存活时间。配 TTL 的原因很简单，缓存不是永久真理，过一段时间就应该自然失效，避免老数据一直占着内存，也减少结果过期的风险。  
在这个项目里，TTL 跟 source signature 这种强校验一起配合，效果会更稳。

### 15.11 LRU 是什么，为什么常跟缓存一起出现

建议回答：

LRU 是“最近最少使用淘汰”策略。缓存空间总是有限的，所以需要决定先淘汰谁。LRU 的直觉是：如果一个结果很久没被访问，后面继续被访问的概率通常也更低。对 query cache 这类场景来说，这是一个简单又实用的策略。

### 15.12 什么是缓存击穿、穿透、雪崩

建议回答：

这三个词经常一起考。

- 缓存击穿：某个热点 key 失效瞬间，大量请求一起打到后端
- 缓存穿透：请求的数据本来就不存在，所以一直绕过缓存打到底层
- 缓存雪崩：一批缓存大面积同时失效，导致后端压力陡增

如果面试官追问项目里怎么考虑，你可以答：当前阶段主要是单实例缓存，重点先放在 key 设计、TTL 和命中统计上；如果后面做分布式缓存，就要更认真处理这些问题。

### 15.13 embedding 是什么

建议回答：

embedding 可以理解成把文本映射成稠密向量，让“语义相近”的文本在向量空间里更接近。RAG 里常用它把 query 和文档 chunk 都编码成向量，再通过相似度搜索做召回。

### 15.14 向量检索为什么能做语义搜索

建议回答：

因为 embedding 模型学到的不是字面完全一样才接近，而是语义相关也会接近。比如用户说“人广附近哪里有中二”，文档里可能写的是正式地名和机种全称，向量检索有机会把这种语义相似但字面不同的内容召回出来。

### 15.15 余弦相似度是什么，为什么向量检索常用它

建议回答：

余弦相似度看的是两个向量方向有多接近，而不是只看绝对长度。文本 embedding 里，我们更关心“语义方向是不是相近”，所以余弦相似度很常见。它对很多文本检索场景都比较自然。

### 15.16 什么是 BM25

建议回答：

BM25 是一种经典的关键词检索打分方法，本质上还是词项匹配，只不过比简单 TF-IDF 更适合搜索场景。它会综合考虑词频、逆文档频率和文档长度归一化，所以在专有名词、型号、精确关键词这类场景里通常很有用。

### 15.17 为什么向量检索还不够，还要 BM25

建议回答：

因为两者擅长的东西不一样。向量检索强在语义相近，BM25 强在关键词精确命中。像店名、机种名、特定缩写这种东西，BM25 往往更稳；而用户口语表达、近义说法这类，向量检索更有优势。所以 hybrid search 本质上是在补短板。

### 15.18 rerank 是干什么的

建议回答：

rerank 是第二阶段排序。第一阶段检索更像广撒网，把一批可能相关的候选召回；第二阶段再用更精细的模型重新排序，把最相关的结果提到前面。它解决的是“能召回，但前几名未必最好”的问题。

### 15.19 为什么说检索分成召回和精排两阶段

建议回答：

因为这是一种很常见的性能和效果折中。你不可能对全库每条文档都做昂贵精排，所以先用快方法召回一批候选，再对候选做精排。这样既能控制成本，也能兼顾结果质量。

### 15.20 什么是分块，为什么 RAG 要分块

建议回答：

分块就是把原始文档切成更适合检索和喂给模型的小片段。因为原始文档往往太长，直接 embedding 整篇会丢掉局部细节，直接塞给模型也不现实。分块后，检索粒度更细，召回到的内容也更容易直接用于回答。

### 15.21 为什么块太大或太小都不好

建议回答：

块太大，信息会很杂，embedding 后语义容易被冲淡，检索不够精准；块太小，又容易丢失上下文，召回出来的内容不完整。所以 chunk size 本质上是在做信息密度和上下文完整性的平衡。

### 15.22 语义分块和固定分块有什么区别

建议回答：

固定分块更简单，按字数或 token 长度切，稳定、便宜、容易预测；语义分块会尽量沿着段落或语义边界切，通常更适合长文本，但成本和复杂度也更高。这个项目里我更倾向于按文档类型选择，而不是默认全开语义分块。

### 15.23 top-k 是什么

建议回答：

top-k 就是检索阶段最终取前 k 个候选结果。k 太小，可能漏召回；k 太大，又会引入更多噪声，还增加 rerank 和上下文拼装成本。所以它是一个很典型的效果和成本平衡参数。

### 15.24 什么是 snapshot，为什么它能加快恢复

建议回答：

snapshot 可以理解成某个阶段处理结果的持久化快照。对 RAG 来说，如果 chunk 和 embedding 都已经算好了，就没必要每次重启都从源文档重新走一遍。把这些中间产物落下来，恢复时直接加载，就能省掉不少冷启动时间。

### 15.25 为什么 snapshot 要做版本和模型签名校验

建议回答：

因为快照最危险的地方不在“读不出来”，而在“能读出来但已经不该用了”。模型换了、chunk 配置换了、知识源变了，如果还继续复用旧 snapshot，结果就会悄悄变脏。加版本号和模型签名，本质上是在防这种隐蔽错误。

### 15.26 什么叫幂等，哪些操作最好做成幂等

建议回答：

幂等的意思是同一个操作执行一次和执行多次，最终结果一致。像 snapshot 恢复、session 持久化覆盖写、某些缓存刷新动作，最好尽量往幂等方向设计。这样重试时系统更稳，也更容易做故障恢复。

### 15.27 为什么说文件持久化也有一致性问题

建议回答：

很多人会觉得只有数据库才谈一致性，其实文件也一样。比如写到一半进程挂了、多个写请求互相覆盖、旧文件和新文件切换不完整，这些都可能导致状态不一致。所以哪怕是本地文件方案，也要考虑写入时机、覆盖策略、损坏隔离和恢复逻辑。

### 15.28 为什么要做批量刷盘，而不是每次改动都立刻写盘

建议回答：

因为每次都立即写盘虽然直觉上更安全，但在高频路径上会带来很多 I/O 开销。批量刷盘的本质是拿一点点实时性换更好的吞吐和更少的写放大。前提当然是你要接受这是一种工程折中，并配好脏标记和合适的 flush 时机。

### 15.29 什么是背压，流式系统里为什么要关心它

建议回答：

背压可以理解成下游处理不过来，反过来影响上游继续生产数据。在流式输出场景里，如果后端产生事件很快、前端消费很慢，或者网络传输不稳定，就可能出现积压。虽然这个项目不是超高吞吐流处理系统，但在设计 SSE 和事件粒度时，还是要意识到这个问题。

### 15.30 为什么说结构化上下文比一大段自然语言更适合 Agent

建议回答：

因为 Agent 不只是聊天，它要读结果、做判断、再决定下一步动作。结构化上下文更像给模型准备好的工作台：哪些是 query，哪些是检索目录，哪些是店铺详情，哪些是路线信息，一眼能分清。这样模型更容易稳定消费，也更方便程序侧调试。

### 15.31 JSON Schema 在工具系统里是干什么的

建议回答：

它本质上是在描述“这个工具接受什么输入，字段类型是什么，哪些是必填”。放到 Agent 工具系统里，价值很直接：模型知道怎么构造参数，运行时也能做校验，避免把错误参数直接打到执行层。

### 15.32 为什么工具参数校验很重要

建议回答：

因为模型生成的参数并不天然可靠。没有校验的话，类型错、字段缺失、枚举值非法这些问题都会一路传到工具执行层，最后变成更难排查的运行时错误。参数校验其实是在给 Agent 系统加一道很关键的护栏。

### 15.33 为什么说 prompt 太长会影响稳定性

建议回答：

因为上下文越长，注意力分散得越厉害，模型越难抓住真正关键的部分。再加上多任务信息混在一起，容易出现该看的没看、不该放大的却被放大。所以这个项目里我会尽量把上下文拆层、把 worker 职责收窄，而不是一股脑全塞给一个 prompt。

### 15.34 temperature 是什么，这类系统里为什么通常不会太高

建议回答：

temperature 可以理解成采样随机性的强弱。越高，输出越发散；越低，输出越稳定。对创意写作来说高一点问题不大，但对 Agent、工具调用、参数生成这类场景，通常更希望稳定和可预测，所以不会把 temperature 开得太高。

### 15.35 什么是 hallucination，在这个项目里主要表现在哪

建议回答：

hallucination 就是模型说了看起来合理、但实际上没有依据或不正确的内容。在这个项目里，典型风险点包括：把不存在的店铺信息说得很像真的、把路线细节编出来、或者把知识库里没有的内容硬补全。所以我才会强调 tool 提供确定性结果、agent 基于结构化 observation 回答，而不是纯靠模型自由发挥。

### 15.36 为什么要把“可解释性”看得这么重

建议回答：

因为 Agent 系统的问题通常不是“完全不能跑”，而是“偶尔答偏、偶尔调错、偶尔恢复错”。这类问题最难排查。如果 rewrite 结果、tool observation、cache 命中情况、snapshot 校验状态都能看到，你定位问题会快很多。所以可解释性在这里不是锦上添花，而是维护成本的核心。

## 16. 前端基础问答

这一节主要补前端常见基础题，但我会尽量和这个项目挂钩，避免回答得像纯八股。

### 16.1 React 为什么适合这个项目

建议回答：

因为这个项目前端不是单纯静态页面，而是一个状态很多、交互连续的聊天式应用。React 适合处理这种“状态驱动 UI”的场景：消息列表、流式事件、地图结果、会话切换、加载状态，都会随着数据变化频繁更新。用组件化方式拆开之后，界面和状态的关系会更清楚。

### 16.2 什么是组件化，组件化的好处是什么

建议回答：

组件化就是把 UI 拆成一块块可复用、可组合的最小单元。好处是职责更清晰，也更方便维护。像聊天输入框、消息列表、地图面板、会话侧栏、状态提示这些，本质上都可以是独立组件。后面某一块要改样式或交互，不需要把整页一起推翻。

### 16.3 什么是状态，为什么前端要管理状态

建议回答：

状态可以理解成“会影响界面展示的数据”。比如当前 session、消息列表、流式输出内容、工具执行中的中间状态、地图选中的店铺，这些一变，界面就得跟着变。前端要做状态管理，本质上就是保证数据变化和 UI 变化能稳定对齐。

### 16.4 local state 和 global state 的区别是什么

建议回答：

local state 只影响某个组件自己，比如输入框里的临时内容、某个弹窗是否打开；global state 则会被多个区域共享，比如当前 session、当前消息流、地图结果这种。区分这两类状态的意义在于，不要把所有东西都塞到全局，也不要把本该共享的数据拆得太散。

### 16.5 为什么状态设计不好会让前端变难维护

建议回答：

因为一旦状态边界不清，就会出现数据源不唯一、组件互相改来改去、某一处更新另一处没同步的问题。聊天类应用尤其明显，流式输出一来，消息列表、加载状态、滚动行为、工具状态都在变。如果状态模型不清楚，前端会很快变得很乱。

### 16.6 useEffect 是干什么的

建议回答：

`useEffect` 主要用来处理副作用，也就是那些“不只是渲染 UI”的事情，比如订阅事件流、发请求、绑定监听器、清理连接。像这个项目里接 SSE、监听流式事件、组件卸载时关闭连接，都属于很典型的 `useEffect` 场景。

### 16.7 为什么副作用和渲染要分开

建议回答：

因为渲染本质上应该是纯的：给定同样的状态，产出同样的 UI。副作用则会真正影响外部世界，比如发请求、开连接、写缓存。如果把这两类逻辑混在一起，组件就会变得很难预测，也更容易重复请求或产生脏状态。

### 16.8 为什么列表渲染要关注 key

建议回答：

因为 key 决定了 React 怎么识别“这是不是同一个节点”。在聊天列表里，如果 key 不稳定，流式更新时就可能出现节点复用错误、内容闪烁、滚动异常这类问题。稳定 key 的本质是给 React 一个可靠的身份标识。

### 16.9 受控组件是什么

建议回答：

受控组件就是输入框的值由 React 状态统一管理，而不是让 DOM 自己记。这样好处是输入值、提交逻辑、禁用状态、清空行为都能被应用层明确控制。聊天输入框就是很典型的受控组件场景。

### 16.10 为什么前端也要考虑性能

建议回答：

因为体验不只取决于后端快不快。像聊天流式渲染、地图结果更新、长消息列表滚动，如果前端频繁无意义重渲染，用户一样会感觉卡。前端性能优化不一定是炫技，很多时候就是少做重复工作。

### 16.11 什么是重渲染，为什么会发生

建议回答：

重渲染就是组件因为 props、state 或上层变化而再次执行渲染逻辑。它本身不是坏事，问题在于“不必要的重渲染”。比如某个局部状态变化，却让整个消息区和地图区都重新跑一遍，这种就值得优化。

### 16.12 前端怎么理解“单向数据流”

建议回答：

单向数据流就是数据从状态源头往下传给组件，组件通过事件把变更意图往上送，再由状态更新后重新驱动 UI。好处是数据来源清楚，排查问题时更容易知道“是谁改了谁”。

### 16.13 为什么流式聊天界面比普通表单页更复杂

建议回答：

因为它不是点一下提交、等结果回来就结束了。聊天界面里，消息可能一边生成一边展示，还会穿插工具状态、错误提示、滚动定位、会话切换、取消重试这些行为。它的复杂度更接近一个小型实时应用。

### 16.14 前端怎么处理异步请求的竞态问题

建议回答：

核心思路是让旧请求的结果不要覆盖新状态。比如用户快速切换 session 或连续发送请求时，前一个异步结果回来得更晚，如果没有做请求标识、取消或结果归属判断，就可能把新界面覆盖掉。聊天和流式场景里，这种问题很常见。

### 16.15 为什么错误状态也要设计

建议回答：

因为真实系统不会只存在成功路径。SSE 中断、工具失败、网络超时、空结果、地图服务异常，这些都需要界面能表达出来。好的错误状态不是简单弹个报错，而是让用户知道发生了什么、现在还能做什么。

### 16.16 为什么前端也需要做 loading 和 skeleton

建议回答：

因为用户感知的是等待体验，不是代码实现。尤其是 Agent 项目，很多动作不是瞬间完成的。适当的 loading、阶段提示或 skeleton，能让等待更可理解，也能减少“是不是卡死了”的心理负担。

### 16.17 什么是防抖和节流

建议回答：

防抖是“连续触发时，只在最后一次真正执行”；节流是“连续触发时，固定时间内最多执行一次”。如果前端有搜索输入、地图拖拽、窗口 resize 这类高频事件，这两个概念很常用。

### 16.18 为什么前端也要关注可维护性，不只是好不好看

建议回答：

因为 UI 会一直变。今天只是聊天框，明天可能加工具面板、路线卡片、过滤条件、埋点、国际化。如果组件边界和状态结构一开始就乱，后面每次加功能都会越来越痛苦。前端可维护性本质上也是工程能力。

### 16.19 什么叫前后端分离，它的好处是什么

建议回答：

前后端分离就是前端负责交互和展示，后端负责业务逻辑、数据和 Agent 能力，双方通过 API 协作。好处是职责清楚，开发节奏更独立，也更适合这种前端要做流式交互、后端要做复杂推理编排的项目。

### 16.20 为什么这个项目的前端不只是“调接口”

建议回答：

因为它还承担了会话体验组织、流式事件消费、消息状态拼接、地图联动和错误反馈这些工作。尤其在 Agent 项目里，前端某种程度上是在把“系统运行过程”翻译成用户能理解的体验。

### 16.21 这个项目的前端和普通聊天页面最大的区别是什么

建议回答：

最大的区别是它不是“发一条消息，等一个最终字符串回来”这么简单。这个项目里，前端要先创建会话，再订阅事件流，然后根据不同阶段的事件去更新消息、进度、地图和路线。也就是说，前端不是只展示结果，它还要承接 Agent 的执行过程。

### 16.22 为什么这里要拆“派发会话”和“订阅 SSE”两步

建议回答：

因为一次 Agent 请求不是同步瞬时完成的。先用会话接口把任务派发出去，再单独用 SSE 订阅过程事件，前端才能同时兼顾两件事：第一，拿到一个稳定的 session id；第二，在执行过程中持续接收 `session.started`、`tool.started`、`assistant.token`、`assistant.completed` 这类事件。  
如果把它硬塞成一个普通同步请求，前端就只能傻等，体验会差很多。

### 16.23 为什么这个项目的前端适合用 Zustand 一类轻量状态管理

建议回答：

因为这里的共享状态主要集中在会话、消息流、地图 artifacts、发送状态这些 UI 层数据上，复杂度还没有大到需要特别重的状态框架。用轻量方案比较合适，一方面足够表达共享状态，另一方面不至于把本来就已经复杂的 SSE 和聊天状态再包装得更重。

### 16.24 这个项目里前端最难管理的状态是什么

建议回答：

我觉得最难的是“同一个会话在多个时间维度上的状态”。比如：

- 用户刚提交，还没收到任何事件
- 已经开始流式返回 token
- 工具执行到一半
- 路线事件先到了，但最终 assistant 文本还没结束
- SSE 断了，但会话本身可能还在后台跑

这些状态不是一个 `loading=true/false` 能表达完的，所以前端要把消息状态、会话状态和地图状态分开看。

### 16.25 为什么导航路线要允许先于最终回答渲染

建议回答：

因为路线本身就是高价值结果，没必要等整段自然语言都生成完才展示。这个项目里导航链路会先收到路线 ready 事件，前端可以先把 route artifacts 渲染出来，让用户先看到路线和地图，再等最终 assistant 文本补全。这种设计会让系统显得更实时，也更像任务型产品。

### 16.26 为什么地图不让模型直接输出 HTML，而是走结构化 artifacts

建议回答：

因为地图这类 UI 比较重，如果让模型自由输出 HTML 或组件片段，稳定性会很差，也不好控。这个项目选择的是后端输出结构化 artifacts，前端用固定组件去渲染。这样模型只负责提供结构化结果，具体怎么画地图由前端掌控，边界更清楚，也更安全。

### 16.27 这个项目里前端怎么处理历史会话恢复

建议回答：

恢复历史会话时，前端不是只拿一段聊天文本回来，而是还要恢复地图相关的 artifacts，比如 shops、route、destination、client location。这样用户重新打开旧会话时，不只是看到文字，还能看到当时的候选点位和路线结果。这一点很像“结果态恢复”，不是简单消息回显。

### 16.28 为什么 SSE 断开后不能只靠重新拉整页数据

建议回答：

因为这样会丢掉过程体验，也可能错过中间事件。这个项目更合理的做法是：流式过程中尽量依赖事件恢复，会话结束后再用 session detail 做最终状态校准。也就是说，SSE 更偏“过程恢复”，session detail 更偏“结果恢复”，两者职责不一样。

### 16.29 replay buffer 和 session store 在前端体验上分别解决什么问题

建议回答：

从前端视角看，replay buffer 解决的是“流还没结束时断线重连，怎么把漏掉的事件补回来”；session store 解决的是“就算流已经结束，或者页面刷新了，我怎么把最终会话结果重新恢复出来”。  
一个偏执行中恢复，一个偏执行后恢复。

### 16.30 这个项目为什么要把浏览器定位和逆地理拆成前后端两段

建议回答：

因为这两件事天然属于不同边界。浏览器定位更像客户端能力，应该由前端获取；逆地理解析更像服务能力，而且涉及第三方地图 key，不适合暴露到浏览器。所以前端只负责拿坐标，后端负责调用地图服务把区域信息补全，再回注到会话上下文里。

### 16.31 为什么前端会先做定位缓存预热

建议回答：

因为用户真正发第一条消息时再现拿定位，体验上会更慢，而且浏览器定位本身有延迟和抖动。先做预热缓存的好处是，很多时候新会话启动时可以直接复用已有坐标和地区信息，减少一次额外等待。这个设计本质上是在用前端小缓存换首条消息体验。

### 16.32 这个项目里地图和聊天为什么要联动，而不是完全分开

建议回答：

因为用户问的问题本身就不是纯文本问题。像“附近哪里有机厅”“从我这去某家怎么走”，结果天然包含空间信息。如果聊天只给文字、地图单独一套逻辑，用户就得自己脑补。把两者联动起来，前端可以把同一轮 Agent 结果同时翻译成自然语言和空间视图，体验会完整很多。

### 16.33 这个项目前端最容易出 bug 的点是什么

建议回答：

我会说是“异步状态叠加”。比如：

- 会话刚切换，旧 SSE 还没完全结束
- 新请求发出后，旧请求结果晚到
- 路线 artifacts 先到了，但最终详情还没刷新
- 页面刷新后，流式中间态和最终态不一致

这种 bug 不是样式问题，而是状态归属和时序问题，所以前端设计时必须很清楚每个数据属于哪个 session、哪个阶段。

### 16.34 这个项目里为什么会话切换容易出前端问题

建议回答：

因为会话切换不是只换一段文本，它还会连带切换当前 SSE 订阅、消息列表、地图 artifacts、正在进行中的发送状态。如果边界没处理好，就会出现“旧会话的流式事件写进新会话界面”这种很典型的错乱。所以会话切换时，本质上是在切一整组状态，而不是切一个 id。

### 16.35 这个项目里前端为什么要区分“过程态”和“结果态”

建议回答：

因为流式事件和最终会话详情不是一回事。过程态来自 SSE，用来告诉用户当前执行到哪一步；结果态来自会话详情，用来保证最终一致。比如导航过程中可以先收到路线事件，但最终 assistant 完成后，还要再用会话详情把 shops、route、destination 这些结构化结果收敛成最终状态。

### 16.36 为什么这个项目里地图路线要做坐标系处理

建议回答：

因为浏览器定位、地图服务和路线结果不一定都用同一种坐标系。这个项目里浏览器位置更偏 WGS84，而高德地图展示统一更偏 GCJ02。如果前端不在渲染前做坐标转换，就会出现点位偏移、路线和 marker 对不齐的问题。所以这不是地图“显示一下”这么简单，而是带有地理数据一致性要求。

### 16.37 这个项目里为什么候选店铺和路线结果要共用一套地图 artifacts 思路

建议回答：

因为它们虽然业务不同，但前端消费方式很像，都是“后端给结构化地图结果，前端负责渲染”。把候选店铺和路线都纳入统一 artifacts 思路之后，前端可以用更稳定的方式处理 shops、route、destination、view_payload 这些字段，而不是为每种地图场景都单独发明一套临时协议。

### 16.38 为什么前端不应该只依赖 assistant 文本来驱动地图

建议回答：

因为文本不稳定，也不适合程序解析。今天模型说“离你最近的是 A 店”，明天可能换个说法，前端很难可靠提取结构化信息。地图这种能力更适合由结构化 artifacts 驱动，assistant 文本只负责给人读。这样 UI 和文案不会互相绑死。

### 16.39 这个项目里前端如何理解“同一个 session 的多个视图”

建议回答：

一个 session 里其实同时有几层视图：聊天文本视图、工具执行过程视图、地图视图、路线视图。它们服务的是同一轮任务，但展示重点不同。前端要做的不是把这些东西混成一坨，而是让它们围绕同一个 session 协同更新。

### 16.40 为什么说这个项目的前端更像“状态机 UI”

建议回答：

因为界面会随着后端执行阶段不断迁移：创建会话、开始执行、工具运行、返回 token、路线 ready、最终完成、历史恢复。每个阶段能展示的东西和允许的交互都不一样。用“状态机 UI”来理解会更准确，而不是把它看成普通页面渲染。

## 17. Agent 基础问答

这一节补的是更偏 AI Agent 基础原理的问题，适合面试官从“你会不会用大模型”继续往“你理解不理解 Agent”追问时使用。

### 17.1 什么是 Agent

建议回答：

我会把 Agent 理解成“能基于目标做多步决策和执行的系统”，不只是一个会生成文本的模型。它通常需要理解任务、选择动作、调用工具、读取结果、调整下一步，再给出最终答案。也就是说，Agent 的关键不在模型本身，而在“模型 + 状态 + 工具 + 执行循环”。

### 17.2 Agent 和普通聊天机器人有什么区别

建议回答：

普通聊天机器人更像单轮或多轮对话生成器，重点是把话说出来；Agent 则更强调任务完成，它要真的去查、去算、去调用外部能力。前者偏表达，后者偏执行。

### 17.3 什么是 ReAct

建议回答：

ReAct 可以理解成 reasoning 和 acting 交替进行的一种范式。不是模型先想完再一次性回答，而是先做一轮推理，决定要不要调用工具；拿到工具结果后，再继续下一轮推理。这个项目里的主 Agent 和 worker 调工具，本质上就是沿着这条思路在走。

### 17.4 为什么 ReAct 比单次生成更适合任务型场景

建议回答：

因为任务型场景往往依赖外部真实信息。单次生成只能靠模型内部知识和上下文猜；ReAct 则允许系统在中途停下来，真的去查数据库、查知识库、解地点、算路线，再把结果带回来继续推理。这样答案会更接地，也更可控。

### 17.5 什么是 tool calling

建议回答：

tool calling 就是让模型不只是输出自然语言，还能输出结构化工具调用意图，由系统去执行真实函数或服务，再把结果回传给模型。它相当于给模型加了“手和脚”，让它不只会说，还能做事。

### 17.6 为什么 tool calling 不能等同于 Agent

建议回答：

因为 tool calling 只是能力接口，不是完整执行框架。只有当系统还能维护状态、决定下一步、根据 observation 调整计划时，它才更像 Agent。否则只是“模型偶尔调用一下函数”。

### 17.7 什么是 observation

建议回答：

observation 就是工具执行后的结果，也可以理解成 Agent 下一轮推理要读取的现实反馈。没有 observation，Agent 的动作就不会真正影响后续推理链路，系统也更像是假装在调用工具。

### 17.8 为什么 Agent 需要 memory

建议回答：

因为多步执行如果不记状态，每一步都会像重新开始。memory 的价值不是把所有聊天记录永远保存，而是让当前目标、关键约束、最近 observation、阶段性结果能被后续步骤继续消费。

### 17.9 短期记忆和长期记忆怎么区分

建议回答：

短期记忆更像运行时工作内存，服务当前任务；长期记忆更像跨会话、跨时间保存的用户偏好或长期知识。这个项目现在更偏短期记忆，因为主要目标是把当前搜索、检索和路线任务做好。

### 17.10 为什么 Agent 需要规划能力

建议回答：

因为很多任务不是一步到位的。哪怕不做很重的 planner，系统也至少要有“先做什么、后做什么、失败怎么回退”的基本决策能力。否则工具再多，也只是随机调用。

### 17.11 什么是单 Agent，什么是多 Agent

建议回答：

单 Agent 是一个执行主体负责理解、决策、调用和整合；多 Agent 则是把不同职责拆给不同角色，比如 orchestrator、search worker、navigation worker。拆分的意义不是为了看起来高级，而是为了收敛上下文和权限边界。

### 17.12 为什么不是 Agent 越多越好

建议回答：

因为 Agent 越多，协调成本越高，状态同步和调试也越复杂。如果任务边界并不明显，强行拆很多 Agent，最后只会让系统更难维护。所以我更倾向于按真实职责拆，而不是为了“多 Agent”而多 Agent。

### 17.13 什么是系统 prompt，它为什么重要

建议回答：

系统 prompt 可以理解成模型在当前角色下长期遵循的顶层规则。它决定模型该用什么身份、遵守什么边界、优先读取什么信息、如何使用工具。对 Agent 来说，这不是文案问题，而是行为约束问题。

### 17.14 为什么 prompt 工程不能替代系统设计

建议回答：

因为 prompt 只能影响模型怎么想，不能替你补状态边界、工具权限、缓存策略、恢复机制这些系统层问题。一个系统如果基础结构不清楚，单靠改 prompt 最多只是把问题往后拖。

### 17.15 什么是上下文窗口，为什么 Agent 容易受它限制

建议回答：

上下文窗口就是模型单次能处理的信息上限。Agent 场景里，很容易把用户历史、memory、工具结果、知识检索内容都塞进去，很快就会变长。一长就不只是成本问题，还会直接影响稳定性和注意力分配，所以需要靠结构化上下文和分层组织来控制。

### 17.16 为什么说 Agent 的核心难点不只是模型能力

建议回答：

因为真实问题经常出在模型外面。比如工具参数不稳定、memory 组织混乱、检索召回不准、状态恢复脏、错误不可观察，这些都不是模型本身能独立解决的。Agent 之所以难，是因为它天然是系统工程。

### 17.17 什么是 deterministic，为什么 Agent 里总在强调它

建议回答：

deterministic 就是相同输入下尽量给出稳定可预期的结果。Agent 系统里，越靠近执行层的东西越应该 deterministic，比如工具、参数校验、缓存 key、snapshot 校验。这样你才能把不确定性主要留在“需要推理和生成的地方”。

### 17.18 什么是 guardrail

建议回答：

guardrail 可以理解成护栏机制，用来限制系统不要偏得太离谱。比如工具参数校验、可用工具白名单、snapshot 签名校验、rewrite 规则约束，这些都可以算 guardrail。它们的目标不是让系统更聪明，而是让系统更稳。

### 17.19 为什么说 Agent 要可观测

建议回答：

因为它不是单步黑盒，而是多步状态机。只看最终答案，很多问题根本不知道出在哪。只有能看到中间事件、工具调用、memory 变化、cache 命中、错误分布，你才有机会真正调优。

### 17.20 评估 Agent 为什么比评估普通问答更难

建议回答：

因为你评估的不只是“最后一句话对不对”，还包括它有没有选对工具、有没有走对链路、有没有合理使用 memory、有没有稳定恢复。也就是说，Agent 评估是链路级评估，不只是答案级评估。

### 17.21 什么是任务成功率，为什么它重要

建议回答：

任务成功率就是用户提出一个目标后，系统最终有没有把这件事完成。对 Agent 来说，这往往比单纯的文本相似度更重要。因为用户要的不是“看起来像对的回答”，而是真正完成搜索、定位、推荐或路线规划。

### 17.22 为什么说 Agent 项目很容易从“聪明”变成“脆”

建议回答：

因为一旦系统越来越依赖多步推理和外部工具，链路上的任何一环出问题都会把结果带偏。看起来更聪明，实际上也更脆。所以工程上必须不断补护栏、补缓存、补恢复、补观测。

### 17.23 什么是 orchestration

建议回答：

orchestration 就是编排，核心是决定谁先做、谁后做、谁拿什么上下文、谁能用什么工具、结果怎么回收。多 Agent 或多 worker 系统里，真正的复杂度很多都在 orchestration，而不只是 prompt。

### 17.24 为什么结构化输出在 Agent 里很重要

建议回答：

因为系统后面通常还要继续消费模型结果。自然语言对人友好，但对程序不够稳定；结构化输出则更适合做参数传递、状态记录和链路控制。像 tool calling、context payload、rewrite 字段，本质上都在利用结构化输出。

### 17.25 为什么你说这个项目更像工程化 Agent，而不是 demo Agent

建议回答：

因为它不只是能演示一次调用，而是开始考虑了运行时 memory、worker 分工、query rewrite、hybrid retrieval、cache、snapshot、session 持久化和可观测性。这些东西单看都不炫，但合起来才像一个能持续运行和持续迭代的系统。

### 17.26 这个项目里主 Agent 和 worker 的一次真实协作链路是什么样的

建议回答：

可以举“某地点附近有没有机厅”这个例子。主 Agent 先判断这是搜索型任务，不直接自己查，而是把任务派给 search worker。search worker 再根据上下文里的地点 hint、rewrite 字段和 client location，决定先用地点解析还是直接查结构化数据。如果需要，它会先解地点，再调 `db_query_tool` 做距离排序或区域筛选，必要时再补一轮知识检索。最后把结果写回 runtime state，由主 Agent 负责整合成用户能读的最终回答。

### 17.27 为什么这个项目的主 Agent 不直接自己把所有工具都调完

建议回答：

因为主 Agent 的职责更偏调度和收口，而不是把所有执行细节都塞自己身上。如果它既做任务拆分、又做工具执行、又做最终表达，prompt 会越来越重，工具权限也会越来越散。拆成 worker 之后，每个执行角色拿到的是更聚焦的任务和更小的工具集合，稳定性会更好。

### 17.28 这个项目里 worker 是怎么知道优先调哪个工具的

建议回答：

不是靠“模型自由发挥”这么简单，而是靠几层东西一起约束：系统 prompt、worker prompt、allowed tools、structured context 和 rewrite hint。比如搜索 worker 会被明确告知，遇到地标、商圈、校园、地铁站这类地点表达时，优先先做地点解析；遇到 FAQ、长评论、规则说明这类问题时，优先走知识检索。这意味着工具选择不是完全黑盒，而是受工程约束的。

### 17.29 为什么这个项目要把 rewrite 结果显式放进 context payload

建议回答：

因为如果 rewrite 只存在某个内部变量里，模型看不见，调试也看不见，下游工具更难稳定复用。把它显式写进 `query` 相关结构化字段之后，worker 能直接把它当 normalization hint 来消费，比如 `rewrite_keyword`、`rewrite_knowledge_query`、`place_query` 这些。这样 query rewrite 就从“某个隐蔽预处理步骤”变成“运行时可见状态”。

### 17.30 这个项目为什么要把 structured data 和 knowledge retrieval 都保留

建议回答：

因为这两类信息解决的问题不一样。结构化查询更适合回答“有哪些店”“在哪”“距离多远”“支持什么机种”这类事实；知识检索更适合 FAQ、长评论、活动说明、细节补充这类文本证据问题。只靠其中一种都会有短板，所以这个项目保留了两条链路，再由 worker 根据问题类型决定怎么组合。

### 17.31 这个项目里 context builder 的价值是什么

建议回答：

它的价值不是简单把数据拼起来，而是决定“模型到底先看到什么、怎么看这些信息”。比如把 query、directory、search catalog、shop details、route 分开，就是为了减少信息互相打架。Agent 能不能稳定读懂上下文，很多时候不取决于模型多聪明，而取决于你给它的上下文是不是有结构。

### 17.32 这个项目为什么强调 tool result 要回写 runtime state

建议回答：

因为多步执行里，前一步工具的结果必须能被后一步继续消费，否则每一步都会重复查或者重复推理。比如地点一旦解析成功，后面排序、路线规划、最终总结都应该复用这份结果。把 tool result 回写 runtime state，本质上就是让 Agent 真正具备“状态推进”能力。

### 17.33 这个项目里 memory 和 session state 的关系怎么讲

建议回答：

可以这样区分：memory 更偏运行时工作状态，关注当前这轮或最近几轮任务需要记住什么；session state 更偏会话级容器，除了 memory 之外，还会承载消息历史、执行状态、最终 artifacts、恢复所需的数据。简单说，memory 是会话状态里最重要的一块，但不是全部。

### 17.34 为什么这个项目里要同时有 replay buffer 和 session state

建议回答：

因为它们解决的不是同一个问题。replay buffer 主要服务流式事件恢复，解决的是“SSE 断了之后怎么补过程”；session state 主要服务会话级状态恢复，解决的是“会话结束后怎么继续看最终结果”。一个偏事件流层，一个偏会话存储层。

### 17.35 这个项目里 navigation worker 和 search worker 的边界怎么区分

建议回答：

search worker 主要负责找候选目标，回答“去哪”“附近有什么”“哪些店符合条件”；navigation worker 主要负责拿着明确目标去做路线相关工作，回答“怎么去”“多远”“走路还是地铁怎样”。前者偏目标发现，后者偏路径执行。这样拆完之后，路线工具和搜索工具不会混着乱用。

### 17.36 这个项目为什么路线优先支持结构化 route artifacts，而不是只返回一段文字

建议回答：

因为路线这种结果天然有结构，比如起点、终点、距离、时长、polyline。只返回一段文字虽然人能看，但前端不好复用，也没法稳定画地图。把路线做成结构化 artifacts 后，前端既能展示文字总结，也能直接渲染地图路径，这对任务型产品更有价值。

### 17.37 这个项目里的 MCP 工具和 builtin tool 是怎么协作的

建议回答：

builtin tool 更像本地稳定能力，比如结构化查询、地点解析、路线 fallback、知识检索；MCP 工具更像运行时发现的外部能力，比如某些地图路线能力。系统先通过统一 registry 暴露它们，再由 worker 在 allowed tools 范围内选择。这样做的好处是，主链路不依赖某个外部工具一定存在，但一旦存在又能被自然接入。

### 17.38 为什么这个项目里 allowed tools 很重要

建议回答：

因为它直接决定某个 Agent/worker 在当前任务里能做什么，不能做什么。没有这层限制，模型很容易在不该用的时候乱用工具，甚至跨职责调用。allowed tools 相当于给每个角色划了权限边界，这对稳定性很关键。

### 17.39 这个项目怎么避免 Agent 在零结果时原地打转

建议回答：

一个方法是 prompt 层明确约束，比如结构化查询返回零结果时，不要拿同一组条件反复重试，而是改走一次更合理的 fallback，比如知识检索或放宽地点理解。另一个方法是 runtime 层有 loop guard，防止链路无限重复。也就是说，这个项目不是只靠模型自觉停止，而是有制度化约束。

### 17.40 这个项目里为什么说最终回答仍然应该由 Agent 生成

建议回答：

因为工具只会给原始结果，不会自动知道用户这次最关心什么。最终回答需要综合用户问题、工具结果、路线/店铺信息和上下文重点做表达，这部分更适合由 Agent 来完成。否则系统就会退化成“工具输出拼接器”，可读性和灵活性都不够。

### 17.41 这个项目里为什么需要 orchestrator，而不是接口里直接调 runtime

建议回答：

因为接口层更适合做协议收发，不适合直接背负并发控制、会话执行调度、生命周期管理这些职责。orchestrator 的价值是把“同一个 session 能不能重复执行”“请求进来后怎么启动 runtime”“执行状态怎么切换”这些问题收进一层统一逻辑。这样 HTTP 入口、runtime、session 状态三者边界会更清楚。

### 17.42 这个项目里为什么要防同一 session 并发执行

建议回答：

因为同一个 session 如果同时跑两条执行链路，很容易把 working memory、消息状态和最终 artifacts 搅乱。比如一条链路还在写地点结果，另一条已经开始写路线结果，最后会话状态就不一致了。所以同一 session 的串行约束，本质上是在保护状态一致性。

### 17.43 这个项目里为什么 prompt 里要显式写很多工具使用规则

建议回答：

因为仅仅把工具暴露给模型，并不代表模型就会稳定地按你想要的方式用。像“遇到地点名优先先解析”“零结果时不要死循环重试”“FAQ 类问题优先走知识检索”，这些都属于经验约束。如果不在 prompt 和 skill 里显式写出来，链路会更飘，调试成本也更高。

### 17.44 这个项目里为什么 prompt 还不够，还要 context builder

建议回答：

因为 prompt 解决的是“告诉模型该怎么做”，context builder 解决的是“模型到底看到了什么材料”。如果材料本身混乱，prompt 再好也很难稳定发挥。这个项目里把 query、search catalog、shop details、route 分层，本质上就是在给 prompt 提供可消费的输入结构。

### 17.45 这个项目里为什么 search worker 不只查结构化数据，还会做知识回退

建议回答：

因为用户问的东西不总是纯筛选题。有些问题表面像在问店，实际想知道的是活动、评论、机种补充、规则说明，单靠结构化表不够。所以 search worker 的职责不是机械查表，而是在需要的时候把结构化事实和文本证据结合起来。

### 17.46 这个项目里为什么地点解析不直接等同于区域筛选

建议回答：

因为用户说的“人广”“徐家汇地铁站”“某大学附近”并不一定是行政区，它更像一个地标或 place query。直接拿它当 city/county 过滤，很容易错。这个项目后来专门把商圈、高校、地铁站这类表达抽出来，再走 `place_query -> location resolve -> nearby search` 这条链路，就是为了把“地标”和“行政区”分开处理。

### 17.47 这个项目里为什么 query rewrite 不只是替换同义词

建议回答：

因为它最终服务的是下游执行，不是文本美化。除了别名归一化，它还要产出更适合工具消费的结构化 hint，比如 shop name、place query、knowledge query、区域提示。也就是说，rewrite 在这里更像“查询理解层”，而不是单纯做词面替换。

### 17.48 这个项目里为什么 location resolve 不能无限重试

建议回答：

因为地点解析本身也是成本，而且一旦反复失败，继续重试往往只是在放大噪声。更合理的做法是：解析一次或少量回退后，就转入区域搜索、知识回退或者明确告诉用户地点不够具体。否则 Agent 很容易在一个不确定地点上反复兜圈。

### 17.49 这个项目里为什么 loop guard 是必要的

建议回答：

因为 ReAct 链路如果没有循环保护，模型可能会在“查不到 -> 换个说法再查 -> 还查不到”这种模式里打转。loop guard 的意义不是限制模型能力，而是防止系统在异常情况下无限消耗 token 和工具调用成本。它是典型的工程护栏。

### 17.50 这个项目里为什么说 worker 的 allowed tools 是一种最小权限设计

建议回答：

因为每个 worker 只应该拿完成当前任务所需的最小工具集合。search worker 不应该随便拿所有路线工具，navigation worker 也不该拥有和它任务无关的查询能力。这样做的价值不只是安全，还能减少模型在工具选择上的噪声，让角色更聚焦。

### 17.51 这个项目里为什么说 route_plan_tool 和 MCP 路线工具是互补关系

建议回答：

因为 builtin route tool 更像一个稳定 fallback，保证在外部工具不可用时系统仍然能完成基本路线能力；MCP 路线工具则更像可发现的增强能力，一旦存在就优先利用。两者结合起来，系统既不会被外部依赖卡死，也能在外部能力存在时拿到更好的结果。

### 17.52 这个项目里为什么 client location 要直接注入上下文，而不是只放在工具参数里

建议回答：

因为当前位置不只是某个单次工具调用的参数，它还是理解用户意图的重要背景。像“附近”“从我这过去”“当前城市还有别的吗”这种表达，都需要 agent 在推理阶段就感知到位置上下文。只把它藏在某个工具参数里，会削弱整个推理链路对位置的利用。

### 17.53 这个项目里为什么最终要同时保留自然语言回答和结构化结果

建议回答：

因为两者服务不同对象。自然语言回答是给用户快速理解结论，结构化结果是给前端复用、给后续步骤继续消费、给会话恢复使用。只保留文字，系统可操作性会很差；只保留结构化结果，用户读起来又不自然。所以两条输出都要有。

### 17.54 这个项目里怎么体现“agent 服务于产品”，而不是“产品配合 agent”

建议回答：

体现在很多地方。比如路线结果不是只返回一段解释，而是直接变成地图 artifacts；定位信息不是为了展示炫技，而是服务“附近搜索”和导航起点；query rewrite 也不是为了让文本更好看，而是为了提升任务成功率。也就是说，Agent 在这里是为具体产品目标服务的，不是单独秀一套 AI 技术。

### 17.55 如果面试官问：这个项目最容易被误解成“只是 prompt 工程”的点是什么

建议回答：

我会说是表面上看起来好像很多能力都跟 prompt 有关，但真正把系统拉开差距的，其实是工具边界、状态组织、缓存恢复、结构化上下文和前端联动。prompt 很重要，但它只是这套系统里的一层。如果没有这些工程层支撑，prompt 调得再花也很难稳定。

## 18. 使用建议

如果你时间不多，优先记住：

1. 30 秒项目介绍
2. ReAct 是怎么实现的
3. Query Rewrite 的演进过程
4. RAG 怎么从 demo 做成工程能力
5. 缓存和持久化为什么值得讲

如果面试更深，重点展开：

1. 为什么是主 Agent + worker，而不是一个大 prompt
2. 为什么 query rewrite 选择规则优先
3. 为什么 RAG 需要 hybrid + reranker + cache
4. 为什么 session store 要从整文件切到单 session 单文件

如果你准备二面或主管面，建议再多练三类表达：

1. 架构权衡题  
   比如为什么现在不做长期记忆、为什么先不用 Redis、为什么先不用外部向量数据库。

2. 失败与降级题  
   比如某个工具挂了怎么办、snapshot 恢复失败怎么办、rewrite 没命中怎么办。

3. 演进路线题  
   比如单机方案怎么扩到多实例、进程内缓存怎么外移、文件持久化怎么迁移到外部存储。

最重要的一点是：不要把自己讲成“写了一堆功能的人”，而要讲成“能识别问题、做取舍、把系统持续往工程化推进的人”。
