- **实现两阶段重排序机制**：设计向量召回 + Cross-encoder精排的检索流程，先用向量检索召回候选文档（top_k × 5），再使用cross-encoder模型对候选结果重新打分，预期提升10-20% Top-1准确率；支持多种reranker模型（Sentence Transformers、关键词匹配），并实现自动降级机制保证系统稳定性。
- **开发混合检索算法**：自实现BM25算法（无需外部依赖），结合向量检索和关键词检索优势；通过可配置的alpha权重参数实现得分融合（`final_score = alpha × vector + (1-alpha) × bm25`），适配不同规模知识库场景；支持中英文混合分词和Min-Max归一化。
- **建立完整评估体系**：编写评估脚本支持批量测试，实现Top-1准确率、Hit@K召回率、片段匹配率等多维度指标；基于27个测试用例验证优化效果，并通过alpha参数调优实验找到最佳配置（纯向量检索达到81.48%准确率）。
- **完善工程化实践**：新增配置项支持灵活部署，引入 `FAISS` 作为本地向量索引 backend，支持索引文件与 metadata sidecar 持久化；编写多篇技术文档并沉淀核心代码，使 RAG 从纯内存检索演进到更可复用的工程形态。

## RAG优化配套说法

如果面试官问：”你在这个项目里做了哪些技术优化？”

可以这样回答：

> 我主要做了RAG检索系统的优化。原来只有纯向量检索和内存级索引，对于小规模知识库效果还行，但可扩展性和持久化能力不足。我实现了几个核心功能：
> 
> **第一个是重排序**。采用两阶段检索策略，先用向量检索快速召回20个候选文档，再用cross-encoder模型对这些候选精排，返回最终的4个结果。这样既保证了召回速度，又提升了排序精度。我实现了三种reranker，还做了自动降级机制。
> 
> **第二个是混合检索**。自己实现了BM25算法，把向量检索和关键词检索结合起来。通过alpha权重参数可以灵活调节两种方法的比例，比如alpha=0.6就是向量60%、BM25 40%。这对专有名词、代码片段这类需要精确匹配的场景特别有用。
> 
> **第三个是FAISS向量索引持久化**。我把RAG的向量 backend 从单纯 memory 模式扩展成可选 `faiss` 模式，让知识库 embedding 后的索引能够落盘保存，减少服务重启后的重复构建成本，也让知识库检索更接近真实工程系统。
> 
> 我还写了完整的评估脚本，基于27个测试用例验证效果。虽然当前小规模知识库下纯向量检索已经有81%准确率，但这套架构为未来扩展到大规模场景做好了准备。

如果面试官问：”为什么选择两阶段检索而不是直接用reranker？”

可以这样回答：

> 主要是性能和成本的权衡。Cross-encoder虽然精度高，但计算成本也高，如果对所有文档都做cross-encoder打分会很慢。两阶段策略是先用轻量的向量检索快速过滤掉大部分不相关文档，然后只对前20个候选做精排，这样既保证了精度，延迟也控制在可接受范围（增加50-100ms）。这是工业界比较成熟的做法，像搜索引擎也是类似的多阶段架构。

如果面试官问：”你怎么评估优化效果的？”

可以这样回答：

> 我建立了完整的评估体系。首先准备了27个测试用例，每个用例包含查询、期望标题和期望片段。然后定义了三个指标：Top-1准确率（第一个结果命中率）、Hit@K准确率（前K个结果的召回率）、片段匹配率（内容质量）。
> 
> 我写了评估脚本可以批量跑测试，对比不同配置的效果。比如我测试了alpha从0.3到0.8的不同值，发现当前知识库下alpha=0.8（接近纯向量）效果最好。这个发现也很有意义，说明小规模数据集上语义检索比关键词检索更有效，但这套架构已经为大规模场景做好了准备。
