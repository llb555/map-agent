## 3. 面试口语稿终稿

### 3.1 一分钟版

> 我最近重点准备的是大模型应用开发方向，其中一个代表项目是 Arcadegent。  
> 它是一个面向音游机厅搜索和导航场景的 AI 应用，后端基于 FastAPI，前端基于 React，核心能力包括多轮对话、Agent 编排、RAG 检索、FAISS 向量索引、地图服务接入和 SSE 流式反馈。  
> 我在这个项目里主要做的是系统链路理解和功能改造，比如梳理主 Agent、worker、工具调用、working memory 这套流程，也补齐了图片输入到模型的多模态链路，并处理了不同模型 provider 在视觉能力上的兼容问题。  
> 这个项目让我比较系统地掌握了大模型应用开发里常见的对话系统、RAG、FAISS 向量索引、工具调用和工程化落地方式。  

### 3.2 三分钟版

> 我最近重点准备的是大模型应用开发方向，想找的是 Agent、RAG、向量检索、智能问答这类偏工程落地的岗位。  
> 我有一个比较能代表这个方向的项目叫 Arcadegent，它本质上是一个垂直场景 AI 应用，场景是音游机厅搜索、附近推荐和路线规划。用户可以通过自然语言提问，系统会调用不同的 worker 和工具来完成检索、地理解析、路线建议，然后通过 SSE 把执行过程和结果实时推回前端。  
> 在技术上，它不是简单地调一次模型 API，而是有 main agent、search worker、navigation worker、tool registry、session store、context builder 和 provider adapter 这些模块。我重点理解并梳理了从前端提交问题，到后端创建 session，再到 Agent 调度工具、更新 working memory、最后把结果通过 SSE 回推前端的完整链路。  
> 同时，项目的知识库能力不是只停留在“知道 RAG”，而是做了文档切分、向量化和 FAISS 本地索引持久化，让知识检索更接近真实工程系统。  
> 另外我还做了一个比较有代表性的能力扩展，就是图片输入。原来系统虽然支持上传图片，但模型并没有真正看到图片内容。我补齐了图片进入模型消息的多模态传递链路，随后在联调中发现当前 DeepSeek 接口并不支持 `image_url` 格式，所以我又加了一层视觉 provider 自动分流和友好降级提示。这个过程让我对 provider 差异、能力边界和应用层兼容设计有了更具体的理解。  
> 所以如果用一句话总结，我比较适合的是大模型应用工程、Agent 编排、RAG、FAISS 向量检索、工具调用以及多模态接入这类方向。
