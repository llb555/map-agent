## 1. 项目定位类追问

### Q1：这个项目和普通聊天机器人有什么区别？

**参考回答**

> 最大区别是它不是只返回一段文本，而是一个带结构化工具执行链路的业务型 Agent。  
> 它会先识别用户意图，再调用搜索、地理解析、路线规划等工具，把结构化结果写入 working memory，最后输出文本回答，同时通过 SSE 把过程事件和地图 artifacts 推给前端。  
> 所以它的核心价值在于“推理 + 工具 + UI 展示”的闭环，而不是模型单点输出。  

**面试官为什么问**

- 看你是不是只会说“我接了个大模型 API”
- 看你能不能说出业务型 Agent 和普通问答机器人的差异

---

### Q2：这个项目最核心的技术难点是什么？

**参考回答**

> 我觉得核心难点不在某一个算法点，而在于把多轮对话、Agent 编排、工具调用、会话状态和前端流式展示稳定地串起来。  
> 其中一个比较典型的难点是 provider 兼容问题。很多服务都号称兼容 OpenAI，但在工具调用、多模态消息格式和接口能力上会有差异。比如图片输入这块，我就遇到了 DeepSeek 当前接口不接受 `image_url` 的问题，最后通过视觉 provider 分流和友好降级来处理。  

**继续追问可能是**

- 那你怎么判断是 provider 问题，不是你代码的问题？

**接法**

> 我是先确认前端上传和后端解析是否完整，再确认模型消息里是否真的带上了图片内容。如果链路完整，但 provider 返回的是“unknown variant image_url, expected text”这类明确的反序列化错误，就说明问题不在业务链路，而在 provider 能力边界。

## 2. Agent 与架构类追问

### Q3：为什么要设计 `main_agent` 和 worker？

**参考回答**

> 因为这个项目里不同任务的职责不一样。  
> 主 Agent 更像调度器，负责理解用户当前是搜索、附近推荐还是导航，以及决定是否需要调用 worker。  
> search worker 和 navigation worker 更像执行角色，各自关注的工具、上下文重点和最终目标不同。这样拆开以后，prompt 边界更清晰，也更方便做 allowed tools 控制和后续扩展。  

**如果继续追问：为什么不让一个 Agent 全做完？**

> 一个 Agent 全做完虽然实现更快，但会把意图理解、工具决策、结果组织全混在一起。随着工具数量和场景增加，prompt 会越来越杂，行为也更难控。拆成主控和 worker 的好处是角色清晰、上下文更聚焦、调试也更容易。  

---

### Q4：这个项目里的 Agent 真的是“智能体”吗，还是只是流程编排？

**参考回答**

> 我会比较实事求是地定义它：它是一个业务型 Agent 系统，本质是基于大模型的状态驱动编排。  
> 它不是完全自主的通用智能体，但它确实具备了三个关键特征：第一，有基于上下文的决策；第二，有外部工具调用能力；第三，有跨步骤的状态记忆和结果回填。  
> 所以从工程角度看，它已经符合很多公司对 Agent 应用的定义。  

---

### Q5：`working memory` 在这里起什么作用？

**参考回答**

> `working memory` 的作用是把工具结果、上下文状态和中间产物在一个会话里串起来。  
> 比如搜索结果、路线、用户位置、最后一次工具结果、知识检索命中都可以放进去。这样后面的步骤就不用重新解析之前的结果，主 Agent 生成回复时也能直接读取这些结构化信息。  
> 如果没有这一层，很多上下文只能靠自然语言 history 传递，会更脆弱，也更浪费 token。  

---

## 3. RAG 类追问

### Q6：这个项目里的 RAG 解决了什么问题？

**参考回答**

> 主要解决模型本身不知道项目私有资料的问题。  
> 比如门店补充说明、文档资料、FAQ、运营内容，这些信息不会稳定地存在于模型参数里。RAG 的作用是先把相关资料切分、向量化并写入 FAISS 索引，再把命中的片段作为上下文参与回答，这样既提高回答的针对性，也比把所有文档直接塞给模型更可控。  

---

### Q7：为什么不直接把所有知识库内容拼进 prompt？

**参考回答**

> 主要是成本、上下文长度和噪声控制的问题。  
> 如果把所有文档都塞进去，首先会浪费 token，其次模型会看到很多不相关内容，反而容易降低回答质量。RAG 的核心就是先检索，再只提供与当前问题强相关的片段，让上下文更精简也更有效。  

---

### Q8：你这里算不算用了向量数据库？

**参考回答**

> 可以说项目里已经用了 FAISS 作为本地向量索引 backend，所以我不是只停留在“知道向量数据库概念”，而是做了实际接入和工程化使用。  
> 当然我也会实事求是地区分边界：这更适合表述成“做过 FAISS 本地向量索引和 RAG 检索增强”，不一定等同于做过大规模托管向量数据库平台。  
> 如果岗位特别看重这块，我会进一步说明 FAISS 更适合单机、本地、低运维成本场景，而 pgvector、Milvus 更适合更强的数据管理或服务化需求。  

---

### Q8.1：为什么这里选择 FAISS，而不是 pgvector 或 Milvus？

**参考回答**

> 这个项目当前更偏应用工程和本地可运行能力，所以 FAISS 是一个性价比很高的选择。  
> 它的优点是接入轻、检索快、单机部署简单，适合把知识索引直接持久化到本地文件，降低额外基础设施依赖。  
> 如果系统后面演进到更强的多用户、服务化管理、在线更新或和数据库深度结合的场景，再考虑 pgvector 或 Milvus 会更自然。  

---

### Q8.2：FAISS 在这个项目里具体解决了什么问题？

**参考回答**

> 它主要解决的是知识库向量索引的持久化和检索工程化问题。  
> 早期如果只用 memory backend，服务重启后索引需要重建，知识库规模一大也不够稳。引入 FAISS 之后，向量索引可以落到本地文件，知识库上传和删除之后重建索引也更可控，项目在“可运行 demo”和“可持续使用的工程系统”之间就更进了一步。  

---

## 4. 多模态与 Provider 兼容类追问

### Q9：你做的图片能力具体是什么？

**参考回答**

> 原来系统虽然支持上传图片，但图片只是作为附件存在，模型其实看不到图片内容。  
> 我做的事情是把图片从“文件存在”升级成“真正进入模型消息的多模态输入”，也就是让当前轮 user message 不只是文本，还能包含图片内容。  
> 在这个过程中我还处理了 provider 差异问题，因为不同兼容接口对图片消息格式的支持不一致。  

---

### Q10：为什么会想到做视觉 provider 分流？

**参考回答**

> 因为联调时发现当前 DeepSeek 接口会直接拒绝 `image_url`，报的是明确的消息格式错误。  
> 这说明问题不是模型答不好，而是当前 provider 根本不接受图片输入。  
> 如果继续把所有请求都发给主模型，用户只会看到底层 400 错误。所以更合理的做法是：纯文本继续走原 provider，带图片的请求优先走视觉模型；如果没有视觉模型配置，就返回明确的人话提示。  

---

### Q11：你怎么判断什么时候走视觉模型？

**参考回答**

> 我在 provider router 里做了一个很轻量的路由判断：只要当前消息内容里包含图片块，就优先走 vision provider；否则继续走 primary provider。  
> 这样做的好处是侵入性小，不会影响原有纯文本链路，也方便未来继续扩展到文档理解或其他多模态场景。  

---

### Q12：这算不算推理优化？

**参考回答**

> 如果按底层推理框架的定义，这不算 TensorRT 或 GPU 层面的推理优化。  
> 但它属于应用层的能力优化和调用链优化，因为它让请求能更合理地匹配到具备相应能力的模型，并且避免无效调用和用户看到原始报错。  
> 我会把它定义成 provider routing 和能力分流，而不是底层推理加速。  

---

## 5. 前后端与 SSE 类追问

### Q13：为什么要用 SSE，而不是普通同步返回？

**参考回答**

> 因为这个项目的执行过程是多阶段的，不只是“问一句答一句”。  
> 用户发起请求后，系统可能要经历会话启动、子 Agent 切换、工具调用、路线生成、token 流式返回等多个阶段。如果全部等后端跑完再返回，用户会感觉系统很慢，也完全看不到中间过程。  
> SSE 很适合这种后端单向推送的场景，前端可以实时展示进度和最终结果。  

---

### Q14：为什么不用 WebSocket？

**参考回答**

> 因为这里的通信模式本质上是服务端单向推送，前端不需要高频双向交互。  
> SSE 的实现和接入成本更低，浏览器原生支持 `EventSource`，自动重连和事件回放机制也比较适合会话场景。  
> WebSocket 更适合强双工、高频交互的系统，比如协同编辑或多人实时聊天，但对这个项目来说有点重。  

---

### Q15：前端状态管理为什么要用 Zustand？

**参考回答**

> 因为这个项目前端有不少跨组件共享状态，比如当前会话、历史会话、流式回复、地图 artifacts、输入框状态、执行阶段等。  
> Zustand 比较轻，心智负担低，适合这种中等复杂度的状态管理场景。  
> 如果用纯 props 传递会很快变得混乱；如果一开始上更重的方案，又有点过度设计。  

---

## 6. 工程判断与设计取舍类追问

### Q16：你在这个项目里最像真实工作的一个问题是什么？

**参考回答**

> 我觉得是多 provider 兼容和多模态输入这块。  
> 因为它不是一个“代码写错就修一下”的问题，而是需要区分业务链路、消息格式、provider 能力边界和用户体验。  
> 最后方案不是简单修 bug，而是做成一个更稳的产品行为：支持时走视觉模型，不支持时给明确降级提示。这个思路很像真实工作里的工程取舍。  

---

### Q17：如果继续扩展这个项目，你会优先做什么？

**参考回答**

> 如果从岗位相关性出发，我会优先做三件事。  
> 第一，补一层更清晰的模型能力配置，比如文本模型、视觉模型、检索模型分开配置。  
> 第二，把评估做起来，比如针对搜索命中率、RAG 命中质量、FAISS 召回效果和多模态回答稳定性建立测试集。  
> 第三，如果知识规模继续变大，再考虑更明确的向量存储、召回策略和索引更新机制设计。  

---

### Q18：这个项目最大的局限是什么？

**参考回答**

> 如果从求职表达角度说，最大的局限是它更偏大模型应用工程，而不是模型训练或底层推理。  
> 所以它很适合证明我会 Agent、RAG、对话系统和工程化落地，但不适合单独证明我做过大规模微调平台或 GPU 级推理优化。  
> 我会明确承认这个边界，然后用原理理解和其他学习准备去补齐。

## 7. 如果被问到你没真正做过的点，怎么稳住

### Q19：你做过微调吗？

**参考回答**

> 这个项目主线不是微调，重点是应用编排、RAG 和多模型集成。  
> 但我理解微调和 prompt / RAG 的分工，比如当任务风格稳定、标签明确、需要更强一致性时，SFT 或 LoRA 会更合适；而这个项目里很多问题更适合通过工具调用和外部知识增强来解决，所以没有把微调作为主线方案。  

---

### Q20：你做过底层推理优化吗？

**参考回答**

> 如果是算子融合、TensorRT、KV Cache 管理、vLLM 那类底层推理优化，这个项目不是主线。  
> 我做得更多的是应用层的优化和编排，比如 provider routing、消息格式兼容、上下文组织和回退机制。  
> 我会如实区分这两层，不会把应用层优化说成底层引擎优化。

## 9. 前后端联动类追问

### Q21：你怎么用一句话讲清前端、后端和 Python 是怎么联动的？

**参考回答**

> 前端负责发请求和展示，FastAPI 负责接请求和推事件，Python 负责查数据、跑 Agent、调工具，再把结果通过 JSON 或 SSE 返回给前端。  

**如果继续追问：能不能展开一点？**

> 用户在前端输入问题后，前端会先调用会话派发接口创建或续用 session，然后订阅 SSE 事件流。后端收到请求后由 Python 里的 orchestrator 和 ReactRuntime 负责编排 agent、worker 和工具调用，过程中不断推送事件，前端再把这些事件渲染成回复、进度和地图结果。  

---

### Q22：为什么后端要有 `build_container` 这一层？

**参考回答**

> 因为项目里有很多长生命周期依赖，比如数据仓库、session store、tool registry、RAG 服务、Agent runtime、地图服务。如果每个接口自己 new 一遍，不仅重复，还会让依赖关系散掉。  
> `build_container` 的作用就是把这些对象集中装配好，让 API 层通过依赖注入拿来用。这样更利于测试替换、环境切换和模块边界清晰。  

**面试官为什么问**

- 看你是否理解“组合根”和依赖装配
- 看你是不是只会看接口，不理解系统层次

---

### Q23：为什么前端状态管理要集中在 controller hook 和 store，而不是散在组件里？

**参考回答**

> 因为这个项目的状态不是普通表单状态，而是“异步会话 + SSE + 地图 artifacts + 历史会话恢复”的组合。  
> 如果把状态分散在很多组件里，很容易出现会话切换、流式事件、地图更新不同步的问题。所以项目用了 `useChatSessionController` 统一收口业务流程，再用 Zustand store 保存共享状态，这样更容易控制状态一致性。  

---

## 10. 地图与定位类追问

### Q24：为什么地图结果不用模型直接输出 HTML 或组件，而是用结构化 artifacts？

**参考回答**

> 因为地图展示属于高约束 UI，不适合让模型自由生成 HTML 或 TSX。  
> 如果把地图渲染交给模型，结果会很不稳定，也不利于安全和维护。所以这里采用的是“assistant 继续输出正文，地图由后端返回结构化 artifacts，前端用固定 React 组件渲染”的模式。  
> 这样模型只负责内容理解和生成，地图展示则由可控的前端组件完成。  

---

### Q25：`map artifacts` 在这个项目里解决了什么问题？

**参考回答**

> 它解决的是“模型文本回答”和“地图展示结果”如何稳定协同的问题。  
> 纯文本回答不适合承载结构化地图信息，所以后端会单独返回 `shops`、`route`、`client_location`、`destination`、`view_payload` 这类结构化字段，前端再统一组装成 `ChatMapArtifacts` 去驱动地图组件。  
> 这样做的好处是地图展示和文本解耦，但又能围绕同一个会话协同更新。  

---

### Q26：为什么浏览器定位只在前端拿坐标，逆地理放到后端做？

**参考回答**

> 主要有两个原因。  
> 第一，浏览器原生更适合拿用户坐标，但不应该把高德 REST key 暴露给前端。  
> 第二，坐标和地区解析是两种不同职责，前端负责采集定位，后端负责调用高德逆地理并补充省市区等信息，这样密钥安全和职责边界都更清楚。  

**继续追问：那定位失败怎么办？**

> 项目做了降级。浏览器拒绝定位或 HTTPS 条件不满足时，不会阻塞聊天；如果有缓存就优先复用缓存，没有的话就只是不注入 location，上下文里少一部分地区信息，但对话仍然能继续。  

---

### Q27：为什么要把 `client_location` 写进 session memory？

**参考回答**

> 因为定位不是每一轮都能稳定重新拿到，而且用户很多问题是连续的，比如第一轮问附近机厅，第二轮继续导航。如果只在当前请求里带一次坐标，后面轮次就断了。  
> 写进 session memory 后，即使后续轮次没有重新上传定位，runtime 也还能从 memory 里恢复这一份上下文。  

---

## 11. 工具系统与 MCP 类追问

### Q28：为什么要做动态工具注册，而不是直接写死工具列表？

**参考回答**

> 因为项目后面既要支持内建工具，也要支持 MCP 外部工具。如果全写死在代码里，新增一个工具就要改很多地方，扩展性和治理性都不好。  
> 动态工具注册的思路是把工具 schema、executor 和 manifest 拆开，让 provider 在启动时自动装配，这样工具系统更接近插件式结构。  

---

### Q29：新增一个 builtin tool 时，你们现在的接入方式是什么？

**参考回答**

> 现在的结构更接近插件式：  
> 先在 `tools_manifest.json` 里声明要加载哪个 tool JSON；  
> 再给这个工具写自己的 JSON Schema 和 executor；  
> `BuiltinToolProvider` 启动时会自动装配这些定义。  
> 这样新增工具不需要再回到一个巨大的中心 handler 文件里做很多手工登记。  

---

### Q30：MCP 工具接入最容易出什么问题？

**参考回答**

> 最常见的问题不是“连不上”，而是 schema 和命名变化。  
> 比如路线工具在不同服务里名字不一定完全一样，如果本地代码写死工具名，对方一改就挂。所以项目做了 discovery、本地命名投影，以及 `route_tool_name` 这种显式配置，来减少对远端命名的硬耦合。  

---

### Q31：如果路线规划突然失效，你会怎么排查？

**参考回答**

> 我会先看 `/health`。  
> 因为这里已经把 MCP discovery、可用工具列表、选中的路线工具和 last_error 暴露出来了。  
> 如果 MCP 路线工具不可用，再看 REST fallback 是否正常；如果 REST 也失败，最后看是不是已经退化成离线直线距离估算。  
> 也就是说我会按 MCP -> REST -> offline fallback 这条链路逐层排查。

## 12. 会话设计与边界类追问

### Q32：为什么这个项目要做 session，而不是每次请求都无状态处理？

**参考回答**

> 因为这个项目不是单轮问答，而是典型的多轮业务对话。  
> 搜索结果、路线、定位、知识命中、执行阶段这些状态都需要跨轮复用。如果每次都无状态处理，就要么重复计算，要么把大量上下文重新塞给模型，成本和稳定性都不好。  
> 所以 session 在这里既是用户体验需要，也是 runtime 状态管理需要。  

---

### Q33：怎么保证不同用户或不同标签页之间的会话不会串？

**参考回答**

> 项目里除了 `session_id`，还有 `client_id` 这个维度。  
> 会话列表、会话详情和继续对话时都可以按 `client_id` 做作用域控制，后端也有对应的 ownership 校验。这样即使多个客户端都在使用同一服务，也不会轻易串会话。  

---

### Q34：这个项目为什么要有 fallback reply？

**参考回答**

> 因为真实系统里不能假设模型每次都会稳定返回完整答案。  
> 如果某一步失败了，API 仍然应该给用户一个可解释的结果，而不是完全空白或直接抛底层异常。所以 runtime 里设计了 fallback reply：当模型文本为空、但 route、shops 或 last_error 已经存在时，系统仍然能给出一个最低可用的用户回复。  

---

## 13. 可观测性与排障类追问

### Q35：这个项目里哪些设计是为了“更容易排障”？

**参考回答**

> 有几类设计明显是为了可观测性。  
> 第一是 SSE 事件流，让执行阶段可见；  
> 第二是 `/health`，把 tool provider、MCP discovery、路线工具状态暴露出来；  
> 第三是 session state 和 replay buffer，可以回看会话里发生过什么；  
> 第四是多层 fallback，让失败不至于直接黑盒化。  

---

### Q36：如果面试官问“你怎么证明自己不只是会写代码，还会做工程判断”，你会拿什么例子？

**参考回答**

> 我会拿三个例子。  
> 第一，地图不用模型自由输出，而是用结构化 artifacts 驱动固定组件，这体现了边界控制。  
> 第二，浏览器定位和逆地理解耦，前端拿坐标、后端持有 key 并补全地区信息，这体现了安全和职责分离。  
> 第三，多模态图片请求做 provider routing，而不是把所有请求都硬塞给主模型，这体现了能力匹配和故障隔离。

## 14. 最后建议

如果你要结合 `guidings/` 和 `dev-details/` 做二面准备，最值得重点补的就是这几类问题：

- 前后端联动怎么讲
- 地图 artifacts 为什么这么设计
- 定位与逆地理为什么拆前后端
- 动态工具注册为什么比写死工具更合理
- MCP 工具为什么需要 discovery 和可观测性
- session / client_id / fallback reply 为什么是工程必需品

这些问题的价值在于，它们能把你从“会讲 RAG 和 Agent”进一步拉到“会讲系统设计和工程判断”。

---

## 15. 结尾提醒

追问题其实不怕多，怕的是你一开始把项目说得太大。

最稳的打法是：

- 把项目定义为 **大模型应用工程项目**
- 主打 **Agent / RAG / 对话系统 / 工程化**
- 多模态当作 **亮点扩展**
- 微调和底层推理 **如实讲边界**

这样你后面的每一个追问，基本都还能接住。
