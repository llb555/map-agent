# Arcadegent 岗位应聘准备手册

这份文档面向一个具体目标：

- 你想用 `Arcadegent` 这个项目去应聘偏 **大模型应用开发 / Agent / RAG / 智能问答** 方向的岗位
- 你希望知道这个项目 **能证明什么**
- 你希望把它整理成 **简历可写、面试可讲、追问能接住** 的材料

本文按“岗位要求拆解 -> 项目匹配 -> 风险补位 -> 面试表达 -> 行动清单”的顺序整理。

---

## 1. 先说结论

`Arcadegent` 这个项目可以用于应聘，而且比较适合用于应聘以下类型岗位：

- 大模型应用开发工程师
- AI 应用工程师
- Agent / RAG / 智能问答开发工程师
- LLM 平台集成 / 对话系统开发工程师

它最能证明的是：

- 你理解 **LLM 应用工程链路**
- 你做过 **Agent 编排**
- 你接触过 **RAG**
- 你做过 **多轮对话系统**
- 你能完成 **前后端联动 + API 集成 + 工程化落地**

它相对不够强的地方是：

- 不能单独强证明你做过 **模型训练**
- 不能单独强证明你做过 **大规模微调**
- 不能单独强证明你做过 **底层推理加速**

所以最合适的包装方式不是：

> 我做了一个大模型底层平台

而是：

> 我完成了一个基于 LLM、Agent、RAG 和地图服务的垂直场景 AI 应用，重点负责系统链路理解、功能改造、能力补齐和工程化落地。

## 8. 面试时最值得主动强调的亮点

建议你主动强调下面这些点，而不是泛泛地说“我做了一个 AI 项目”。

### 8.1 你理解完整请求链路

面试官最怕候选人只会背词，不知道系统怎么跑。

你可以讲：

- 前端如何发起会话
- 后端如何防止同 session 并发执行
- Runtime 如何组织 context
- LLM 如何决定调哪个工具
- 工具结果如何回填 memory
- SSE 如何把中间阶段回推前端

### 8.2 你知道 Agent 不是魔法

你要让面试官感觉到：

- 你知道 Agent 其实是“Prompt + 状态 + 工具 + 控制流”
- 你知道 worker 的职责边界
- 你知道 tool calling 为什么要做 schema 和权限控制

### 8.3 你遇到过真实兼容问题

比如这次图片能力就是很好的例子：

- 不是模型“答不好”
- 而是 provider 根本不接受那种消息格式
- 于是你做了 routing 和 fallback

这非常像真实工作场景。

### 8.4 你会做工程上的权衡

比如：

- 图片请求不应该强塞给不支持视觉的主模型
- 纯文本仍然继续走原模型，避免无谓改动
- 没有视觉模型配置时，要给用户明确中文提示，而不是暴露底层 JSON 错误

这类话术很加分。

## 9. 面试官大概率会追问什么

下面这些问题你最好提前准备。

### 9.1 “这个项目里 Agent 是怎么工作的？”

推荐回答：

> 这个项目不是单一 LLM 调用，而是把不同职责拆开了。主入口是 `main_agent`，它负责理解用户当前意图，比如是搜索、附近推荐还是导航。如果任务偏执行型，就交给 `search_worker` 或 `navigation_worker`。worker 本身不会直接访问所有底层能力，而是通过 tool registry 去调用数据库查询、地理解析、路线规划等工具。整个过程中 session state 会保存 working memory，比如门店列表、路线、用户位置和最后一次工具结果，最后再由主 Agent 组织成用户可见回复。

### 9.2 “为什么要用 worker，不直接让一个 Agent 做完？”

推荐回答：

> 主要是为了职责分离和 prompt 边界清晰。搜索和导航虽然都属于同一个业务域，但对工具依赖、上下文重点和输出目标不一样。拆成 worker 以后，主 Agent 更像调度器，worker 更像具体执行角色，这样更容易做 prompt 约束、allowed tools 控制和后续扩展。

### 9.3 “RAG 在这里解决了什么问题？”

推荐回答：

> 它主要解决模型自身知识不覆盖项目私有资料的问题，比如门店补充说明、文档资料、FAQ 或运营信息。如果完全只靠模型参数，既不实时也不稳定。RAG 可以先把文档切分、向量化并写入 FAISS 索引，再把相关片段检索出来参与回答，这样既提高针对性，也比把大量原始文档直接塞给模型更可控。

### 9.4 “这个项目算不算做过多模态？”

推荐回答：

> 可以说做过应用层多模态接入。因为我补了图片附件进入模型消息的链路，也处理了 provider 在视觉能力支持上的差异。严格说这不是训练视觉模型，但属于多模态能力在应用系统里的接入、兼容和路由设计。

### 9.5 “你做过微调吗？”

推荐回答：

> 这个项目主线不是微调，重点是应用编排、RAG 和多模型集成。微调方面我理解 SFT、LoRA、QLoRA 的适用场景，比如当任务风格稳定、标注数据相对明确时，微调比单纯 prompt 更稳定；但这个项目本身更多是在应用层解决上下文、工具和外部知识的问题，所以没有把微调作为核心方案。

### 9.6 “你做过推理优化吗？”

推荐回答：

> 如果按底层推理框架的定义，这个项目不是在做 TensorRT、算子融合或 GPU 级优化。它更偏应用层优化，比如 provider 路由、消息格式兼容、上下文构造、失败回退和能力分流。我会如实区分这两层，不会把应用层优化说成底层推理加速。

## 10. 哪些地方不要夸大

这部分很重要，能帮你避免面试时“越说越虚”。

### 10.1 不要硬说“从 0 到 1 独立开发”

如果事实更接近：

- 本地复现
- 深入阅读
- 架构拆解
- 二次开发
- 功能补齐

那就如实这么说。

这并不丢人，反而更稳。

### 10.2 不要硬说“做过模型微调平台”

如果你没有：

- 数据构造
- 训练脚本
- loss/metric 设计
- checkpoint 管理
- 离线评估

就不要把这个项目往“训练平台”上靠。

### 10.3 不要把应用层兼容说成底层推理优化

你可以说：

- 我做了 provider 兼容
- 我做了多模型分流
- 我做了消息格式适配

但不要说：

- 我做了大模型推理引擎优化

### 10.4 不要乱报效果数据

如果没有明确统计，别说：

- 提升了多少准确率
- 减少了多少延迟
- 提高了多少转化

可以改成：

- 提升了系统可用性
- 降低了用户看到原始底层错误的概率
- 让图片场景具备可扩展的 provider 路由能力
