# Arcadegent 模拟面试 20 题

这份文档是给你做面试临场练习用的。

和前面的简历稿、追问题库不同，这份更像一份可以直接拿来开口练的材料。每道题都分成三层：

- `短答版`：适合先快速回答
- `展开版`：适合面试官继续深挖时接着说
- `可能追问`：提醒你下一步大概率会被问什么

建议使用方法：

1. 先把 `短答版` 背熟
2. 再把 `展开版` 练到能自然说出来
3. 最后看 `可能追问`，做模拟问答

---

## 1. 你先用 30 秒介绍一下这个项目

### 短答版

Arcadegent 是一个面向音游机厅搜索、附近推荐和路线规划的 AI 应用。前端用 React，后端用 FastAPI，核心是用 Agent 把搜索、地理解析、路线规划、知识检索和地图展示串起来，并通过 SSE 把执行过程实时推给前端。

### 展开版

它不是普通的聊天机器人，而是一个垂直场景的业务型 Agent 应用。用户用自然语言提问后，后端会先判断意图，再调搜索 worker、导航 worker 或知识检索能力，把结果写入 working memory，最后返回文字回答、门店信息和地图路线。前端不仅展示最终结果，还会通过 SSE 实时看到工具执行和回复生成过程。

### 可能追问

- 它和普通聊天机器人最大的区别是什么？

## 2. 你在这个项目里主要做了什么

### 短答版

我主要做了三类事情：一是完整梳理前后端到 Agent 的请求链路；二是补齐了图片输入到模型的多模态链路；三是把 RAG 的向量索引能力补到更工程化的 FAISS 持久化形态。

### 展开版

更具体地说，我先把前端发起会话、后端创建 session、Agent 调 worker、工具执行、working memory 回填、SSE 回推前端这条链路彻底读通了。然后我做了两个比较有代表性的改造：一个是让图片真的进入模型消息，而不是只作为附件存在；另一个是把知识库从纯 memory backend 往 FAISS 本地向量索引持久化方向推进，让 RAG 更像真实工程系统。

### 可能追问

- 哪一部分最能代表你的实际工程能力？

## 3. 这个项目和普通聊天机器人有什么区别

### 短答版

普通聊天机器人通常是输入一句、输出一段文本；这个项目是带工具调用、状态管理和地图展示的 Agent 系统，核心是“理解意图 + 调工具 + 结构化结果展示”。

### 展开版

它会先识别用户是想查机厅、找附近、做路线规划，还是问知识库问题，然后调用对应工具，把搜索结果、路线、用户位置和知识命中写进 working memory，最后再组织成自然语言回复，同时把门店和路线用地图展示出来。所以它的价值不只是模型回答本身，而是模型和工具、前端 UI、地图能力共同完成任务。

### 可能追问

- 那它到底算 Agent 还是流程编排？

---

## 4. 为什么这里要设计 `main_agent` 和 worker

### 短答版

因为不同任务的职责不一样。主 Agent 更适合做意图理解和调度，worker 更适合做具体执行，这样比一个大 Agent 包打天下更清晰、更可控。

### 展开版

如果所有事情都塞进一个 Agent，prompt 会越来越长，工具边界也会越来越模糊。现在的拆分方式是：`main_agent` 负责理解用户问题并决定怎么走，`search_worker` 负责搜索相关任务，`navigation_worker` 负责导航和路线任务。这样每个角色关注点更单一，allowed tools 也更容易控制，后续扩展新场景时成本更低。

### 可能追问

- 为什么不直接写死流程，而要保留 Agent 调度？

---

## 5. `working memory` 在这里有什么作用

### 短答版

它的作用是保存结构化中间结果，让多轮对话不只靠自然语言历史传递上下文。

### 展开版

像门店列表、路线结果、用户位置、知识检索命中、最近一次工具结果，这些都更适合放在 working memory 里。这样下一轮如果用户说“帮我导航到第一个”或者“这家店附近还有别的吗”，系统就能直接复用结构化状态，而不是从自然语言里重新猜。这样更稳定，也更省 token。

### 可能追问

- `working memory` 和 `session_state` 的区别是什么？

---

## 6. 为什么聊天链路要用异步会话加 SSE

### 短答版

因为 Agent 执行是多阶段的，如果等全部完成再返回，用户只会感觉系统卡住了。SSE 可以把执行过程实时推给前端，体验会更好。

### 展开版

这个项目里一次请求可能会经历会话创建、子 Agent 切换、工具调用、路线生成、token 输出等多个阶段。采用 `POST /api/chat/sessions` 先派发任务，再用 `GET /api/stream/{session_id}` 持续接收事件，前端就能看到 `session.started`、`tool.started`、`assistant.token`、`assistant.completed` 这些过程事件。对 Agent 产品来说，这种“过程可见”很重要。

### 可能追问

- 为什么用 SSE，不用 WebSocket？

---

## 7. 为什么这里选 SSE，而不是 WebSocket

### 短答版

因为这个场景本质上是后端单向推送，SSE 已经够用，而且实现成本更低。

### 展开版

前端主要是发起请求后订阅后端事件，不需要高频双向交互，所以 SSE 比较合适。浏览器原生支持 `EventSource`，自动重连和事件回放机制也比较适合会话场景。WebSocket 更适合协同编辑、多用户实时交互这种双工场景，但对这个项目来说偏重了。

### 可能追问

- 断线重连之后怎么恢复？

---

## 8. 如果前端断线，聊天过程怎么恢复

### 短答版

后端维护 replay buffer，SSE 事件有递增 id，前端重连后可以从上次的 event id 之后继续补发。

### 展开版

这里不是简单断了就重来，而是把事件流恢复考虑进去了。SSE 事件会带 id，前端重连时可以带 `Last-Event-ID` 或等价参数，后端从 replay buffer 里把漏掉的事件补回来。如果会话已经结束，还能通过 session store 读取最终状态，所以过程恢复和结果恢复都兼顾到了。

### 可能追问

- replay buffer 和 session store 为什么要同时存在？

---

## 9. 这个项目里的 RAG 主要解决什么问题

### 短答版

主要是解决模型不知道项目私有知识的问题，让回答能基于知识库内容，而不是只靠模型参数记忆。

### 展开版

像 FAQ、补充说明、业务文档这些信息，并不会稳定存在于模型参数里。RAG 的做法是把文档切分、向量化，再检索出和当前问题最相关的片段，作为上下文交给模型回答。这样回答会更贴近项目实际知识，同时也比把所有文档都直接塞进 prompt 更可控、更节省 token。

### 可能追问

- 为什么不直接把所有文档拼进 prompt？

---

## 10. 为什么引入 FAISS

### 短答版

因为单纯的 memory backend 更像 demo，FAISS 能把向量索引持久化到本地文件，让 RAG 更工程化。

### 展开版

如果只有内存索引，服务一重启就要重建，知识库一变大也不够稳。引入 FAISS 之后，向量索引可以落盘保存，知识库上传、删除、重建的链路也更清晰。它很适合这个阶段的单机、本地、低运维成本场景，所以我会把它定位成“本地向量索引 backend 的工程增强”。

### 可能追问

- 为什么这里不直接上 pgvector、Milvus 这类方案？

---

## 11. FAISS 和托管向量数据库分别适合什么场景

### 短答版

FAISS 更适合单机、本地、低运维成本场景；托管向量数据库更适合多用户、服务化、在线更新和更强的数据管理需求。

### 展开版

这个项目当前重点是把 RAG 跑通并工程化，不是先上复杂基础设施，所以 FAISS 性价比高。它接入轻、检索快、落盘简单。托管向量数据库或者服务化方案更适合知识规模更大、需要高并发、多租户、在线更新、权限控制或者和现有数据库深度结合的场景。面试时我会明确说自己在这个项目里更偏 FAISS 本地向量索引实践，而不是大规模向量数据库平台建设。

### 可能追问

- 以后如果业务扩大，你会怎么演进？

---

## 12. 你做的图片识别能力具体是什么

### 短答版

本质上是把图片从“只是上传了附件”变成“真正进入模型消息的多模态输入”。

### 展开版

原来系统虽然支持上传图片，但模型并没有真正看到图片内容。我补齐了当前轮 user message 的多模态内容结构，让图片作为消息块随请求一起发给支持视觉的模型。这样用户上传截图、照片之类的内容时，模型才能基于图片本身做理解，而不是只看到文本描述。

### 可能追问

- 你为什么后来又做了视觉 provider 分流？

## 13. 为什么要做视觉 provider 分流

### 短答版

因为联调时发现当前 DeepSeek 接口并不接受 `image_url` 这类图片消息格式，所以不能把带图请求继续发给原来的文本 provider。

### 展开版

这个问题的关键在于区分“业务链路错了”还是“provider 能力不支持”。我先确认前端上传、后端解析和消息组装都没问题，最后看到 provider 返回的是明确的图片消息反序列化错误，就说明不是我链路丢图，而是当前 provider 不支持这种多模态输入。所以更合理的方案是：纯文本继续走主 provider，带图片时优先走 vision provider；如果没配置视觉模型，就给用户友好降级提示。

### 可能追问

- 你怎么判断什么时候走 vision provider？

## 14. 你怎么判断是代码问题还是 provider 问题

### 短答版

我会按链路逐层排查，先确认上传和解析，再确认消息体是否真的带图，最后再看 provider 返回的具体错误。

### 展开版

这种问题不能一上来就怪模型。我会先看前端是否把图片正确上传，再看后端是否解析出了图片元信息，然后确认构造给模型的消息里是否真的包含图片块。如果这些都正常，但 provider 返回的是类似“不识别 `image_url` 变体”这类明确错误，那基本可以判断问题不在业务链路，而在 provider 的能力边界或兼容实现上。

### 可能追问

- 这种兼容问题你最后怎么对用户做体验处理？

## 15. 这个项目里的 MCP 你怎么理解

### 短答版

我会把 MCP 理解成一种标准化的外部工具接入方式，让 Agent 能动态发现和调用外部能力，而不是把所有工具都写死在系统里。

### 展开版

在这个项目里，内建工具和 MCP 工具会一起进入工具注册体系。MCP 的价值不只是“能调外部工具”，更重要的是让工具发现、schema 描述、能力注册和调用边界更标准化。这样后续接地图、路线或者其他外部服务时，不需要把所有能力都深埋在业务代码里，扩展性更好。

### 可能追问

- 动态工具注册为什么是有意义的，而不是越灵活越乱？

---

## 16. `tool registry` 的作用是什么

### 短答版

它是工具治理中心，负责统一管理工具描述、参数校验、调用边界和执行记录。

### 展开版

如果没有 registry，工具调用会散落在各个 Agent 或业务模块里，后面很难管控。现在统一收口之后，Agent 不直接接触所有底层实现，而是通过工具注册表拿到可调用能力。这样可以更方便地做 schema 校验、权限控制、工具命名管理、内建工具和 MCP 工具的统一接入，也更利于后续调试和扩展。

### 可能追问

- provider 化和 manifest 化对工具系统带来了什么好处？

---

## 17. 浏览器定位和逆地理编码为什么要拆开

### 短答版

因为浏览器定位和逆地理编码本质上是两个不同层次的问题，一个负责拿坐标，一个负责把坐标变成更可读、更可用的地理信息。

### 展开版

前端浏览器定位更接近客户端能力，负责获取用户当前经纬度；后端逆地理编码则更接近业务服务，负责把经纬度转成行政区、地标、地址信息，并写入 session memory 供后续上下文使用。拆开之后，前端和后端边界更清晰，也方便在某些情况下复用历史位置，而不是每次都重新拿浏览器定位。

### 可能追问

- 这个位置上下文最后是怎么进入模型的？

---

## 18. 前端为什么用 Zustand 做状态管理

### 短答版

因为这个项目有不少跨组件共享状态，Zustand 比较轻，适合这类中等复杂度场景。

### 展开版

前端要同时管理当前会话、历史会话、流式回复、地图 artifacts、发送状态、子 Agent 状态和筛选条件。如果只靠 props 传递会很快变乱；如果一开始上更重的状态管理方案，又会增加心智负担。Zustand 在这个项目里比较合适，既能集中管理共享状态，又不会带来过度设计。

### 可能追问

- 这里最关键的前端状态是哪几类？

## 19. 这个项目最大的难点是什么

### 短答版

我认为最大的难点不是某一个算法，而是把多轮对话、Agent 编排、RAG、地图服务、前端流式展示和 provider 兼容稳定地串成一个可用系统。

### 展开版

这个项目很典型的一点是“每一块单看都不算特别夸张，但组合起来就很考验工程能力”。比如多轮上下文不能只靠聊天历史，工具调用不能散，前端又要实时看到过程，知识库还要可维护，同时 provider 的能力边界还不统一。所以它真正难的是系统集成、状态组织、兼容处理和产品体验，而不是某一个单独模块。

### 可能追问

- 你最像真实工作的一次问题处理是什么？

## 20. 如果继续做下去，你会优先优化什么

### 短答版

我会优先做三件事：更清晰的模型能力配置、更系统的评估体系，以及更成熟的检索和索引更新机制。

### 展开版

第一，我会把文本模型、视觉模型、embedding 模型、reranker 模型的配置边界拉得更清楚。第二，我会补评估集，针对搜索命中率、RAG 召回质量、多模态回答稳定性、工具调用成功率做更系统的评估。第三，如果知识库规模继续增长，我会进一步考虑索引更新策略、混合检索、重排序和更服务化的向量存储演进路径。

### 可能追问

- 如果让你做成更像生产系统的一版，第一步会改什么？
