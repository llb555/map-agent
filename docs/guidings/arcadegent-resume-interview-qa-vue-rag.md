# Arcadegent 简历项目面试问答准备

> 适用简历描述：  
> **Arcadegent 音游机厅智能检索与路线规划系统｜AI 应用开发**  
> **技术栈：** FastAPI + Vue 3 + TypeScript + Vite + Pinia + SSE + RAG + MCP + Supabase + 高德地图 API + LLM

## 0. 面试讲述总口径

### 30 秒项目介绍

Arcadegent 是一个面向音游玩家的 AI 机厅搜索与路线规划系统。用户可以用自然语言询问“附近哪里有机厅”“某个城市哪里有舞萌”“怎么去最近的机厅”等问题。系统后端通过 Agent 编排搜索、知识库、地理解析和路线规划等工具，前端通过 SSE 实时展示执行阶段、工具状态、流式回复和地图结果。

这个项目不是简单接一个大模型 API，而是把 **Agent 编排、RAG 知识检索、地图服务、异步会话、流式前端展示、工程化测试** 串成了一个可用的业务应用。

### 1 分钟项目介绍

项目面向音游玩家，解决“去哪打机、怎么去、店铺信息是否可靠”这类问题。系统支持结构化筛选，也支持自然语言问答。后端基于 FastAPI 提供接口，通过 Agent 将用户问题分发给搜索 worker、导航 worker 和工具注册中心；工具层统一管理数据库查询、RAG 知识检索、地理解析、路线规划等能力。

前端使用 Vue 3、TypeScript、Vite 和 Pinia 管理页面状态，通过 SSE 接收后端实时事件，把 Agent 的执行过程、工具调用状态、流式回复和地图结果渐进式展示出来。数据层支持 JSONL 和 Supabase，地图能力接入高德 API，RAG 用于补充结构化数据库没有覆盖的评论、活动和文档资料。

### 一句话亮点

我做的不是“模型问答 demo”，而是一个带 **工具调用、RAG、流式会话、地图 artifacts、幂等重连和回归测试** 的 AI 应用工程化实践。

## 1. 项目基础类问题

### Q1：这个项目主要解决什么问题？

**回答：**

它解决的是音游玩家找机厅和规划路线的问题。传统方式需要用户自己去地图、社群、机厅列表里查信息，而且结构化数据和非结构化资料是分散的。这个项目把机厅搜索、附近推荐、路线规划、知识库补充和历史会话整合在一起，用户可以直接用自然语言提问，系统自动判断意图并调用对应工具。

比如用户问“上海附近有没有舞萌比较多的店”，系统会识别这是附近推荐加机种筛选的问题，再结合用户位置、机厅数据、地图信息和知识库资料生成结果。

**可能追问：为什么这个项目适合做 AI？**

**追问回答：**

因为用户的问题并不总是标准筛选条件。用户可能说“下班后附近哪里适合打舞萌”“某某商场附近有没有机厅”“帮我找最近的店并规划路线”。这些表达里包含意图理解、地点解析、业务筛选和路线规划，适合让大模型负责自然语言理解和任务分发，再由确定性的工具完成数据查询和地图计算。

---

### Q2：这个项目和普通机厅列表网站有什么区别？

**回答：**

普通列表网站更像搜索筛选页面，用户需要知道自己要筛什么。这个项目除了保留条件筛选，还增加了 Agent 问答能力。用户可以用自然语言描述需求，系统会自动调用搜索、RAG、地理解析和路线规划工具，并通过 SSE 展示执行过程。

简单说，它从“用户手动筛选”升级成了“用户提出目标，Agent 帮用户拆解并完成查询”。

**可能追问：是不是直接让模型生成结果？**

**追问回答：**

不是。模型主要负责理解意图和决策调用哪个工具，真正的机厅数据、路线数据和知识库内容都来自工具查询。这样可以避免模型凭空编造，也更容易控制结果质量。

---

### Q3：为什么不直接用 ChatGPT 做问答？

**回答：**

直接用 ChatGPT 最大的问题是它不知道项目里的实时机厅数据、用户位置、地图路线和私有知识库内容。这个项目需要把大模型和业务数据连接起来，所以采用 Agent + 工具调用 + RAG 的方式。

模型负责理解用户问题，工具负责查询真实数据，RAG 负责检索非结构化资料，地图 API 负责路线和地理能力。这样回答更可靠，也能落到具体业务动作上。

---

## 2. 整体架构类问题

### Q4：整体架构是怎样的？

**回答：**

整体是前后端分离架构。

前端负责三类体验：

- 条件筛选和地图展示
- 自然语言对话
- SSE 实时展示 Agent 执行过程

后端负责五类能力：

- 接收用户会话请求
- Agent 意图识别和任务分发
- 工具注册、校验和执行
- RAG 知识库检索
- 机厅数据、地图路线和会话状态管理

一次典型请求流程是：

1. 用户在前端提交问题
2. 后端创建异步会话并立即返回 session id
3. 前端用 session id 建立 SSE 连接
4. 后端 Agent 识别意图并调用工具
5. 每个阶段通过 SSE 推送给前端
6. 最终回复、知识引用、地图结果一起展示

**可能追问：为什么要异步会话，不直接同步返回？**

**追问回答：**

Agent 执行链路可能比较长，比如要识别意图、查数据库、查知识库、解析地理位置、规划路线。如果同步等待，用户会一直盲等，也容易超时。异步会话可以让请求先被接受，前端通过 SSE 实时看到进度，体验更好，也更容易做断线重连和历史恢复。

---

### Q5：你的项目分层是怎么设计的？

**回答：**

可以分成四层：

第一层是前端交互层，负责 Vue 页面、Pinia 状态管理、地图渲染和 SSE 消息消费。

第二层是 API 层，负责会话接口、机厅检索接口、知识库管理接口和地图相关接口。

第三层是 Agent 编排层，负责意图识别、worker 分发、working memory 管理和工具调用决策。

第四层是工具与数据层，包括数据库查询、RAG 检索、地理解析、路线规划、Supabase / JSONL 数据源和高德地图 API。

这样的好处是每层职责清晰。前端不关心 Agent 内部怎么调用工具，Agent 也不直接关心底层数据源是 JSONL 还是 Supabase。

## 3. Agent 编排类问题

### Q6：你这个 Agent 是怎么工作的？

**回答：**

项目里 Agent 不是一个大 prompt 一次性回答，而是分成主控 Agent、worker 和工具注册中心。

主控 Agent 负责判断用户意图，比如搜索、附近推荐、路线规划或知识问答。然后它把具体任务分发给对应 worker。搜索 worker 更关注机厅检索和知识库补充，导航 worker 更关注位置解析和路线规划。工具注册中心统一暴露可调用工具，控制每个 worker 能调用哪些工具。

这样做的好处是职责清楚，工具权限可控，也方便后续扩展。

**可能追问：为什么不一个 Agent 全做？**

**追问回答：**

一个 Agent 全做实现快，但问题是 prompt 会越来越长，工具边界会越来越乱。比如搜索和导航需要的上下文、工具和输出目标都不同，混在一起后模型更容易调用错工具。拆成主控和 worker 后，每个角色只关注自己的任务，行为更稳定。

---

### Q7：main agent 和 worker 的区别是什么？

**回答：**

main agent 更像调度器，负责理解用户问题、判断任务类型、决定是否调用 worker。worker 更像执行者，只处理某一类任务。

举例来说，用户问“帮我找最近的机厅并规划路线”。main agent 会判断这不是普通搜索，而是附近推荐加导航任务，于是会让搜索 worker 先找候选机厅，再让 navigation worker 做路线规划。

**可能追问：worker 之间怎么传递结果？**

**追问回答：**

通过 working memory。工具调用得到的门店、路线、位置、知识库命中等结果都会以结构化方式保存。后续 worker 或最终总结阶段可以直接读取这些结构化结果，而不是只依赖聊天文本。

---

### Q8：working memory 是什么？为什么需要它？

**回答：**

working memory 可以理解成一次会话里的结构化上下文。它保存中间结果，比如搜索到的门店列表、用户位置、路线结果、知识库片段、最近一次工具调用信息等。

如果没有 working memory，后续步骤只能从聊天历史里重新理解信息，容易丢字段，也浪费 token。用 working memory 后，Agent 的每一步都能读到稳定的结构化数据，前端也能基于这些数据渲染地图结果。

**可能追问：它和聊天 history 有什么区别？**

**追问回答：**

聊天 history 是给模型看的自然语言上下文，适合保留对话语义；working memory 是程序可读的结构化状态，适合保存工具结果和业务字段。两者是互补关系。

---

### Q9：意图识别怎么做？

**回答：**

意图识别结合了规则和模型上下文。对于明显关键词，比如“附近”“最近”“路线”“怎么去”，可以先做轻量判断。复杂表达再交给 Agent 根据上下文决定走搜索、附近推荐、导航或知识问答。

我没有把所有逻辑都交给模型，是因为一些高频业务意图用规则更稳定、更便宜；模型主要处理自然语言表达的灵活性。

**可能追问：如果模型识别错了怎么办？**

**追问回答：**

一方面通过固定问句集做回归测试，保证常见问题的意图路由稳定；另一方面工具层和结果层也有兜底，比如数据库没命中时可以补查知识库，路线缺坐标时可以返回可解释的失败原因，而不是直接崩掉。

---

## 4. 工具调用系统类问题

### Q10：工具调用系统怎么设计的？

**回答：**

工具系统是基于 manifest、JSON Schema 和 executor 组织的。

manifest 描述工具的名字、用途和参数 schema；JSON Schema 用来校验模型生成的参数；executor 是真正执行工具逻辑的地方，比如查数据库、查知识库、解析地理位置、规划路线。

工具注册中心负责统一发现工具、校验参数、检查权限、执行工具并记录结果。这样新增工具时，只需要补充工具声明和执行器，不需要改 Agent 主流程。

**可能追问：为什么要 JSON Schema？**

**追问回答：**

因为模型生成的工具参数不一定可靠。JSON Schema 可以在执行前校验字段类型、必填项和范围，比如经纬度必须是数字、分页大小不能无限大。这样能把错误挡在工具执行前，提高系统稳定性。

---

### Q11：你做了哪些工具？

**回答：**

核心工具包括：

- 数据库检索工具：根据关键词、地区、机种、距离等条件查机厅
- RAG 知识检索工具：检索文档资料、评论、活动和补充说明
- 地理解析工具：把地点、地址、用户位置转成坐标或区域信息
- 路线规划工具：调用地图能力生成路线、距离和耗时
- 总结工具：把结构化结果组织成用户能看懂的回答

这些工具不是直接暴露给用户，而是由 Agent 根据意图选择调用。

---

### Q12：工具调用如何避免失控？

**回答：**

主要从四个方面控制：

第一是权限控制，不同 worker 只能调用允许范围内的工具。  
第二是参数校验，工具执行前必须通过 schema。  
第三是超时和重试，避免某个外部服务卡住整个会话。  
第四是工具预算，限制一次会话内某类工具最多调用多少次，避免模型循环调用。

**可能追问：为什么需要工具预算？**

**追问回答：**

Agent 有时会因为结果不满意反复尝试调用工具。如果不限制，可能导致外部 API 成本升高、响应变慢，甚至触发限流。工具预算就是给 Agent 加一层工程边界。

---

### Q13：工具失败时怎么处理？

**回答：**

工具失败不会直接让整个系统崩掉。工具层会返回统一的失败结构，包括错误类型、错误信息、trace id、重试次数和 fallback 原因。Agent 可以根据这些信息决定是否换工具、降级回答，或者告诉用户当前失败原因。

例如地图路线失败时，可以保留候选机厅和地图点位；RAG 失败时，可以继续返回结构化数据库结果。

---

## 5. RAG 类问题

### Q14：RAG 在这个项目里解决什么问题？

**回答：**

结构化数据库适合保存机厅名称、地址、机种数量、地区等标准字段，但很多信息是非结构化的，比如玩家评论、活动说明、门店补充资料、攻略文档等。这些内容不适合全部塞进数据库字段里。

RAG 的作用是把这些文档切分、向量化并建立索引。用户提问时，系统先检索相关片段，再把片段作为上下文交给模型回答。这样既能利用私有知识，又能减少模型胡编。

**可能追问：为什么不直接让模型记住这些资料？**

**追问回答：**

模型参数不会实时包含项目里的私有数据，而且资料会变化。RAG 的优势是知识可以独立更新，上传文档后重新索引即可，不需要重新训练模型。

---

### Q15：RAG 的处理流程是什么？

**回答：**

一般流程是：

1. 用户上传文档或准备知识源
2. 系统解析文档内容
3. 把长文本切成 chunk
4. 对 chunk 生成 embedding
5. 写入向量索引和元数据
6. 用户提问时先检索相关 chunk
7. 把命中片段交给模型生成回答

在这个项目里，RAG 不只是一次性全量重建，而是支持按文件记录状态和增量索引。上传、删除和失败重试都可以作为任务处理。

**更完整的面试回答：**

这个项目里的 RAG 可以按“文档入库 -> 索引构建 -> 查询召回 -> Agent 使用”来讲。

第一步是文档入库。用户可以上传 Markdown、TXT、JSON、PDF、Word 等知识文件。后端会先识别文件类型，再把文档解析成纯文本。不同格式的处理方式不一样，比如文本类文件可以直接读取，JSON / JSONL 会提取其中的正文内容，PDF 和 Word 需要先解析文档结构。

第二步是切分和索引。系统不会把整篇文档直接塞给模型，而是把文本切成多个 chunk，并给每个 chunk 记录来源文件、标题、页码、source_uri 等 metadata。然后对 chunk 生成 embedding，写入向量索引。

第三步是检索。用户提问时，如果 Agent 判断需要知识库补充，就会调用知识检索工具。系统会把用户问题转成 query，对 query 生成 embedding，然后在向量索引里召回相关 chunk，也可以结合关键词检索做混合召回。

第四步是给 Agent 使用。检索命中的片段不会直接原样返回给用户，而是写入 working memory。最终 Agent 会结合结构化数据库结果、知识库片段和当前问题生成回答。

**简洁版表达：**

> 项目里的 RAG 主要用于补充结构化机厅数据库没有覆盖的评论、活动和文档资料。实现上，后端支持多格式文档上传，解析文本后按 chunk 切分，生成 embedding，并写入内存或 FAISS 向量索引。查询时，Agent 通过知识检索工具把用户问题转成 query，召回相关 chunk，并把命中片段写入 working memory，最后结合数据库结果和知识片段生成回答。为了让它更接近真实产品，我还做了文件级 pending / indexing / ready / failed 状态、增量索引、失败重试和混合检索能力。

---

### Q16：为什么要做增量索引？

**回答：**

如果每次上传或删除文档都全量重建，文件多了会很慢，也浪费 embedding 成本。增量索引的思路是按文件计算状态和内容 hash，只有变化的文件重新处理，没变的 chunk 可以复用。

这样用户上传一个新文档时，不需要把整个知识库都重新跑一遍，系统更接近真实可用的产品。

**可能追问：增量索引有哪些状态？**

**追问回答：**

通常会有 pending、indexing、ready、failed。pending 表示等待处理，indexing 表示正在索引，ready 表示可检索，failed 表示失败并保留失败原因，用户可以单独重试。

---

### Q17：PDF 是怎么处理的？

**回答：**

PDF 在项目里主要进入知识库 RAG 流程。处理流程可以概括为：

```text
上传 PDF
  ↓
读取 PDF 页面文本
  ↓
按页提取正文
  ↓
文本清洗
  ↓
切分 chunk
  ↓
生成 embedding
  ↓
写入向量索引
  ↓
用于 RAG 检索
```

用户上传 PDF 后，后端会识别文件类型，然后用 PDF 解析库逐页读取文本。提取出的文本会保留 metadata，比如文件名、页码、source_uri 和文档类型。之后它会进入统一的 RAG 流程：文本清洗、chunk 切分、embedding 生成和索引写入。

这样用户提问时，如果命中 PDF 里的内容，系统就能把对应片段作为知识上下文交给 Agent。

**可能追问：为什么不直接把 PDF 给模型？**

**追问回答：**

因为 PDF 可能很长，直接塞给模型会超出上下文窗口，也会带来很多和问题无关的噪声，成本也高。RAG 的做法是先解析和索引，查询时只召回相关片段，这样更精准，也方便追踪来源。

**可能追问：扫描版 PDF 怎么办？**

**追问回答：**

当前更适合处理有文本层的 PDF。如果 PDF 是扫描件，本质上页面是图片，普通文本提取拿不到内容。后续可以接 OCR，比如 PaddleOCR、Tesseract、云 OCR 或多模态视觉模型，先把图片页识别成文本，再进入同一套 chunk 和 embedding 流程。

**可能追问：PDF 处理有什么难点？**

**追问回答：**

PDF 的难点主要是文本顺序、表格、分栏、页眉页脚和扫描件。普通文本文件可以直接读取，PDF 需要先做结构解析，而且页面里视觉上的阅读顺序不一定等于提取出来的文本顺序。所以项目里更关注“提取可检索正文 + 保留页码来源 + 出错可追踪”，而不是盲目追求完全还原版式。

**简洁版表达：**

> PDF 进入知识库后，我会先识别文件类型，用 PDF 解析库逐页提取文本，并保留文件名、页码、source_uri 等 metadata。提取出的文本不会直接塞给模型，而是进入统一 RAG 流程：文本清洗、chunk 切分、embedding 生成和向量索引。用户查询时只召回相关 chunk，并把命中片段提供给 Agent 生成回答。当前主要支持有文本层的 PDF，如果是扫描版 PDF，可以扩展 OCR，把图片页识别成文本后再进入同一套索引流程。

---

### Q18：混合检索是什么？为什么需要？

**回答：**

混合检索就是把向量检索和关键词检索结合起来。

向量检索擅长语义相似，比如“店铺评价怎么样”和“玩家反馈不错”语义接近；关键词检索擅长精确命中，比如店名、机种名、城市名。如果只用一种方式，召回可能不稳定。

混合检索可以先分别得到候选结果，再用融合算法排序，提高召回质量。

**具体怎么做：**

一次知识库查询大概是这样：

1. 用户问题进入知识检索工具
2. 对用户 query 生成 embedding
3. 用 query embedding 去向量索引里找语义相似 chunk
4. 同时用 BM25 / 关键词检索找字面匹配 chunk
5. 两路检索分别得到候选结果
6. 用融合算法合并候选
7. 对相同 chunk 去重
8. 返回 top-k 片段给 Agent

可以理解成：

```text
用户问题
  ├─ 向量检索：找语义相似内容
  ├─ BM25 检索：找关键词命中内容
  ↓
候选结果融合排序
  ↓
Top-K 知识片段
  ↓
写入 working memory，交给 Agent 生成回答
```

**可能追问：RRF 是什么？**

**追问回答：**

RRF 是 Reciprocal Rank Fusion，简单理解就是按不同检索器里的排名给结果打分，再把分数融合。它不强依赖不同检索器的原始分数尺度，所以比较适合把向量检索和关键词检索结果合并。

常见公式可以简单理解为：

```text
score = 1 / (k + rank)
```

如果一个 chunk 在向量检索里排第 2，在 BM25 里排第 5，那么它的融合分数就是：

```text
1 / (k + 2) + 1 / (k + 5)
```

最后按融合分数排序。

RRF 的好处是不用强行比较“向量相似度分数”和“BM25 分数”。因为这两个分数的尺度不同，直接相加不太合理。RRF 只看排名，所以更稳定。

**举例说明：**

用户问：

> 广州有没有维护比较好的舞萌店？

向量检索可能召回“玩家反馈机器维护不错”“店员会定期调试设备”这类语义相关片段；BM25 可能更容易命中“广州”“舞萌”“维护”这些关键词。融合后，系统更倾向于保留既和“维护好”语义相关，又包含“广州 / 舞萌”等关键实体的片段。

**简洁版表达：**

> 项目里的混合检索是把向量检索和 BM25 关键词检索结合起来。向量检索负责语义召回，适合处理“评价怎么样、维护好不好”这类表达；BM25 负责精确词召回，适合店名、城市、机种名这类关键词。查询时系统会同时跑两路检索，然后用 RRF 按排名融合结果，去重后返回 top-k chunk 给 Agent。这样比单独使用向量检索更稳，既能理解语义，也能保证关键实体不丢。

---

### Q19：RAG 如何避免回答胡编？

**回答：**

主要有三点：

第一，回答尽量基于检索到的片段，而不是让模型自由发挥。  
第二，保留知识来源和片段摘要，前端可以展示引用或线索。  
第三，当检索结果不足时，系统应该明确说“不确定”或引导用户补充信息，而不是强行编答案。

---

### Q20：chunk 怎么切？切太大太小有什么问题？

**回答：**

chunk 太大，检索结果里会带很多无关内容，浪费上下文；chunk 太小，语义可能被切碎，模型拿不到完整信息。

所以一般要根据文档类型设定合适长度，并保留一定 overlap。比如 FAQ 可以切得短一点，长文档可以按段落或固定长度切，同时保留少量重叠，避免上下文断裂。

---

## 6. SSE 和异步会话类问题

### Q21：为什么使用 SSE，而不是 WebSocket？

**回答：**

这个项目的流式通信主要是服务端向客户端单向推送，比如执行阶段、工具状态、token 片段和最终结果。SSE 天然适合这种场景。

相比 WebSocket，SSE 更简单，基于 HTTP，浏览器原生支持 EventSource，断线后也有 Last-Event-ID 这类重连机制。WebSocket 更适合双向高频通信，比如多人协作或实时游戏。这个项目用 SSE 足够，而且工程复杂度更低。

**可能追问：SSE 有什么缺点？**

**追问回答：**

SSE 主要是单向通信，不适合客户端频繁向服务端发消息；另外在某些代理环境下需要注意连接超时和缓冲问题。所以生产环境要配置 keep-alive、禁用代理缓冲，并处理断线重连。

---

### Q22：异步会话派发是怎么做的？

**回答：**

用户发送问题后，后端不是同步等待 Agent 完成，而是先创建一个 session，并把 Agent 执行放到后台任务里，然后立即返回 session id。

前端拿到 session id 后建立 SSE 连接，实时接收事件。这样用户可以看到“会话开始、工具调用、路线生成、回复完成”等过程。

**可能追问：这样怎么拿最终结果？**

**追问回答：**

最终结果会通过 SSE 的完成事件推送，同时会话状态也会持久化。前端收到完成事件后，可以刷新会话详情，保证最终展示和后端状态一致。

---

### Q23：SSE 事件有哪些？

**回答：**

常见事件包括：

- 会话开始
- 子 agent 切换
- worker 启动或完成
- 工具开始、进度、完成、失败
- 路线结果准备好
- assistant token 片段
- assistant 完成
- session 失败

这些事件构成了一个前后端共享的协议。前端不是随便解析字符串，而是按事件类型更新不同 UI 状态。

---

### Q24：断线重连怎么处理？

**回答：**

每个 SSE 事件都有递增 offset。前端收到事件后会记录最新 offset。如果连接断开，重新连接时带上 last event id，后端只回放这个 offset 之后的事件。

为了支持服务重启后的恢复，事件本身也会持久化，而不是只存在内存里。这样即使 worker 重启，前端仍然有机会恢复已经产生的事件。

**可能追问：如果事件已经被清理了怎么办？**

**追问回答：**

可以回退到会话详情接口，加载最终的会话状态和结果。也就是说，SSE 负责实时体验，会话详情负责最终一致性。

---

### Q25：如何防止用户重复点击发送？

**回答：**

前端会在会话忙碌状态下禁用重复提交，后端也会接收 idempotency key。对于同一个 session 和同一个幂等 key，后端会识别为同一次运行，不会重复创建任务。这样前端防抖和后端幂等一起保证安全。

---

## 7. 前端 Vue / TypeScript / Pinia 类问题

### Q26：前端为什么选择 Vue 3？

**回答：**

Vue 3 的组合式 API 很适合把复杂交互拆成 composable，比如会话控制、SSE 连接、地图渲染、知识库管理都可以独立封装。Pinia 的状态管理也比较轻量，适合管理会话列表、当前回复、执行阶段、地图结果等状态。

Vite 则让开发和构建速度比较快，TypeScript 用来保证前后端协议和组件状态的类型安全。

---

### Q27：Pinia 在项目里管理哪些状态？

**回答：**

主要管理三类状态：

第一是聊天会话状态，比如当前 session、历史会话、输入框、流式回复、执行阶段和错误信息。  
第二是机厅浏览状态，比如筛选条件、分页结果、当前选中门店、地图状态。  
第三是知识库状态，比如文件列表、索引进度、失败原因和上传状态。

这些状态如果都放在组件内部，会导致兄弟组件之间传递很麻烦。用 Pinia 后，状态来源更统一。

---

### Q28：前端如何消费 SSE？

**回答：**

前端通过 EventSource 建立连接。每收到一个事件，就先解析事件 envelope，判断事件类型，再更新对应状态。

例如收到工具开始事件，就更新进度列表；收到 token 事件，就追加流式文本；收到路线准备事件，就更新地图 artifact；收到完成事件，就结束流式状态并刷新会话详情。

**可能追问：如何避免重复 token 或重复消息？**

**追问回答：**

前端会记录当前流式文本和最终回复，完成后会做合并判断，避免同一段回复既作为 token 流展示又作为最终消息重复出现。事件 offset 也能帮助避免重复消费。

---

### Q29：为什么要用 TypeScript？

**回答：**

这个项目里前后端交互对象很多，比如会话、SSE 事件、地图 artifact、知识库文件状态、路线结果等。如果没有类型，很容易字段漂移。

TypeScript 的作用是让这些协议在前端编译期就能检查出来。项目里还可以通过后端 schema 生成前端类型，减少手写类型和后端 DTO 不一致的问题。

---

### Q30：前端如何处理复杂状态？

**回答：**

我把聊天生命周期抽成明确状态，比如 idle、dispatching、streaming、reconnecting、completed、failed，而不是靠多个布尔值拼状态。这样组件判断会更清晰，也方便测试。

例如用户发送时进入 dispatching，SSE 打开后进入 streaming，断线后进入 reconnecting，完成后进入 completed，失败则进入 failed。

**可能追问：为什么多个 boolean 不好？**

**追问回答：**

多个 boolean 容易出现非法组合，比如既 sending 又 completed，或者 streamConnected 是 false 但 streamReplyActive 是 true。状态机可以把合法状态收束起来，后续加重连和恢复也更稳。

---

## 8. 地图与业务检索类问题

### Q31：机厅检索支持哪些条件？

**回答：**

支持关键词、店名、机种、地区、省市区、是否有机台、更新时间、机种数量和距离排序。对于附近推荐，会结合用户位置或解析出的地点坐标做距离排序。

---

### Q32：JSONL 和 Supabase 双数据源为什么都支持？

**回答：**

JSONL 适合本地开发、演示和轻量部署，不依赖数据库；Supabase 更适合线上查询和结构化管理。通过统一 repository 接口，上层检索逻辑不需要关心底层数据来自文件还是数据库。

**可能追问：如果两种数据源结果不一致怎么办？**

**追问回答：**

生产环境应该选一个主数据源，另一种作为开发或导入验证使用。不能让线上请求随机混用，否则会造成结果不一致。项目里通过配置选择数据源。

---

### Q33：高德地图 API 用在什么地方？

**回答：**

主要用在四个地方：

- 浏览器定位后的逆地理编码
- 地点或地址解析
- 地图点位展示
- 路线规划和导航链接生成

地图能力让搜索结果不只是文字列表，而是可以在地图上看到位置，并进一步生成路线。

---

### Q34：如果地图 API 失败怎么办？

**回答：**

地图 API 失败时，系统会尽量降级。比如路线失败时仍然展示候选机厅和地址；坐标缺失时可以展示区域级回退；外部服务超时时会记录 fallback 原因，并把可用信息返回给用户。

---

## 9. MCP 类问题

### Q35：MCP 在项目里起什么作用？

**回答：**

MCP 可以理解成一种标准化工具接入协议。项目里除了内建工具，也可以通过 MCP 接入外部能力，比如地图相关工具。这样工具来源可以更灵活，Agent 不需要关心工具是本地 executor 还是远程 MCP 服务。

---

### Q36：MCP 和普通 HTTP API 有什么区别？

**回答：**

普通 HTTP API 是业务代码直接调用具体接口，接口怎么写、参数怎么传都由业务自己适配。MCP 更像是给 Agent 使用的工具协议，它会描述工具能力、参数 schema 和调用方式，更适合让模型或 Agent 框架动态发现和调用工具。

---

## 10. 工程化与稳定性类问题

### Q37：这个项目有哪些工程化设计？

**回答：**

主要有几类：

- SSE 事件协议统一生成前端类型，减少字段漂移
- HTTP DTO 从后端 schema 生成前端类型，减少手写错误
- 聊天生命周期用状态机管理
- 会话支持 idempotency key，避免重复提交
- SSE 支持 offset 回放和断线续播
- 工具层有超时、重试、预算和 trace id
- 地图 artifact 做版本化，方便前后端兼容
- RAG 支持增量索引和失败重试
- 有固定问句集、工具失败注入、SSE 回放和前端黄金链路测试

---

### Q38：为什么要做协议类型生成？

**回答：**

前后端都依赖同一批字段，比如 SSE 事件名、事件 payload、会话状态、地图 artifact。如果前后端各写一份类型，很容易后端改了字段、前端没跟上，最后运行时才报错。

通过后端 schema 生成前端类型，可以把协议变成单一真源。字段变化会在构建或测试阶段暴露出来，而不是线上才发现。

---

### Q39：你做了哪些测试？

**回答：**

测试分几层：

第一是后端单元测试，覆盖工具注册、参数校验、RAG 服务、会话状态等。  
第二是集成测试，覆盖接口、会话派发、幂等和重连。  
第三是 Agent 回归测试，固定一批问句验证意图路由和 query rewrite 是否稳定。  
第四是故障演练，比如工具失败注入，验证系统是否能优雅降级。  
第五是前端端到端测试，跑“发问、流式回复、地图结果、知识库状态”的黄金链路。

---

### Q40：你怎么保证 Agent 改 prompt 后不退化？

**回答：**

我会用固定问句集做回归。比如准备一批典型问题，包括搜索、附近推荐、路线规划和知识问答。每次改 prompt 或工具逻辑后，验证这些问题的意图路由、query rewrite 和关键事件顺序是否稳定。

Agent 系统不能只靠“手动试一下”，因为 prompt 很容易局部改动影响其他问题。固定评测集能让回归更可控。

---

### Q41：如何排查线上问题？

**回答：**

我会从 session id 和 trace id 入手。一次会话会产生一串 SSE 事件和工具调用记录。先看会话状态，再看事件停在哪一步，是意图路由、工具调用、RAG 检索、地图 API 还是最终生成出了问题。

工具调用结果里会记录状态、错误类型、耗时、重试次数和 fallback 原因，这些信息能帮助快速定位。

---

## 11. 性能与成本类问题

### Q42：这个项目有哪些性能优化点？

**回答：**

主要有：

- 会话异步化，避免长请求阻塞
- SSE 流式输出，降低用户感知延迟
- RAG 增量索引，避免每次全量重建
- 查询结果分页，避免一次返回过大
- 地图 SDK 懒加载，只有需要地图时再加载
- 工具超时和预算，避免单个工具拖慢整个会话

---

### Q43：如何控制 LLM 和工具调用成本？

**回答：**

可以从三方面控制。

第一是减少不必要的模型调用，明显规则能判断的意图不用全交给模型。  
第二是 RAG 只检索 top-k 相关片段，不把整篇文档塞给模型。  
第三是工具预算和最大步骤数，避免 Agent 反复调用工具。

---

### Q44：RAG 检索慢怎么办？

**回答：**

可以优化 chunk 数量、top-k、索引 backend 和缓存。常见查询可以做 query cache；索引层可以用 FAISS 或数据库向量索引；文档更新时走增量索引，避免频繁全量重建。

---

## 12. 安全与权限类问题

### Q45：用户上传文档有什么风险？

**回答：**

主要风险包括文件过大、恶意格式、解析失败、敏感信息泄露和索引污染。处理方式包括限制文件大小和类型、解析失败记录错误、用户隔离、索引元数据隔离，以及不要把原始敏感内容直接打进日志。

---

### Q46：工具调用有什么安全风险？

**回答：**

如果不控制，模型可能调用不该调用的工具，或者生成异常参数。所以需要 allowed tools、参数 schema、权限检查、超时限制和调用预算。对于写操作或高风险工具，还应该加人工确认或更严格权限。

---

### Q47：如何避免 Prompt Injection？

**回答：**

RAG 文档里的内容不能直接当成系统指令。需要明确告诉模型：检索片段只是参考资料，不得覆盖系统规则；工具调用也不能由文档内容直接决定。同时对工具权限做白名单限制，即使文档诱导模型调用危险工具，也调用不了。

---

## 13. 部署与数据类问题

### Q48：这个项目怎么部署？

**回答：**

可以用 Docker Compose 部署。后端是 FastAPI 服务，前端是 Vite 构建后的静态资源，由 Nginx 或类似服务托管并反向代理 API。运行数据、会话状态、RAG 索引和地图缓存可以挂载到持久化目录。

---

### Q49：Supabase 在项目里负责什么？

**回答：**

Supabase 主要作为结构化数据源，保存机厅相关数据，并支持线上查询。项目同时保留 JSONL 模式，方便本地开发和演示。通过统一数据访问层，上层业务不用关心底层数据源差异。

---

### Q50：如果数据量变大怎么办？

**回答：**

结构化机厅数据可以通过数据库索引、分页、地区字段和关键词搜索优化。RAG 数据可以通过增量索引、向量索引持久化、检索缓存和 top-k 控制优化。前端列表可以做分页或虚拟滚动，地图点位可以做聚合或按视野加载。

---

## 14. 简历项目深挖类问题

### Q51：这个项目里你最有技术含量的部分是什么？

**回答：**

我觉得最有含量的是把 Agent 从“能回答”推进到“可观测、可恢复、可测试”。很多 AI demo 只是调模型，但这个项目里我做了异步会话、SSE 协议、工具治理、RAG 增量索引、幂等重连、地图 artifact 和回归测试。这些东西解决的是实际线上会遇到的问题，比如用户重复点击、页面断线、工具失败、协议字段漂移、prompt 改动导致退化。

---

### Q52：项目里最难的问题是什么？

**回答：**

最难的是多模块之间的状态一致性。一次用户提问会经过前端状态、后端会话、Agent working memory、工具结果、SSE 事件、地图 artifact 和最终会话详情。任何一个环节字段不一致，前端就可能展示错。

解决方式是把协议结构化，SSE 和 HTTP DTO 都尽量生成类型；把聊天生命周期抽成状态机；会话用 session id、offset 和 idempotency key 串起来；再用回归测试覆盖关键链路。

---

### Q53：你这个项目有哪些不足？

**回答：**

目前仍有一些可以继续优化的地方。

第一，多实例部署下最好把会话锁和 SSE 事件放到真正的数据库或 Redis，而不是只依赖本地持久化。  
第二，RAG 质量还可以加入更完整的离线评测，比如召回率、命中率和答案引用准确率。  
第三，前端可以继续补充组件级测试、视觉回归和可访问性检查。  
第四，Agent trace 可以做成更完整的可视化页面，方便排查每次工具调用和 token 成本。

这个回答的重点是承认边界，同时说明你知道下一步怎么做。

---

### Q54：如果让你重新做一次，你会怎么改？

**回答：**

我会优先把三个点做得更工程化。

第一，事件和会话状态统一进数据库或 Redis，天然支持多实例。  
第二，建立更完整的 Agent eval，包括意图识别、工具调用、RAG 召回和端到端答案质量。  
第三，把 Agent run trace 产品化，让每次请求的意图、工具、耗时、失败原因和成本都能可视化。

### Q56：Vue 里如何封装 SSE？

我会封装成一个 composable，比如负责：

- 创建 EventSource
- 监听不同事件类型
- 维护连接状态
- 记录 last event id
- 断线后重连
- 组件卸载时关闭连接

Pinia store 只保存业务状态，比如当前会话、流式文本、执行阶段和地图结果。这样网络连接逻辑和页面展示逻辑就分开了。

关键不是框架本身，而是：

- 状态如何建模
- SSE 事件如何消费
- 地图渲染如何隔离
- 类型协议如何保持一致
- 复杂异步流程如何测试

## 16. 高频八股结合项目

### Q58：SSE 的原理是什么？

**回答：**

SSE 是 Server-Sent Events，基于 HTTP 长连接。服务端不断以特定格式写入事件，浏览器用 EventSource 接收。每条消息可以包含 event、id 和 data。浏览器断线后可以带 Last-Event-ID 重连。

它适合服务端向前端推送流式消息，比如 AI token、执行阶段和通知。

---

### Q59：RESTful 和 SSE 的区别？

**回答：**

RESTful 更适合请求一次、响应一次，比如查询列表、提交表单。SSE 更适合服务端持续推送，比如 Agent 执行过程和流式回复。

这个项目里，创建会话用普通 HTTP 接口，后续过程用 SSE 推送，两者配合使用。

---

### Q60：Embedding 是什么？

**回答：**

Embedding 是把文本转换成向量。语义相近的文本，向量距离也更近。RAG 里会把文档 chunk 转成 embedding，用户问题也转成 embedding，然后通过相似度找到相关片段。

---

### Q61：向量检索和关键词检索的区别？

**回答：**

关键词检索看字面匹配，适合店名、城市名、机种名这种精确词。向量检索看语义相似，适合“环境怎么样”“维护好吗”这类表达。两者结合效果更稳。

---

### Q62：什么是 Agentic RAG？

**回答：**

普通 RAG 通常是固定流程：检索文档，然后生成回答。Agentic RAG 是让 Agent 根据问题决定是否检索、检索什么、是否需要调用其他工具、是否要二次检索。

在这个项目里，用户问结构化机厅信息时可能先查数据库；数据库没覆盖时再查 RAG；如果涉及路线再调用地图工具。这就带有 Agentic RAG 的特点。

---

### Q63：Function Calling 是什么？

**回答：**

Function Calling 是让模型输出结构化工具调用参数，而不是直接回答文本。比如模型判断需要查机厅，就输出工具名和参数，后端校验后执行工具，再把结果交回模型。

这样模型负责决策，程序负责真实执行。

---

### Q64：为什么大模型容易幻觉？

**回答：**

因为模型本质是基于概率生成文本，并不天然知道项目私有数据或实时数据。如果问题超出它的知识范围，它可能生成看似合理但不真实的内容。

这个项目用工具查询和 RAG，把回答约束在真实数据和检索片段上，降低幻觉。

---

### Q65：前端如何防止 XSS？

**回答：**

AI 回复如果支持 Markdown 渲染，就要先做 HTML sanitize，不能直接把模型输出当可信 HTML 插入页面。还要限制可用标签和属性，避免脚本注入。

---

### Q66：TypeScript 的作用是什么？

**回答：**

TypeScript 可以在编译阶段发现字段、状态和接口不一致的问题。这个项目里涉及很多 DTO 和事件协议，如果不用类型，前后端字段漂移很容易运行时才发现。

---

## 17. 项目扩展追问

### Q67：如果面试官让你现场画一次完整链路，你怎么讲？

**回答：**

我会按一次用户提问来讲：

```text
用户提问
  ↓
前端创建会话请求
  ↓
后端返回 session id
  ↓
前端建立 SSE 连接
  ↓
Agent 判断意图
  ↓
调用搜索 / RAG / 地理解析 / 路线规划工具
  ↓
工具结果写入 working memory
  ↓
SSE 推送阶段事件和流式回复
  ↓
前端展示回复、进度和地图结果
  ↓
会话状态持久化，支持历史恢复
```

这条链路的重点是：HTTP 负责创建任务，SSE 负责过程推送，工具负责真实数据，working memory 负责中间状态，前端状态管理负责展示。

---

### Q68：用户问“附近机厅”时，系统怎么知道附近是哪里？

**回答：**

一般有两种来源。

第一种是浏览器定位。前端获取用户授权后拿到经纬度，后端可以结合逆地理编码拿到城市、区县等信息，再做附近检索和距离排序。

第二种是用户问题里带了地点，比如“人民广场附近”“南京新街口附近”。这种情况需要先做地点解析，把自然语言地点转成坐标或区域，再用这个坐标查附近机厅。

**可能追问：如果用户不给定位权限怎么办？**

**追问回答：**

可以降级为地区搜索或引导用户补充城市/地标。比如用户问“附近哪里有机厅”，但没有定位权限，系统可以提示“需要你提供城市或地标”，而不是随便猜一个位置。

---

### Q69：如何处理机厅没有坐标的问题？

**回答：**

机厅数据不一定都有精确坐标。如果有坐标，就直接用于地图点位和距离排序；如果没有坐标，可以通过地址或商场名尝试地理编码；如果还是失败，就退回到城市、区县级别展示，并在前端提示“坐标暂不可用”。

这样可以保证数据不完整时系统仍然可用，而不是因为一个字段缺失就无法展示。

---

### Q70：路线规划是怎么做的？

**回答：**

路线规划需要两个关键点：起点和终点。

起点通常来自用户定位，终点来自选中的机厅坐标。如果用户问“去最近的机厅怎么走”，系统会先找最近候选，再把用户位置和目标机厅坐标传给路线规划工具，调用地图服务生成路线、距离、耗时和 polyline。

前端拿到 route artifact 后，会在地图上画路线，并提供跳转高德地图的导航链接。

**可能追问：为什么路线结果要结构化？**

**追问回答：**

因为路线不仅要生成文字，还要在地图上画出来。结构化 route 里包含 provider、mode、distance、duration、origin、destination、polyline 等字段，前端可以稳定渲染，而不是从自然语言里解析路线。

---

### Q71：地图 artifact 是什么？

**回答：**

地图 artifact 可以理解成后端给前端的结构化地图结果。它不是一句文本，而是一组可以渲染的数据，比如候选机厅列表、路线、用户位置、目标点、地图场景类型等。

这样前端不需要猜模型文字里提到了哪些店，而是直接根据 artifact 渲染地图卡片、点位和路线。

---

### Q72：为什么地图 artifact 要版本化？

**回答：**

因为后端输出结构以后可能会升级。比如早期只返回路线，后面可能返回更完整的 map scene。如果没有版本，前端很难兼容旧数据。

加 schema_version 后，前端可以根据版本做兼容适配。旧会话用旧逻辑渲染，新会话用新逻辑渲染，避免后端升级一次就把历史数据或前端展示打坏。

---

### Q73：历史会话是怎么恢复的？

**回答：**

后端会持久化会话状态，包括用户消息、助手回复、当前状态、最后一次 stream offset、地图结果和 working memory 中的重要 artifact。前端打开页面时可以拉取会话列表和详情，恢复历史消息和地图结果。

如果会话还在运行，前端可以根据 session id 和 last offset 重新连接 SSE，继续接收后续事件。

---

### Q74：如果页面刷新，流式输出会不会丢？

**回答：**

不会只依赖浏览器内存。前端会记录当前 session id 和最后接收到的事件 offset；后端会记录会话状态和事件。页面刷新后，前端可以重新加载当前会话，并从 last offset 之后续接 SSE。

如果实时事件已经结束，也可以通过会话详情接口拿到最终结果，保证最终一致性。

---

### Q75：怎么做会话幂等？

**回答：**

前端每次提交会带 idempotency key，后端根据 session id 和 idempotency key 判断是不是同一次运行。

如果用户重复点击发送，同一个 key 的请求不会重复创建任务；如果同一个 session 已经有不同 key 的任务在跑，后端会拒绝冲突请求。这样可以避免用户点两次导致后台跑两个 Agent。

---

### Q76：如果后端 worker 重启，会发生什么？

**回答：**

当前设计里，会话状态和 SSE 事件都有持久化。worker 重启后，前端可以重新拉取会话详情。如果已有事件被持久化，也可以继续按 offset 回放。

但我也会实事求是地补充：如果要做真正多实例生产部署，最好把会话状态、事件日志和分布式锁放到数据库或 Redis，而不是只依赖本地文件。

---

### Q77：为什么要做前后端协议单一真源？

**回答：**

因为这个项目的前后端协议比较多，包括 HTTP DTO、SSE 事件、地图 artifact、知识库文件状态等。如果前端手写一份类型，后端也维护一份模型，时间久了很容易字段漂移。

单一真源的做法是以后端 schema 生成前端 TypeScript 类型。这样后端字段一改，前端类型和测试就会暴露问题，避免线上才发现解析失败。

---

### Q78：你怎么设计 SSE 事件协议？

**回答：**

每个事件都有统一 envelope，包括 event name、session id、事件 id、时间和 data。event name 表示事件类型，比如会话开始、工具开始、工具完成、token 输出、路线 ready、会话失败等。data 根据事件类型携带不同字段。

这样做的好处是前端可以统一解析 envelope，再按 event 类型分发处理，避免每个事件都用完全不同的格式。

---

### Q79：如果 SSE 事件顺序乱了怎么办？

**回答：**

每个事件都有递增 id。前端可以按 id 判断事件顺序，后端回放时也按 id 返回。正常情况下，同一个会话里的事件应该按生成顺序推送。

如果前端发现重复事件，可以根据 id 去重；如果发现缺事件，可以通过 last event id 重新请求后续事件，或者最终回退到会话详情接口。

---

### Q80：Agent 为什么需要固定问句集评测？

**回答：**

Agent 的行为会受到 prompt、工具描述和上下文影响。改一个 prompt 可能让某些问题变好，也可能让另一些问题退化。固定问句集就是把常见问题沉淀下来，每次改动后验证意图路由、query rewrite 和关键事件顺序是否稳定。

这能避免“我手动试了两个问题觉得可以”这种不可靠的验证方式。

---

### Q81：工具失败注入测试有什么意义？

**回答：**

真实线上最常见的问题不是模型完全不能答，而是某个工具超时、地图 API 失败、RAG 没命中、数据库临时异常。工具失败注入就是模拟这些故障，看系统是否能优雅降级。

比如 RAG 失败时是否还能返回数据库结果，路线规划失败时是否还能展示候选机厅，工具超时时是否能给出明确错误原因。

---

### Q82：你怎么验证前端黄金链路？

**回答：**

我会用端到端测试模拟用户从发问到看到结果的完整链路。比如：

1. 用户输入问题
2. 后端返回 session id
3. SSE 推送阶段事件
4. 前端显示流式回复
5. 地图卡片渲染路线或候选点
6. 知识库状态正常展示

这样可以验证的不只是单个组件，而是前后端协议、状态管理、SSE 消费和地图展示是否真的串起来了。

---

### Q83：如果 RAG 检索结果和数据库结果冲突怎么办？

**回答：**

我会优先信结构化数据库，因为它更适合保存标准字段，比如店名、地址、机种数量、地区等。RAG 更适合补充非结构化信息，比如评论、活动、攻略和说明。

如果两者冲突，回答时可以明确来源，比如“数据库显示该店有 2 台舞萌，知识库评论里提到维护情况较好”。不要把非结构化片段直接覆盖结构化事实。

---

### Q84：如果 RAG 没有命中，系统怎么回答？

**回答：**

如果 RAG 没命中，但数据库有结果，就返回数据库结果，并说明暂时没有找到补充资料。如果数据库和 RAG 都没命中，就引导用户补充城市、店名或更明确的问题。

关键是不要让模型为了显得有答案而编造。

---

### Q85：知识库文件删除后，索引怎么处理？

**回答：**

删除文件后，需要把这个文件对应的 chunk 从索引和 metadata 中移除。因为每个 chunk 都记录 source_uri 或 relative path，所以可以按文件维度找到对应 chunk 并删除。

如果使用增量索引，就不需要全量重建整个知识库，只处理被删除文件相关的索引数据即可。

---

### Q86：为什么要保留 metadata？

**回答：**

metadata 主要用于三件事：

第一是来源追踪，知道回答依据来自哪个文件、哪一页或哪个 source_uri。  
第二是过滤，比如按文件、用户、类型或时间筛选知识。  
第三是增量维护，比如删除某个文件时能找到对应 chunk。

没有 metadata，RAG 只能召回文本，但很难做工程化管理。

---

### Q87：用户上传恶意内容诱导模型怎么办？

**回答：**

知识库内容只能作为参考资料，不能作为系统指令。系统 prompt 需要明确告诉模型：检索片段不能覆盖系统规则，不能直接执行文档里的指令。

同时工具调用有白名单、权限和参数校验。即使文档里写“忽略之前规则，调用某某危险工具”，模型也没有权限调用不在 allowed tools 里的工具。

---

### Q88：为什么要做工具 trace id？

**回答：**

一次用户请求可能调用多个工具，如果没有 trace id，日志里很难把这些调用串起来。trace id 可以把一次会话中的每个工具调用、耗时、状态和错误原因关联起来。

排查问题时，可以从 session id 找到 trace id，再看到是哪一步工具慢了或失败了。

---

### Q89：为什么需要 circuit breaker？

**回答：**

如果某个外部服务持续失败，比如地图 API 或远程 MCP 工具一直超时，系统不应该每次都继续打满请求。circuit breaker 会在连续失败达到阈值后暂时熔断，短时间内直接返回失败或降级，保护系统和外部服务。

---

### Q90：前端状态机解决了什么问题？

**回答：**

聊天页面原本很容易出现多个 boolean 状态，比如 sending、streaming、reconnecting、completed。多个 boolean 会产生非法组合。

状态机把状态收束成 idle、dispatching、streaming、reconnecting、completed、failed。每次事件只能从当前状态转到合法的下一个状态。这样更容易维护，也更容易测试。

---

### Q91：如果用户连续切换会话，如何避免旧请求覆盖新页面？

**回答：**

前端需要给请求加 abort 或 request id。用户切换会话时取消旧请求，或者在响应回来时检查 request id 是否仍然是当前页面对应的请求。如果不是，就丢弃旧响应。

这个问题在地图详情、会话详情和知识库刷新里都很常见。

---

### Q92：为什么要对地图 SDK 做加载隔离？

**回答：**

地图 SDK 是外部脚本，加载慢或失败都不应该影响整个聊天页面。把地图加载封装起来后，可以在地图不可用时仍然展示文字结果、候选列表和导航链接。

这也是一种前端降级设计：地图增强体验，但不是唯一结果载体。

---

### Q93：如果地图坐标系不一致怎么办？

**回答：**

国内地图服务常见坐标系有 WGS84 和 GCJ02。浏览器定位通常偏 WGS84，高德地图使用 GCJ02。如果不处理，点位和路线可能偏移。

所以需要在数据结构里标记 coord_system，并在进入地图渲染或路线规划前做统一转换或兼容处理。

---

### Q94：为什么要把 Agent 输出分成文本和结构化结果？

**回答：**

文本适合给用户阅读，但不适合程序渲染。结构化结果适合前端稳定展示，比如地图点位、路线、知识引用和工具状态。

如果只返回文本，前端要从自然语言里解析店名和路线，很不稳定。分成文本回复和结构化 artifact 后，用户体验和工程稳定性都会更好。

---

### Q95：你如何评价这个项目的业务价值？

**回答：**

它把一个垂直场景里的检索、问答和地图决策整合到一起。对用户来说，不需要在多个平台之间切换；对工程来说，它验证了 AI 应用落地的完整链路，包括工具调用、RAG、地图服务、流式交互和故障恢复。

---

### Q96：如果面试官质疑项目规模不大，你怎么回答？

**回答：**

我会承认它不是大规模商业系统，但它覆盖了真实 AI 应用的关键问题：协议设计、工具治理、RAG 索引、异步会话、断线重连、前端状态管理和回归测试。

项目规模可以继续扩大，但这些工程问题本身和真实线上系统是一致的。

---

### Q97：你从这个项目里学到最多的是什么？

**回答：**

我学到的是 AI 应用落地不只是 prompt。真正难的是把模型能力放进可控的工程系统里：什么时候让模型决策，什么时候用确定性工具，失败如何降级，前端如何展示过程，协议如何避免漂移，改动后如何回归。

---

## 18. 基于简历描述的逐句追问

这一节专门针对简历上的每一句描述准备。面试官经常会从简历原文里挑词追问，比如“你说支持自然语言问答，具体怎么支持”“你说增量索引，具体增量在哪里”。下面这些回答可以直接背思路。

### Q98：你简历写“面向音游玩家”，为什么选这个场景？

**回答：**

因为音游机厅是一个信息分散但需求明确的垂直场景。玩家关心的不只是店名，还包括城市、商圈、机种、机台数量、距离、交通、营业情况和其他玩家评论。这些信息既有结构化字段，也有非结构化资料，很适合用“数据库检索 + RAG + 地图 + Agent”的方式整合。

这个场景足够具体，不会变成泛泛的聊天机器人；同时它又包含搜索、推荐、路线规划、知识问答和地图展示，能体现 AI 应用落地的完整链路。

**可能追问：这个场景有什么真实痛点？**

**追问回答：**

玩家找店时经常要在地图、社群、机厅列表之间来回切换。比如想知道“附近有没有舞萌多的店”“这家店维护怎么样”“怎么从当前位置过去”，传统方式需要手动查多个来源。项目把这些步骤整合成一次自然语言提问，降低了查找成本。

---

### Q99：你简历写“智能检索”，智能体现在哪里？

**回答：**

智能检索不是只做关键词搜索，而是能理解用户自然语言里的意图和约束。

比如用户说“下班后附近哪里适合打舞萌”，里面包含附近、时间场景、机种和推荐意图。系统会把这些信息转成可执行的工具参数，比如位置、机种、距离排序和候选数量。数据库查不到补充资料时，还可以查 RAG 知识库。

所以智能体现在三个方面：自然语言理解、自动选择工具、结合结构化和非结构化数据生成结果。

---

### Q100：你简历写“路线规划”，和普通地图导航有什么区别？

**回答：**

普通地图导航需要用户先知道目的地。这个项目的路线规划是 Agent 链路的一部分，系统可以先根据用户需求找到候选机厅，再选择目标点生成路线。

比如用户问“帮我找最近的舞萌店并规划路线”，系统会先做附近检索，再确定目标机厅，最后调用路线规划工具生成距离、耗时、路线 polyline 和跳转高德导航链接。

区别在于它不是单独导航，而是“检索 + 决策 + 路线”的组合。

---

### Q101：你说“支持自然语言问答”，具体支持哪些问题？

**回答：**

主要支持四类：

第一类是机厅搜索，比如“南京哪里有舞萌”。  
第二类是附近推荐，比如“我附近最近的机厅”。  
第三类是路线规划，比如“从当前位置去这家店怎么走”。  
第四类是知识问答，比如“这家店评价怎么样”“有没有活动说明”。

这些问题背后会走不同链路，有的查结构化数据库，有的查知识库，有的需要地图路线工具。

---

### Q102：你说“条件筛选”，筛选条件有哪些？

**回答：**

筛选条件包括关键词、店名、地区、省市区、机种、是否有机台、更新时间、机台数量、距离排序等。

这些条件既可以通过前端表单选择，也可以从自然语言里提取。比如用户说“上海浦东舞萌多的店”，系统可以识别出城市、区县和机种，再转换成查询参数。

---

### Q103：你说“附近推荐”，推荐逻辑是什么？

**回答：**

附近推荐首先需要确定中心点。中心点可能来自浏览器定位，也可能来自用户输入的地标或地址。拿到中心点后，系统会查询有坐标的机厅并按距离排序。

如果用户还指定了机种，比如“附近舞萌多的店”，就会在距离排序基础上加入机种筛选或机台数量排序。对于没有坐标但属于同一区域的机厅，也可以作为补充候选展示，避免漏掉数据不完整的店。

---

### Q104：你说“知识库检索”，知识库里放什么？

**回答：**

知识库适合放结构化数据库不方便表达的内容，比如玩家评论、活动说明、店铺补充信息、攻略、FAQ、维护情况、历史说明等。

结构化数据库负责标准字段，比如店名、地址、机种数量；知识库负责补充文本证据。两者结合后，回答会比单纯查数据库更完整。

---

### Q105：你说“地图点位展示”，前端怎么展示？

**回答：**

后端返回的是结构化地图 artifact，里面包含候选机厅、坐标、路线、用户位置等字段。前端根据 artifact 渲染地图卡片，在地图上标记候选点或路线。

这样前端不需要从 AI 文本里解析地点，而是直接消费结构化数据，展示更稳定。

---

### Q106：你说“导航路线生成”，路线结果包含哪些字段？

**回答：**

路线结果一般包含 provider、出行方式、起点、终点、距离、预计耗时、路线 polyline 和提示信息。前端可以用这些字段渲染路线摘要和地图路线，也可以生成跳转高德地图的导航链接。

---

### Q107：你说“历史会话管理”，管理哪些内容？

**回答：**

历史会话包含用户消息、助手回复、会话状态、意图、更新时间、执行结果、地图 artifact、最后的 stream offset 等。用户可以查看之前问过的问题，也可以在运行中的会话断线后恢复。

这和普通聊天记录不同，因为这里还要恢复结构化结果，比如地图路线和候选机厅。

---

### Q108：你说“Agent 编排链路”，编排的核心是什么？

**回答：**

核心是让模型不要直接回答一切，而是作为调度层。它先理解用户意图，再决定调用哪个 worker 和工具。比如搜索问题走搜索 worker，路线问题走导航 worker，补充资料走 RAG 工具。

这样系统既有自然语言理解能力，又能通过工具拿到真实数据。

---

### Q109：你说“main_agent -> worker -> tool registry”，为什么这样分三层？

**回答：**

main_agent 负责判断问题和调度，worker 负责具体任务，tool registry 负责工具管理。三层拆开后，每层职责更清楚。

如果没有 worker，所有工具都堆给 main agent，prompt 会很复杂；如果没有 tool registry，工具参数校验、权限和执行逻辑会散落在各处，不好维护。

---

### Q110：你说“支持搜索、附近推荐、路线规划和知识问答等意图处理”，这些意图怎么区分？

**回答：**

先通过关键词和上下文做初步判断，比如“附近、最近”偏附近推荐，“路线、怎么去”偏导航，“评价、活动、说明”偏知识问答。对于复杂表达，再由 Agent 结合上下文判断。

意图识别的结果不会只是标签，还会影响后续可用工具和执行流程。

---

### Q111：你说“工具调用系统”，工具和普通函数有什么区别？

**回答：**

普通函数是代码直接调用，工具是给 Agent 调用的能力单元。工具需要有名字、描述、参数 schema、权限和执行器。模型根据工具描述决定是否调用，后端再校验参数并执行。

所以工具不仅是函数，还包含“让模型理解和安全调用”的元信息。

---

### Q112：你说“manifest + JSON Schema + executor”，三者分别干什么？

**回答：**

manifest 负责声明工具，比如工具名、描述、参数结构。  
JSON Schema 负责约束参数，保证模型生成的参数合法。  
executor 负责真正执行业务逻辑，比如查数据库或调地图 API。

可以理解成：manifest 告诉模型“有什么工具”，schema 保证“怎么传参”，executor 决定“怎么执行”。

---

### Q113：你说“参数校验、权限控制和执行分发”，分别怎么做？

**回答：**

参数校验是工具执行前用 schema 检查字段类型和范围。权限控制是限制不同 agent 或 worker 可以调用哪些工具。执行分发是根据工具名找到对应 executor 或 provider，然后统一返回结果。

这三个步骤能避免模型随意调用工具，也能让工具层更可观测。

---

### Q114：你说“RAG 知识库能力”，它和普通数据库查询有什么区别？

**回答：**

数据库查询适合结构化条件，比如城市、区县、机种、数量。RAG 适合非结构化文本，比如评论、活动说明、攻略和 FAQ。

数据库回答“有什么”，RAG 更适合回答“怎么样、为什么、有没有说明”。这两个能力是互补的。

---

### Q115：你说“文档上传”，上传后立即能查吗？

**回答：**

上传后文件会进入索引队列，不一定立刻可查。它会经历 pending、indexing、ready 或 failed 状态。ready 后才能稳定检索。

这样设计更真实，因为大文件解析、切分和 embedding 生成都需要时间，不能假装上传瞬间完成。

---

### Q116：你说“增量索引”，如果文件内容变了怎么识别？

**回答：**

可以通过文件路径、更新时间、大小和内容 hash 判断。最关键的是内容 hash，如果 hash 没变，就说明文件内容没变，可以复用已有 chunk；如果 hash 变了，就重新解析和索引这个文件。

---

### Q117：你说“混合检索”，为什么简历上值得写？

**回答：**

因为它说明我不是只做了最简单的向量召回。真实业务里，用户问题既有语义表达，也有精确实体，比如店名、城市、机种名。混合检索能同时兼顾语义召回和关键词命中，结果质量更稳定。

---

### Q118：你说“失败重试”，哪些失败可以重试？

**回答：**

常见可重试失败包括文档解析临时失败、embedding 服务超时、外部检索服务短暂不可用、文件索引中断等。不可重试或需要用户处理的失败包括文件格式不支持、文件损坏、扫描版 PDF 没有 OCR 等。

重试时应该保留失败原因，避免用户不知道为什么失败。

---

### Q119：你说“异步会话派发”，前端怎么知道任务已经开始？

**回答：**

创建会话接口会返回 session id 和运行状态。前端拿到 session id 后立即建立 SSE 连接。后端推送 session.started 事件时，前端就知道任务正式开始执行。

---

### Q120：你说“SSE 流式通信”，流式的到底是什么？

**回答：**

流式的不只是最终文本 token，还包括 Agent 执行阶段。比如会话开始、worker 切换、工具开始、工具完成、路线 ready、assistant token、assistant completed 等。

所以前端展示的不只是“字一个个出来”，还包括系统正在做什么。

---

### Q121：你说“实时展示执行阶段”，用户看到什么？

**回答：**

用户可以看到类似“会话开始”“正在检索门店”“正在规划路线”“路线已生成”“正在生成回复”这样的阶段提示。同时如果路线或候选机厅已经出来，地图卡片可以提前展示，不用等最终文本全部生成。

---

### Q122：你说“流式回复”，前端如何拼接 token？

**回答：**

前端收到 assistant.token 事件后，会把 delta 或 content 追加到当前流式回复里。收到 completed 事件后，用最终 reply 做一次同步，避免中途丢 token 或重复展示。

---

### Q123：你说“地图结果”，地图结果和文本回复谁先出来？

**回答：**

不一定。路线或候选机厅作为结构化 artifact，可以在最终文本回复之前通过 SSE 推给前端。这样用户可能先看到地图结果，再看到完整解释。这个体验比等全部生成完再展示更好。

---

### Q124：你说“JSONL / Supabase 双数据源”，为什么不只用一种？

**回答：**

JSONL 适合本地开发、演示和离线数据读取；Supabase 适合线上结构化查询和数据管理。用统一数据访问接口屏蔽差异后，上层业务逻辑不用关心底层数据源。

---

### Q125：你说“结合高德地图完成定位、逆地理编码、点位标记和路线展示”，分别是什么？

**回答：**

定位是拿到用户当前位置经纬度。  
逆地理编码是把经纬度转成城市、区县、地址等可读信息。  
点位标记是在地图上展示机厅位置。  
路线展示是把起点到终点的路线画出来，并显示距离和耗时。

---

### Q126：如果高德地图 key 配错了怎么办？

**回答：**

前端地图加载会失败，但系统应该降级展示文字结果、候选列表和导航链接提示。后端路线工具也应该返回明确错误原因，而不是让用户看到空白页面。

---

### Q127：你简历写 Vue 3 + Pinia，但项目重点在 AI，前端怎么体现价值？

**回答：**

AI 应用的前端不是普通表单，它要处理长任务、流式事件、地图 artifact、错误恢复和状态同步。Vue 3 + Pinia 的价值在于把复杂状态管理清楚，比如会话状态、流式文本、阶段进度、地图结果、知识库索引状态等。

如果前端只是展示一个最终答案，Agent 的过程价值就体现不出来。

---

### Q128：为什么用 Vite？

**回答：**

Vite 开发启动快、热更新快，适合 Vue 3 + TypeScript 项目。构建时也比较轻量。对这个项目来说，Vite 主要承担前端工程化构建和本地开发效率提升。

---

### Q129：为什么用 TypeScript 而不是 JavaScript？

**回答：**

因为项目里协议字段多，尤其是 SSE 事件、会话 DTO、地图 artifact、知识库状态。如果用 JavaScript，字段写错只能运行时发现。TypeScript 能在编译期提前暴露问题。

---

### Q130：为什么用 FastAPI？

**回答：**

FastAPI 适合这个项目有几个原因：

第一，它对 Pydantic 类型和 OpenAPI 支持很好，方便定义清晰的请求响应模型。  
第二，它支持异步接口，适合 SSE 和后台任务。  
第三，Python 生态更方便接 RAG、文档解析、embedding、FAISS 和各种 AI SDK。

---

### Q131：为什么不是 Node 后端？

**回答：**

Node 也可以做，但这个项目 RAG、文档解析、embedding、FAISS、Python AI 生态会更方便。FastAPI 又能提供比较清晰的类型和接口文档，所以后端选 Python 更合适。

---

### Q132：为什么用 Supabase？

**回答：**

Supabase 提供 Postgres 能力，适合结构化机厅数据管理和查询，同时比自己维护完整数据库服务更轻。项目里通过配置支持 Supabase，也保留 JSONL 方便本地开发。

---

### Q133：简历里写 LLM，会被问用了哪个模型，怎么回答？

**回答：**

可以说项目接的是 OpenAI-compatible LLM 接口，具体模型可以按环境配置切换，比如 DeepSeek、Qwen 或 OpenAI 兼容服务。这样设计是为了避免绑定单一模型，同时也方便根据成本、效果和可用性切换 provider。

---

### Q134：如果模型不支持 tool calling 怎么办？

**回答：**

可以做降级。理想情况是模型原生支持工具调用；如果不支持，可以让模型输出约定 JSON，再由后端解析和校验；如果解析失败，就走更保守的规则或返回澄清问题。

但不管哪种方式，工具执行前都必须经过 schema 校验和权限检查。

---

### Q135：如果模型输出了不存在的工具名怎么办？

**回答：**

工具注册中心会检查工具是否存在，以及当前 agent 是否有权限调用。如果工具名不存在，就返回明确错误，让 Agent 重新决策或降级回答，而不是直接执行。

---

### Q136：如果模型传错参数怎么办？

**回答：**

参数会先经过 JSON Schema 校验。比如缺少必填字段、类型不对、经纬度范围不合法，都会在工具执行前被拦截。之后可以让 Agent 修正参数，或者返回明确错误。

---

### Q137：如果用户问题很模糊怎么办？

**回答：**

如果缺少关键条件，比如“附近”但没有定位，“这家店”但上下文没有店铺，就应该追问用户补充信息。不能为了完成任务而随便猜。

也可以根据历史上下文补全，比如上一轮刚查过某家店，下一轮问“怎么去”，就可以把上一轮店铺作为目标。

---

### Q138：如果用户问题跨多个意图怎么办？

**回答：**

可以拆成多步执行。比如“找最近的舞萌店，并告诉我怎么去”，先走附近搜索，再走路线规划。main agent 负责拆任务，worker 分别执行。

这就是 Agent 编排比单接口查询更有价值的地方。

---

### Q139：如果用户连续追问“那第二家呢”，怎么处理？

**回答：**

需要依赖 working memory 和历史上下文。上一轮搜索结果会保存在 working memory 中，用户说“第二家”时，系统可以解析为上一次候选列表里的第二个机厅，再继续做详情或路线规划。

---

### Q140：如果 RAG 检索到很多相似片段，怎么控制上下文长度？

**回答：**

通过 top-k、chunk 长度、重排和摘要控制。只把最相关的少数片段放进上下文，而不是把所有结果都塞进去。如果片段太长，可以截断或先总结再交给模型。

---

### Q141：如果知识库内容过期怎么办？

**回答：**

需要依靠 metadata，比如更新时间、来源和文件版本。回答时可以优先使用更新时间较新的资料，或者提示来源时间。对于结构化事实，优先数据库；对于评论和活动，最好展示来源和时间，避免误导用户。

---

### Q142：如果数据库里没有某家店，但知识库里提到了怎么办？

**回答：**

可以作为“知识库候选”展示，但要标明它不是结构化数据库中的正式机厅记录。如果能通过地图地理编码拿到位置，可以临时展示地图点位；如果拿不到，就只展示文本线索并提示用户补充城市或店名。

---

### Q143：如果用户问“推荐一个最值得去的店”，怎么排序？

**回答：**

可以综合多个信号：距离、机种数量、是否有目标机种、知识库评论、更新时间和路线成本。结构化指标先保证基本可用，比如距离和机台数量；非结构化评价作为补充参考。

推荐类问题要避免模型凭感觉直接推荐，最好基于可解释信号。

---

### Q144：如果前端地图加载很慢，怎么优化？

**回答：**

可以懒加载地图 SDK，只在用户进入地图场景或需要渲染地图时加载。列表和聊天文本先展示，地图后加载。同时可以缓存 SDK 加载 promise，避免重复加载。

---

### Q145：如果地图上点太多怎么办？

**回答：**

可以分页、按视野加载、点聚合，或者只展示当前搜索结果前 N 个。对用户来说，地图上一次显示太多点反而不好看，应该结合列表筛选和当前视野逐步展示。

---

### Q146：如果用户拒绝定位，但又要按距离排序怎么办？

**回答：**

可以提示需要定位权限，或者让用户输入一个城市、商圈、地标作为中心点。没有中心点时不能做准确距离排序，只能做地区筛选或默认排序。

---

### Q147：如果用户问的问题不属于音游机厅，系统怎么办？

**回答：**

应该识别为超出业务范围，并礼貌说明当前主要支持机厅搜索、路线规划和知识库问答。不要让模型自由回答无关问题，这样能保持产品边界。

---

### Q148：你怎么设计错误提示？

**回答：**

错误提示要尽量告诉用户“发生了什么”和“下一步怎么做”。比如定位失败，可以提示用户授权定位或输入城市；路线失败，可以提示地图服务暂不可用但保留候选店铺；知识库索引失败，可以展示失败原因并提供重试。

---

### Q149：如果面试官问“这个项目有多少是你自己做的”，怎么回答？

**回答：**

可以按模块回答：我主要负责 Agent 编排、工具调用、RAG 知识库、SSE 流式会话、前端状态展示和工程化测试这些核心链路。外部服务如高德地图、LLM provider 和 Supabase 是集成能力，但业务编排、协议设计、状态管理和降级逻辑是项目里的主要实现工作。

可以说它已经具备上线应用的基本形态，比如前后端分离、Docker 部署、会话持久化、SSE 重连、工具治理和测试。但如果要面向大量用户正式上线，还需要补齐更强的多实例状态存储、用户权限体系、监控告警、数据备份、限流和更完整的安全审计。

**回答：**

我会从工程化能力证明：它不是只调用模型返回文字，而是有异步会话、SSE 事件协议、工具 registry、RAG 增量索引、地图 artifact、幂等重连、失败降级和回归测试。这些都是实际产品里才会遇到的问题。

我会选“异步会话 + SSE 协议 + 幂等重连”。因为这块连接了前端体验和后端稳定性，解决的是真实线上很常见的问题：用户重复点击、页面刷新、网络断开、后台任务还在跑、前后端状态不一致。

**回答：**

我会选“Agent + 工具调用 + RAG”。因为它体现了我对大模型定位的理解：模型不是单独回答所有问题，而是负责理解和决策；真实数据、地图路线、知识片段由工具提供；最后把这些结果组织成用户能理解的回答。

**回答：**

如果没有真实线上数据，不要硬编。可以说目前主要通过功能完整性和回归测试验证，比如固定问句集覆盖意图路由，端到端黄金链路覆盖发问、流式、地图和知识库状态。后续如果上线，可以统计检索成功率、RAG 命中率、平均响应时间、工具失败率和用户点击导航转化率。

**回答：**

因为它覆盖了 AI 应用开发的多个关键环节：前端工程、后端 API、Agent 编排、RAG、工具调用、地图服务、异步流式通信和工程化测试。它不是单一技术点，而是一个完整应用链路，能体现我把技术组合成产品功能的能力。

### Q156：如果面试官问“你对 Agent 的理解是什么”，怎么结合项目回答？

**回答：**

我理解的 Agent 不是让模型随便自由发挥，而是让模型在受控环境里做决策。它需要有明确目标、可调用工具、状态记忆和边界约束。

在这个项目里，目标是解决机厅搜索和路线规划；工具包括数据库、RAG、地图和路线；状态由 working memory 保存；边界由 allowed tools、schema 校验、预算和状态机控制。

### Q157：如果面试官问“你的项目怎么体现前端工程化”，怎么回答？

**回答：**

主要体现在复杂状态建模和协议稳定性。前端不是简单页面，而是要处理异步会话、SSE 流式事件、重连、地图 artifact、知识库索引状态和错误降级。

我会用 Pinia 管理全局状态，用状态机管理聊天生命周期，用 TypeScript 类型约束协议，用端到端测试验证黄金链路。这些都是前端工程化能力。

### Q158：如果面试官问“你的项目怎么体现后端工程化”，怎么回答？

**回答：**

后端工程化体现在工具注册中心、schema 校验、权限控制、超时重试、会话幂等、SSE 事件持久化、RAG 增量索引和回归测试。

这些设计让系统不只是能跑通，而是更容易维护、排查和扩展。

---

### Q159：如果面试官问“你的项目怎么体现全栈能力”，怎么回答？

**回答：**

它覆盖了从前端交互到后端编排再到数据和外部服务的完整链路。前端要展示聊天、地图和知识库状态；后端要处理 Agent、RAG、工具和会话；数据层要支持 JSONL / Supabase；外部服务要接 LLM、MCP 和高德地图。

我不是只做某个页面或某个接口，而是把整条业务链路打通。

---

### Q160：如果面试官问“这个项目和企业知识库问答系统有什么相似和不同”，怎么回答？

**回答：**

相似点是都用了 RAG，都需要文档解析、chunk、embedding、检索和引用来源。不同点是这个项目不是纯知识问答，它还结合了结构化机厅数据和地图路线规划。

企业知识库更多回答“文档里怎么说”，这个项目还要回答“去哪、怎么筛、怎么走”，所以更强调 Agent 工具编排和地图 artifact。

---

### Q161：如果面试官问“你项目里最容易出 bug 的地方在哪里”，怎么回答？

**回答：**

最容易出 bug 的地方是跨模块状态同步，比如 SSE 事件到了但会话详情还没更新、路线 artifact 先于最终回复到达、用户刷新页面后需要恢复流式状态。

所以我做了事件 id、会话状态、状态机和最终详情刷新，确保实时过程和最终结果能对齐。

---

### Q162：如果面试官问“你怎么处理并发”，怎么回答？

**回答：**

主要处理两类并发。

第一是同一个用户重复提交，通过前端忙碌状态和后端 idempotency key 防止重复运行。  
第二是会话状态并发更新，通过 session 状态持久化和运行状态判断，避免一个 session 同时跑多个不同任务。

如果进一步做多实例，可以把锁和状态迁移到 Redis 或数据库。

---

### Q163：如果面试官问“你怎么处理日志”，怎么回答？

**回答：**

关键日志应该围绕 session id、trace id、工具名、耗时、状态和错误原因。这样一次用户请求可以从入口 API 追踪到 Agent、工具调用、RAG 检索和最终结果。

日志不能只打印自然语言问题，也要避免泄露敏感内容。

---

### Q164：如果面试官问“你怎么做接口设计”，怎么回答？

**回答：**

接口分成同步查询类和异步会话类。普通列表、详情、知识库状态可以用 REST 接口；Agent 问答这种长任务用“创建会话 + SSE 订阅事件”的方式。

这样接口职责比较清楚：REST 管资源状态，SSE 管实时过程。

---

### Q165：如果面试官问“为什么不用一次接口直接返回全部结果”，怎么回答？

**回答：**

因为 Agent 执行时间不稳定，可能涉及多个工具。一次接口直接等待会让用户盲等，也更容易超时。异步会话加 SSE 可以让用户实时看到系统在做什么，同时也方便断线重连和后台任务管理。

---

### Q166：如果面试官问“你怎么保证地图结果和文字回答一致”，怎么回答？

**回答：**

地图结果和文字回答都来自同一个 working memory。工具查到的候选门店、路线和目标点会先写入结构化状态，最终回答和地图 artifact 都从这里读取。

这样可以避免文本里说 A 店，地图却展示 B 店。

---

### Q167：如果面试官问“你怎么处理用户隐私位置”，怎么回答？

**回答：**

用户位置只在需要附近推荐或路线规划时使用，不应该无限期保存精确位置。日志里也不应该打印完整精确坐标。前端需要用户授权定位，用户拒绝时提供手动输入城市或地标的替代方案。

---

### Q168：如果面试官问“RAG 的 embedding 模型怎么选”，怎么回答？

**回答：**

选择 embedding 模型主要看语言、效果、成本和部署方式。中文场景可以选中文效果较好的 embedding 模型；本地 demo 可以用轻量模型或本地 fallback；线上可以接 OpenAI-compatible embedding API。

核心是让 embedding 能稳定表达中文机厅、地点、评论这类语义。

---

### Q169：如果面试官问“RAG 为什么还需要 rerank”，怎么回答？

**回答：**

初检索负责召回，可能会召回一些相关但不够准确的片段。rerank 负责对候选片段和 query 做更精细的相关性排序，把最有用的片段排到前面。

如果只靠向量相似度，有时实体或细节不够准。rerank 可以提升最终上下文质量。

---

### Q170：如果面试官问“如果回答需要引用来源，怎么实现”，怎么回答？

**回答：**

检索结果里保留 source_uri、文件名、页码、chunk_id 等 metadata。生成回答时可以把这些信息一并返回给前端，前端展示为“参考来源”或“命中文档”。

这样用户可以追溯回答依据，也方便排查 RAG 答错时是哪份资料的问题。

---

### Q171：如果面试官问“这个项目的下一步优化是什么”，怎么回答？

**回答：**

我会优先做三件事：

第一，把会话状态和 SSE 事件迁移到数据库或 Redis，增强多实例能力。  
第二，做 Agent Run Trace 页面，把每次意图、工具、耗时、token 和失败原因可视化。  
第三，完善 RAG eval，评估召回率、引用准确性和答案质量。

---

### Q172：如果面试官问“你能把这个项目迁移到别的业务吗”，怎么回答？

**回答：**

可以。这个项目的通用部分是 Agent 编排、工具注册、RAG 索引、SSE 流式会话和前端状态展示。业务部分是机厅数据和地图工具。

如果迁移到企业知识库、招聘简历筛选、客服系统或本地生活推荐，只需要替换数据源、工具和 prompt，底层工程框架可以复用。

---

### Q173：如果面试官问“这个项目怎么做权限系统”，怎么回答？

**回答：**

当前项目重点是业务链路。如果要加权限，可以分三层：

第一是用户登录和身份认证，比如 JWT。  
第二是数据权限，比如用户只能访问自己的会话和知识库文件。  
第三是工具权限，比如普通用户不能调用高成本或危险工具。

工具层本身已经有 allowed tools 的概念，可以继续扩展成用户级权限。

---

### Q174：如果面试官问“项目里有没有缓存”，怎么回答？

**回答：**

可以从几个角度讲：客户端缓存用户定位，避免每次都请求定位；后端可以缓存地理解析结果，避免同一地址反复调用地图 API；RAG 可以缓存查询结果或 embedding；前端可以缓存地图 SDK 加载状态，避免重复加载脚本。

缓存的原则是：高成本、重复性强、短时间内变化不大的结果适合缓存。

---

### Q175：如果面试官问“你怎么处理第三方 API 限流”，怎么回答？

**回答：**

可以通过超时、重试、限流、缓存和降级处理。比如地图 API 限流时，不应该无限重试；可以返回已有候选店铺，提示路线暂不可用。对于常见地点解析，可以缓存结果，减少重复请求。

---

### Q176：如果面试官问“怎么评估 RAG 答案质量”，怎么回答？

**回答：**

可以分三层评估：

第一是检索层，看相关文档是否被召回，比如 recall@k。  
第二是引用层，看答案是否引用了正确片段。  
第三是答案层，看回答是否正确、有用、没有幻觉。

可以用固定问题集加人工标注，也可以用 LLM judge 做辅助评估，但关键问题最好有人审。

---

### Q177：如果面试官问“Agent 和传统后端流程的边界在哪里”，怎么回答？

**回答：**

规则明确、风险高、结果必须稳定的部分交给传统后端流程，比如参数校验、权限、数据库查询和路线 API 调用。自然语言理解、任务拆解、结果组织这种不确定性较强的部分交给 Agent。

不能把所有业务逻辑都交给模型，也不能把模型只当模板生成器。边界要按确定性和风险来划分。

---

### Q178：如果面试官问“为什么简历这段能体现 AI 应用落地”，怎么回答？

**回答：**

因为它覆盖了 AI 应用从输入到输出的完整闭环：用户自然语言输入，Agent 做意图和任务分发，工具查询真实数据，RAG 补充非结构化知识，地图 API 生成路线，SSE 流式展示过程，最后还能做幂等、重连、测试和降级。

这比单纯“接一个模型接口”更接近真实业务系统。

---

### Q179：你标题写“AI 应用开发”，不是“AI 算法”，你怎么解释？

**回答：**

我更偏向 AI 应用落地，不是训练基础模型。这个项目重点是把 LLM、RAG、工具调用、地图 API、数据库和前端交互串成一个可用系统。

所以我的价值不是“造模型”，而是解决模型怎么接入业务、怎么调用真实工具、怎么控制失败、怎么把过程展示给用户。

**追问：那你认为 AI 应用开发最重要的能力是什么？**

**回答：**

我认为是三件事：第一，知道模型适合处理什么、不适合处理什么；第二，能把业务能力封装成稳定工具；第三，能做工程兜底，比如校验、重试、降级、可观测和测试。

---

### Q180：项目名里写“智能检索与路线规划”，这两个能力是分开的还是联动的？

**回答：**

底层能力是分开的，检索负责找到候选机厅，路线规划负责从用户位置到目标机厅生成路径。但在产品体验上它们是联动的。

比如用户问“附近有舞萌的店怎么去”，系统会先解析意图和位置，再检索符合条件的机厅，最后对候选结果做距离排序并生成路线。

---

### Q181：你写“自然语言问答”，用户输入会不会很随意？怎么保证能处理？

**回答：**

会很随意，所以不能只靠关键词匹配。我会先让 Agent 判断用户意图，比如搜索、附近推荐、路线规划、知识问答或闲聊，再抽取结构化参数，比如城市、机种、距离、是否需要路线。

如果信息不够，比如用户只说“附近有什么店”，系统会尝试读取定位上下文；如果没有定位，就引导用户补充城市或当前位置。

---

### Q182：如果用户一句话里既要搜索又要导航，系统怎么处理？

**回答：**

会拆成多步任务。比如“帮我找徐家汇附近有中二的机厅，然后给我路线”，它不是一个单纯查询，而是搜索加路线规划。

执行上先做机厅检索，拿到候选店铺和坐标，再选择最合适的目标生成路线。中间结果会进入 working memory，后续步骤直接复用，避免重复查。

---

### Q183：你写“条件筛选”，条件筛选是前端做还是后端做？

**回答：**

优先后端做，因为筛选和排序要尽量靠近数据源，结果也更可信。前端主要负责收集筛选条件、展示筛选状态和结果。

少量纯展示型排序可以在前端补充，比如用户临时切换列表展示顺序。但涉及距离、城市、机种、营业信息这类核心条件，应该由后端统一处理。

---

### Q184：附近推荐的“附近”如何定义？固定半径吗？

**回答：**

可以按默认半径加动态扩展来做。比如先查 3 公里或 5 公里内的机厅，如果结果太少，再扩大到 10 公里。

同时要结合城市密度，不同城市策略不同。市中心机厅密集，可以半径小一点；郊区或数据少的城市，可以扩大范围，并在回答里说明距离。

---

### Q185：如果两个机厅距离差不多，推荐顺序怎么定？

**回答：**

距离只是一个因素，还可以看机种匹配度、是否有目标机台、资料完整度、用户问题里的偏好，以及知识库里的评论或活动信息。

比如用户明确说想打某个机种，那机种匹配度优先；用户说“最近的”，那距离优先。

---

### Q186：你写“历史会话管理”，历史会话对 Agent 有什么帮助？

**回答：**

历史会话不只是展示聊天记录，它还能帮助 Agent 理解上下文。比如用户上一轮查了三家店，下一轮问“第二家怎么走”，系统需要知道“第二家”指哪一个。

所以历史会话要保存用户消息、助手回复、结构化结果、地图 artifact 和关键上下文，而不是只存一段文本。

---

### Q187：历史会话会不会导致上下文越来越长？

**回答：**

会，所以不能把全部历史都塞给模型。可以做摘要和关键状态提取：保留最近几轮对话，加上结构化 memory，比如用户位置、候选店铺、已选目标、路线结果。

这样既能保持连续对话，又能控制 token 成本。

---

### Q188：你写“main_agent -> worker -> tool registry”，为什么 worker 不直接写死逻辑？

**回答：**

写死逻辑短期快，但后面不好扩展。worker 的职责是处理某类任务，比如搜索或导航；工具注册层负责工具发现、参数校验和执行。

这样新增工具时，不需要把所有 worker 都改一遍，只要注册工具并描述能力，Agent 就能按规则调用。

---

### Q189：main agent 如何决定交给哪个 worker？

**回答：**

根据用户意图、上下文和任务类型决定。搜索类问题交给检索 worker，路线类问题交给导航 worker，知识类问题走 RAG。

如果一个请求包含多个任务，main agent 会拆解成多个步骤，由不同 worker 协作完成。

---

### Q190：如果 main agent 判断错 worker 怎么办？

**回答：**

首先在 prompt 和工具描述里约束意图边界。其次 worker 执行前可以做参数检查，如果发现缺少关键字段或任务不匹配，可以返回可恢复错误。

最后用固定问句集做回归测试，比如搜索、导航、知识问答各准备一批典型问题，观察路由是否稳定。

---

### Q191：你说“工具发现”，工具发现具体发现什么？

**回答：**

发现工具名、工具描述、入参 schema、权限要求、超时时间、重试策略和返回结构。

模型不应该凭空调用工具，而是从注册表里选择可用工具；后端也会再次校验工具是否存在、参数是否合法。

---

### Q192：manifest 里一般会放哪些内容？

**回答：**

通常会放工具名称、功能描述、输入参数 schema、输出结构说明、权限级别、超时设置、是否允许重试、预算限制和降级策略。

这能让工具从“随便调一个函数”变成“可治理的能力单元”。

---

### Q193：JSON Schema 在工具调用里有什么实际作用？

**回答：**

它主要做参数约束，比如字段类型、必填项、枚举值、数组结构和范围限制。

比如路线规划必须有起点和终点坐标，机种筛选只能是支持的机种枚举。这样可以提前拦住模型生成的错误参数，避免错误一路传到外部 API。

---

### Q194：executor 的职责是什么？

**回答：**

executor 是真正执行工具逻辑的部分。它接收已经校验过的参数，调用数据库、地图 API、RAG 检索或其他服务，然后返回结构化结果。

这样 manifest 管描述，schema 管校验，executor 管执行，职责比较清楚。

---

### Q195：权限控制在工具层怎么体现？

**回答：**

可以给工具设置权限级别和调用条件。比如普通查询工具默认开放，高成本工具需要预算限制，涉及用户私有数据的工具必须校验用户身份。

这样即使模型想调用某个工具，后端也会做最终权限判断。

---

### Q196：工具预算怎么设计？

**回答：**

可以按一次用户请求设置预算，比如最多调用几次地图 API、最多做几次 RAG 检索、最多重试几次。

预算的目的不是省一点成本而已，更重要是防止 Agent 陷入循环调用或因为外部服务失败而一直重试。

---

### Q197：如果工具返回的数据很多，怎么交给模型？

**回答：**

不能把全部原始数据都塞给模型。应该先做结构化裁剪，比如只保留前 N 条候选、核心字段、距离、匹配原因和必要 metadata。

如果需要展示完整列表，可以由前端直接展示结构化结果，模型只负责总结和解释。

---

### Q198：RAG 为什么适合放“评论、活动信息”？

**回答：**

因为这些信息不一定适合放进结构化表。评论、活动公告、玩家经验、门店说明都是自然语言文本，字段不固定，更新来源也多。

RAG 可以把这些资料切片索引，在用户问到“哪家环境好”“最近有没有活动”时作为补充依据。

---

### Q199：RAG 和 Supabase 数据库分别存什么更合适？

**回答：**

数据库适合存结构化、强约束、经常筛选排序的数据，比如门店名称、地址、坐标、机种、城市、更新时间。

RAG 适合存非结构化文本，比如攻略、评论、活动说明、上传文档和运营资料。两者结合，回答既有准确字段，也有文本补充。

---

### Q200：你写“文档上传、增量索引、混合检索、失败重试”，这几个能力之间的关系是什么？

**回答：**

文档上传是入口，增量索引负责把新文件或变更文件转成可检索 chunk，混合检索负责查询时同时使用关键词和语义召回，失败重试负责保证索引任务或检索任务更稳定。

这是一条完整链路，不是孤立功能。

---

### Q201：为什么文档上传后不要同步阻塞等待索引完成？

**回答：**

因为 PDF 解析、切片、embedding 都可能比较慢。如果上传接口一直等索引结束，用户体验差，也容易超时。

更好的方式是上传后生成后台索引任务，前端展示 pending、indexing、ready、failed 等状态，用户可以看到进度，失败后也能重试。

---

### Q202：混合检索里的关键词检索和向量检索各自解决什么？

**回答：**

关键词检索适合精确词，比如店名、机种名、地名、活动名称。向量检索适合理解语义，比如“环境比较好”“适合新手”“离我近一点”。

两者结合可以避免单一检索方式的短板。

---

### Q203：如果混合检索两个结果排序冲突，怎么合并？

**回答：**

可以用 RRF 这种倒数排名融合方法。它不直接比较不同算法的原始分数，而是看各自排名，把多个排序列表融合成一个更稳的列表。

简单说，两个检索方式都排得靠前的结果会更靠前；只在一个检索方式里靠前的结果也有机会保留。

---

### Q204：RAG 失败重试要注意什么？

**回答：**

不是所有失败都应该重试。临时网络错误、embedding 服务超时可以重试；文件格式不支持、内容为空、权限错误就不应该重复重试。

所以失败要分类记录原因，避免无意义重试。

---

### Q205：你写“异步会话派发”，异步解决了什么具体问题？

**回答：**

它解决的是长任务等待问题。一次 Agent 请求可能要调用模型、数据库、RAG 和地图 API，如果同步等待，前端会一直卡住。

异步派发后，提交请求只负责创建任务和返回会话标识，执行过程通过事件流持续推给前端。

---

### Q206：SSE 为什么适合展示 Agent 执行过程？

**回答：**

因为 Agent 执行是服务端持续产生事件，前端主要接收，不需要频繁双向通信。SSE 正好适合服务端向浏览器单向推送。

它比轮询实时，比 WebSocket 简单，也天然支持文本事件流和断线重连。

---

### Q207：SSE 事件怎么设计才不会乱？

**回答：**

事件要有统一协议，比如事件名、session_id、offset、timestamp、payload 和 error 字段。每类事件的 payload 要有 schema。

前端不应该靠猜字段解析，而应该按事件类型做严格处理。后端增加字段要兼容旧前端，删除字段要谨慎。

---

### Q208：前端实时展示“工具状态”有什么产品价值？

**回答：**

它能让用户知道系统正在做什么。比如正在检索机厅、正在解析位置、正在规划路线，比一个加载圈更可信。

对 AI 应用来说，透明的执行过程能降低用户焦虑，也方便失败时解释原因。

---

### Q209：流式回复和地图结果如何同步？

**回答：**

文本 token 可以边生成边展示，地图结果则通过结构化 artifact 事件单独下发。前端收到地图 artifact 后更新地图区域，收到文本 token 后更新聊天气泡。

这两个通道都属于同一个 session，通过 session_id 和 offset 保证归属和顺序。

---

### Q210：如果文本已经生成，但地图结果失败了，怎么处理？

**回答：**

不应该让整个回答失败。可以继续展示文本结果，同时明确提示路线或地图暂时不可用，并给出失败原因或替代建议。

比如“已找到候选机厅，但路线规划服务暂时失败，可以先查看地址和距离”。

---

### Q211：JSONL 数据源有什么优缺点？

**回答：**

优点是简单、可版本管理、适合本地开发和小规模数据；缺点是并发能力、查询能力、权限控制和在线更新都有限。

所以它适合作为 demo、种子数据或离线备份，不适合长期承载复杂线上查询。

---

### Q212：Supabase 数据源有什么优缺点？

**回答：**

优点是结构化查询、索引、权限控制、在线更新和部署都比较方便；缺点是需要设计表结构、迁移和访问策略，也要考虑网络延迟和服务限额。

在项目里它更适合作为正式数据源。

---

### Q213：双数据源如何保证结果一致？

**回答：**

要定义统一的数据模型。无论数据来自 JSONL 还是 Supabase，进入业务层前都转换成同一种门店对象结构。

这样上层的筛选、排序、地图展示和 Agent 总结不用关心底层数据来源。

---

### Q214：高德地图 API 里“定位”和“逆地理编码”有什么区别？

**回答：**

定位是拿到用户当前经纬度，逆地理编码是把经纬度转换成可读地址，比如城市、区县、街道。

在项目里，定位用于附近推荐和路线起点，逆地理编码用于让用户知道系统理解的位置是否正确。

---

### Q215：路线规划需要哪些输入？

**回答：**

至少需要起点坐标、终点坐标和出行方式，比如步行、公交、驾车。业务上还需要目标机厅信息，这样前端才能把路线和门店卡片对应起来。

如果缺少坐标，就要先做地址解析或提示用户补充。

---

### Q216：地图点位展示时如何避免前端被后端结构绑定死？

**回答：**

后端返回结构化 artifact，里面带 schema_version。前端根据版本选择对应 renderer 或 adapter，把后端数据转换成地图组件需要的格式。

这样后端以后升级地图场景结构，不会直接打坏旧前端。

---

### Q217：Vue 3 在这个项目里适合怎么拆组件？

**回答：**

可以按业务区域拆：聊天面板、消息列表、工具进度、地图视图、筛选面板、会话列表、知识库文件管理。

复杂逻辑放到 composable 和 Pinia store 里，组件尽量负责展示和交互。

---

### Q218：Pinia 里哪些状态应该全局，哪些不应该？

**回答：**

会话列表、当前会话、消息流、地图 artifact、用户位置、知识库文件状态适合放 Pinia，因为多个组件会共享。

输入框临时文本、弹窗开关、局部 hover 状态不一定要全局，放组件内部更简单。

---

### Q219：前端接 SSE 时为什么需要状态机？

**回答：**

因为一次会话不是简单的 loading true/false。它会经历 idle、dispatching、streaming、reconnecting、completed、failed 等状态。

状态机能避免多个布尔值互相打架，比如既显示“发送中”又显示“已完成”。后续加重连、恢复、重复提交保护也更清楚。

---

### Q220：Vite 对这个项目的价值是什么？

**回答：**

Vite 主要提升开发体验和构建效率。前端需要频繁调聊天、地图和状态交互，快速热更新能明显提高开发效率。

同时它对 Vue 3 和 TypeScript 支持成熟，适合中小型前端应用快速迭代。

---

### Q221：TypeScript 在 SSE 协议里有什么价值？

**回答：**

SSE payload 类型很多，如果不用类型约束，前端很容易把字段写错或漏处理事件。

用 TypeScript 定义事件联合类型，可以让不同事件有不同 payload 结构，编译阶段就发现很多协议不一致问题。

---

### Q222：FastAPI 为什么适合做这个后端？

**回答：**

FastAPI 对异步接口、类型标注、数据校验和 OpenAPI 文档支持比较好。这个项目有 SSE、后台任务、外部 API 调用和工具参数校验，FastAPI 比较契合。

另外 Python 生态做 RAG、PDF 解析和 LLM SDK 接入也方便。

---

### Q223：MCP 在简历里怎么讲才不虚？

**回答：**

不要只说“用了 MCP”，要说它解决了什么问题。比如 MCP 可以把外部能力用统一协议暴露给 Agent，让工具更容易接入、发现和治理。

在回答里可以强调：我把 MCP 看作 Agent 连接外部系统的一层协议，而不是一个噱头。

---

### Q224：OpenAI-compatible LLM 是什么意思？

**回答：**

意思是后端调用模型时尽量使用兼容 OpenAI 接口风格的协议，这样可以替换不同模型提供方。

好处是项目不会强绑定某一个模型，后续可以按成本、速度、上下文长度和工具调用能力切换。

---

### Q225：如果换模型后效果变差，怎么发现？

**回答：**

要靠固定评测集。比如准备一批搜索、导航、RAG、上下文追问和异常问题，每次换模型或改 prompt 后跑一遍，看意图路由、工具调用、答案质量是否退化。

不能只靠手动试两三个问题。

---

### Q226：你这个项目如何体现“产品化”？

**回答：**

产品化体现在用户不是面对一个裸聊天框，而是能看到筛选、地图、路线、工具进度、历史会话和知识库状态。

同时系统考虑了失败提示、断线重连、幂等、索引状态和回归测试，这些都是从 demo 走向可用产品需要补的部分。

---

### Q227：面试官问“你做过真实 AI 应用里最容易踩坑的点是什么”，怎么答？

**回答：**

我会说最容易踩坑的是状态一致性。AI 应用链路长，一次请求可能跨前端、后端、模型、工具、RAG、地图 API 和数据库。

如果没有 session_id、offset、trace id、事件协议和状态机，很容易出现重复提交、流式错乱、工具结果和文本不一致的问题。

---

### Q228：面试官问“为什么不用纯 Prompt 解决所有问题”，怎么答？

**回答：**

因为 prompt 适合自然语言理解和表达，不适合承担确定性业务逻辑。比如数据库筛选、坐标计算、路线规划、权限校验都应该由工具和后端完成。

模型负责决策和组织，工具负责真实执行，这样结果更可靠。

---

### Q229：面试官问“这个项目最大的不确定性是什么”，怎么答？

**回答：**

最大的不确定性来自模型输出和外部服务。模型可能意图判断错、参数生成错，地图 API 和 embedding 服务也可能失败。

所以我通过 schema 校验、工具权限、失败分类、降级提示、固定评测集和事件追踪来控制这些不确定性。

---

### Q230：面试官问“你怎么用一句话总结这个项目”，怎么答？

**回答：**

这是一个面向音游玩家的 AI 本地生活应用：用户用自然语言提出需求，Agent 调用检索、RAG 和地图工具，最终以流式对话加地图路线的形式给出可执行结果。

这句话能同时覆盖业务场景、AI 能力、工具调用和前端体验。

## 19. 基于源码实现的深挖追问

### Q231：异步会话派发具体是怎么落地的？

**回答：**

实现上不是前端直接等模型完整返回，而是先提交一次会话请求。后端创建或复用 session，生成运行状态，然后把 Agent 执行放到后台任务里跑，接口先返回 session_id、run_status、idempotency_key 和最后的流 offset。

前端拿到 session_id 后再打开 SSE 连接接收事件。这样一次请求即使包含模型调用、工具调用、地图规划、RAG 检索，用户也能实时看到过程，而不是一直卡在 loading。

**追问：如果后台任务失败怎么办？**

**回答：**

后端会把 session 状态标记为 failed，记录 last_error，并发出 session.failed 事件。前端收到后把生命周期切到 failed，展示错误提示，同时历史会话里也能看到这次失败。

---

### Q232：idempotency_key 在项目里解决了什么问题？

**回答：**

它解决的是用户重复提交和网络重试问题。比如用户点了两次发送，或者前端请求超时后重发，如果没有幂等键，后端可能启动两个 Agent run，造成重复扣费、重复工具调用、结果相互覆盖。

项目里同一个 session 下，如果传入相同 idempotency_key，后端会认为它是同一次运行，已经在跑就返回已有状态，已经结束就返回已有 offset，而不是重新执行。

---

### Q233：如果同一个会话正在运行，用户又发一条新消息怎么办？

**回答：**

后端会做运行占用判断。如果是同一个幂等请求，就复用原运行；如果是新的请求，则返回冲突，避免同一个 session 同时跑两个 Agent。

这样能保证 working memory、历史 turns、SSE 事件和最终回复不会被两个并发任务写乱。

---

### Q234：SSE 断线续播是怎么实现的？

**回答：**

每个 SSE 事件都有递增 id，后端会把事件按 session 存进 replay buffer，并记录最后一个 stream offset。前端重连时带上 last_event_id，后端只补发这个 id 之后的事件。

如果服务重启，事件也可以从持久化日志恢复一部分，至少能保证已经发出的关键事件不会因为短暂断线全部丢失。

---

### Q235：为什么既要保存 session 状态，又要保存 SSE 事件？

**回答：**

它们解决的问题不同。session 状态保存的是最终业务状态，比如历史消息、当前状态、working memory、地图结果、最后错误；SSE 事件保存的是过程流，比如工具开始、工具完成、token 输出。

页面刷新后，session 状态能恢复最终结果；断线重连时，SSE 事件能补齐中间过程。两者结合，体验更完整。

---

### Q236：前端为什么要按生命周期状态机处理聊天？

**回答：**

因为聊天不是一个简单的 loading。源码里把它收束成 idle、dispatching、streaming、reconnecting、completed、failed 这些状态。

这样可以明确规定：什么时候允许发送，什么时候显示恢复连接，什么时候终止流，什么时候展示失败。比 sending、connected、replying 这类布尔值组合更不容易出现矛盾状态。

---

### Q237：如果用户切换会话，旧 SSE 事件还在回来怎么办？

**回答：**

前端应该用 session_id 做归属校验，只处理当前会话对应的事件。切换会话时关闭旧 EventSource 或忽略旧 session 的事件。

后端事件本身也带 session_id，前端解析时可以二次校验，避免旧流把新页面的消息或地图覆盖掉。

---

### Q238：SSE 事件协议如何保证前后端一致？

**回答：**

后端有强类型事件模型，每个事件名对应固定 payload；前端使用生成出来的事件名和类型做解析，不再手写一份不受约束的字符串。

此外还可以做 contract test，验证后端导出的事件名、前端识别的事件名、关键 payload 字段一致。这样协议变更能在测试阶段暴露，而不是上线后才发现前端解析失败。

---

### Q239：assistant.token 事件里为什么既有 delta 又有 content？

**回答：**

delta 适合前端做增量拼接，content 适合做当前累计内容或兜底恢复。只传 delta 的问题是，一旦中间漏事件，前端拼接可能不完整。

同时有 index、total 这类字段，也方便前端判断 token 顺序和展示进度。

---

### Q240：工具调用结果怎么进入对话上下文？

**回答：**

工具结果会被记录成 tool turn，同时关键结果会写入 working memory。比如检索结果写入 shops，路线写入 route，知识库命中写入 knowledge_hits，用户位置写入 client_location。

后续 worker 或 summary 阶段不需要重新查一次，可以直接从 memory 里读取结构化结果。

---

### Q241：working memory 和聊天历史有什么区别？

**回答：**

聊天历史是面向对话的，记录用户、助手和工具说过什么；working memory 是面向执行的，保存结构化状态。

比如“第二家怎么走”这种问题，聊天历史里只有文本，但 working memory 里有候选店铺列表、source_id、坐标和上一次查询条件，Agent 才能稳定理解“第二家”。

---

### Q242：worker 执行完后，结果怎么回到 main agent？

**回答：**

worker 有自己的临时执行状态，完成后会把关键 artifacts 提升回父级 memory，比如 shops、route、destination、view_payload、knowledge_hits。

main agent 再基于这些结构化结果决定是否继续调用工具，还是进入总结阶段生成最终回复。

---

### Q243：项目里如何防止 Agent 无限循环调用工具？

**回答：**

有两层控制。第一层是执行步数上限，超过最大步骤就停止。第二层是工具治理，限制每个工具或工具组在一次 run 里的调用次数。

例如知识检索、地图、导航、总结都有预算，避免模型在失败时不断重复调用同一个工具。

---

### Q244：工具治理在源码里具体包括哪些能力？

**回答：**

包括单工具超时、重试次数、熔断阈值、熔断恢复时间、工具组预算、单工具调用预算、trace id、tool trace id、fallback reason 和耗时统计。

这能让工具调用不只是“能调”，而是可控、可观测、可降级。

---

### Q245：工具 timeout 和 retry 怎么设计比较合理？

**回答：**

不同工具应该不同策略。数据库查询通常超时短、少重试；地图和知识检索可能允许一次重试；总结工具超时也不能太长，否则用户会一直等待。

重试只适合临时错误，比如网络抖动或外部服务超时；参数错误、权限错误、schema 校验失败不应该重试。

---

### Q246：circuit breaker 在工具层解决什么问题？

**回答：**

如果某个外部服务连续失败，比如地图服务不可用，继续每次都调用只会浪费时间并拖慢用户体验。

熔断器会在连续失败达到阈值后暂时打开，短时间内直接返回失败或走降级，等恢复时间过后再尝试。它保护的是系统整体稳定性。

---

### Q247：tool trace id 有什么用？

**回答：**

一次用户请求可能调用多个工具，trace id 能把这些调用串起来。tool trace id 还能定位到某一次具体工具调用。

排查问题时，可以看这次请求调用了哪些工具、每个工具耗时多少、失败原因是什么、有没有触发 fallback 或预算限制。

---

### Q248：工具权限控制怎么做？

**回答：**

每个 Agent 或 worker 有 allowed tools，只能调用允许范围内的工具。工具执行前后端会校验工具名是否匹配允许列表，而不是模型说调用什么就调用什么。

这能防止主 Agent 调用不该调用的工具，也方便不同 worker 收敛职责。

---

### Q249：如果模型输出了不存在的工具，源码会怎么处理？

**回答：**

工具注册表查不到这个工具时会返回 unknown tool 相关错误，执行不会继续下沉到业务服务。

这类错误会被记录成工具失败，并写入事件和 tool turn，方便后续总结或排查。

---

### Q250：工具参数是怎么校验的？

**回答：**

每个工具都有 JSON Schema 描述输入结构。执行前会用 schema 校验字段类型、必填字段、枚举值和数值范围。

比如路线规划需要起点、终点、provider、mode；检索工具需要分页参数、筛选字段和排序字段。校验失败会返回 validation_error，而不是让错误进入数据库或地图 API。

---

### Q251：manifest + schema + executor 这套设计怎么新增一个工具？

**回答：**

新增工具时先写 manifest 条目，让系统能发现它；再写输入 JSON Schema，定义参数边界；最后写 executor，真正调用业务服务。

如果工具需要依赖某个服务，还可以在 provider 层配置依赖注入。这样新增工具不需要改 Agent 主流程，扩展成本更低。

---

### Q252：MCP 工具和内建工具怎么共存？

**回答：**

工具注册表会聚合多个 provider，内建工具来自本地 manifest，MCP 工具来自远端发现。聚合时会检查工具名冲突，再统一暴露给 Agent。

对 Agent 来说，最终看到的是同一套 tool definitions；对后端来说，执行时再路由到对应 provider。

---

### Q253：地图路线为什么要做多级 fallback？

**回答：**

路线规划依赖外部地图服务，存在 key 配错、额度不足、网络失败、MCP 工具不可用等情况。源码里会优先走 MCP 或高德路线能力，不可用时退化到内建路线规划，再不行至少给出直线距离或估算提示。

这样用户不会因为地图服务短暂失败就完全没有结果。

---

### Q254：地图 artifact 为什么不仅返回路线，还返回 scene？

**回答：**

因为地图有不同场景：候选机厅展示和具体路线展示。scene 能告诉前端当前应该用哪种 renderer 或交互方式。

如果只有 route 字段，前端只能猜展示模式；有 schema_version 和 scene 后，前后端协议更清晰，也更容易兼容后续地图升级。

---

### Q255：地图坐标为什么要区分 WGS84 和 GCJ02？

**回答：**

浏览器定位通常是 WGS84，而高德地图使用 GCJ02。若不区分坐标系，点位会偏移，路线也可能不准确。

所以数据结构里明确 coord_system，门店 geo 可以同时保存 gcj02 和 wgs84，路线和地图渲染时按需要转换或选择正确坐标。

---

### Q256：浏览器定位和后端逆地理编码为什么要分开？

**回答：**

浏览器定位更适合在前端拿，因为它需要用户授权；逆地理编码更适合后端做，因为高德 REST key 不应该暴露给浏览器。

前端拿到经纬度后传给后端，后端补齐省市区、adcode、格式化地址，再写入 session memory。定位失败时也不阻塞聊天，只是附近推荐能力会弱一点。

---

### Q257：RAG 知识库的“增量索引”在源码里怎么判断文件变化？

**回答：**

它会扫描知识源目录，记录每个文件的相对路径、后缀、大小、更新时间和内容 hash。索引时比较当前快照和已索引状态。

如果文件没变，就复用旧 chunk 和旧 embedding；如果文件变了，只重建这个文件的 chunk；如果文件删除，就把对应 chunk 从索引里移除。

---

### Q258：增量 chunk diff 是怎么减少成本的？

**回答：**

同一个文件内，切片后会计算每个 chunk 的 hash。如果某些 chunk 内容没变，就复用之前的 embedding；只有新增或变更的 chunk 才重新 embedding。

这对 PDF 或长文档很有意义，因为上传一个小改动时不需要整份文件重新向量化。

---

### Q259：知识库文件状态有哪些？前端怎么展示？

**回答：**

每个文件有 pending、indexing、ready、failed 状态，同时有 chunk_count、content_hash、indexed_at、error、job_id 这些信息。

前端可以据此展示“等待索引”“索引中”“可检索”“失败可重试”，而不是只给用户一个模糊的上传成功。

---

### Q260：知识库后台 job 队列解决了什么？

**回答：**

它把上传接口和索引任务解耦。上传文件后只需要入队索引任务，接口可以快速返回；后台 worker 再慢慢解析、切片、embedding、刷新索引。

这样不会因为大文件解析慢导致上传请求超时，也方便后续扩展进度展示和失败重试。

---

### Q261：RAG 支持哪些文件格式？不同格式怎么处理？

**回答：**

支持 md、txt、json、jsonl、pdf、docx、doc 等。文本类文件直接读取；json/jsonl 会优先解析 content、text、title、source_uri 等字段；PDF 按页提取文本并保留页码来源；docx 会读取文档 XML 中的文本。

最终都会统一转换成文档对象，再进入切片、embedding 和检索流程。

---

### Q262：PDF 处理有什么局限？

**回答：**

当前更适合可复制文本型 PDF，通过文本抽取拿到每页内容。如果是扫描版图片 PDF，就需要 OCR，否则提取不到有效文本。

面试时可以补一句：如果要加强 PDF 能力，会加入 OCR、表格抽取和页码级引用。

---

### Q263：RAG 切片策略是什么？

**回答：**

默认按字符长度和 overlap 切片，并优先使用段落、句号、逗号、空格等分隔符，尽量不把一句话切断。配置允许时也支持语义切片，失败后回退到普通切片。

chunk 太大召回不准、上下文浪费；chunk 太小信息不完整，所以要结合文档类型和模型上下文调整。

---

### Q264：源码里的混合检索是怎么融合分数的？

**回答：**

源码里是向量检索和 BM25 关键词检索分别打分，然后做归一化，再按 alpha 加权融合。

这样关键词命中和语义相似都能影响最终排名。面试里如果提到 RRF，可以说 RRF 是另一种可选融合方案，但当前实现更偏加权分数融合，便于通过 alpha 调整语义和关键词的权重。

---

### Q265：为什么还要 reranker？

**回答：**

第一阶段检索更像“召回”，会尽量找出可能相关的候选。reranker 更像“精排”，用 query 和候选文本重新计算相关性，把最可能有用的片段排到前面。

项目里 reranker 是可开关的，失败时会退回原始排序，避免重排序服务影响主链路。

---

### Q266：RAG 查询缓存怎么设计？

**回答：**

查询缓存会把 query、top_k、索引配置、数据源签名、是否混合检索、是否 reranker 等因素合成 cache key。

这样同一个问题在索引没变时可以复用结果；一旦文件变化、chunk 配置变化或检索配置变化，cache key 也会变化，避免拿到过期结果。

---

### Q267：FAISS 持久化解决什么问题？

**回答：**

如果向量索引只在内存里，服务重启后要重新解析所有文档并 embedding，启动慢且成本高。

FAISS 持久化会保存索引文件和 metadata sidecar，包括 embedding 模型签名、chunk 配置、source signature、file states 和 chunk records。重启时只要签名匹配，就可以直接恢复。

---

### Q268：知识库命中结果如何和机厅业务结合？

**回答：**

知识库命中不只是返回文本片段，还会尝试从 metadata 或文本里提取机厅候选，比如店名、地址、城市、交通、坐标等。

如果命中里没有坐标，还可以用地理解析补齐。这样 RAG 不只是“回答一段话”，也能产出可被地图展示的候选结果。

---

### Q269：结构化数据库没查到，RAG 会怎么参与？

**回答：**

搜索 worker 如果结构化检索没有结果，会尝试用知识库检索做补充。这适合处理数据库尚未收录、来自评论或活动文档的信息。

这是一种降级和补召回机制：数据库优先保证准确，RAG 在空结果或非结构化问题里补充覆盖面。

---

### Q270：JSONL 和 Supabase 的统一模型怎么做？

**回答：**

底层数据源不同，但进入业务层前会映射成统一的机厅 DTO，比如名称、地址、省市区、机种数量、距离、坐标、更新时间等。

这样 Agent、工具、地图组件不需要关心数据来自 JSONL 还是 Supabase，只消费统一结构。

---

### Q271：Supabase 为什么通过 RPC 查询，而不是前端直接查表？

**回答：**

RPC 可以把搜索、筛选、排序、分页和距离计算封装在后端数据库函数里，接口更稳定，权限也更容易控制。

前端直接查表会暴露更多数据结构细节，也更容易出现权限和查询复杂度问题。

---

### Q272：如果 Supabase RPC 版本变化怎么办？

**回答：**

源码里对部分参数有兼容逻辑，比如新参数不可用时可以退回到旧的关键词查询方式。更完整的做法是给 RPC 加版本号，后端 repository 按版本适配。

面试里可以强调：外部数据接口要有兼容层，不能让业务逻辑直接依赖某个临时 SQL 形态。

---

### Q273：附件上传和知识库上传有什么区别？

**回答：**

附件上传是跟随一次聊天请求的临时上下文，比如用户发一张图或一个文档让 Agent 参考；知识库上传是长期索引，后续所有相关查询都可以检索到。

所以附件会提取预览和文本，放进本次请求；知识库文件会进入索引任务，生成 chunk、embedding 和 metadata。

---

### Q274：聊天附件里的 PDF 怎么处理？

**回答：**

聊天附件里的 PDF 会提取前几页文本，做长度裁剪后放入本次请求上下文，避免一次上传大 PDF 把模型上下文撑爆。

而知识库 PDF 是按页进入 RAG 索引，更适合长期检索和页码溯源。

---

### Q275：前端 API 层做了哪些工程化处理？

**回答：**

包括统一 API base、请求超时、AbortController、trace id、JSON 响应校验、错误对象封装、上传请求单独超时，以及 SSE URL 构造。

这些细节能让前端不是散落 fetch，而是有统一的请求边界和错误处理方式。

---

### Q276：如果后端返回非 JSON，前端怎么处理？

**回答：**

前端会检查 content-type，如果不是 JSON，会抛出明确错误，并截取一段响应预览。这样排查 Nginx 错误页、后端异常 HTML、代理配置错误时更快。

这也是前端工程化的一部分：错误要可诊断，而不是只显示“请求失败”。

---

### Q277：简历写 Vue 3 + Pinia，但源码现在不是 Vue，怎么避免被问穿？

**回答：**

最稳的做法是简历只写真实完成的技术栈。如果确实已经做了 Vue 版本或迁移分支，就可以写 Vue 3 + Pinia。

如果只是计划迁移，不建议写成已实现。可以改成“前端状态管理可迁移到 Vue 3 + Pinia”，或者在面试中说明原实现是 React/Zustand，自己理解 Vue 版本该怎么拆 composable、store 和组件。

---

### Q278：如果面试官问“你项目里最真实的线上问题是什么”，怎么答？

**回答：**

我会答“状态一致性和外部依赖不稳定”。比如用户重复点击、页面断线、后台任务还在跑、地图 API 失败、RAG 索引未完成，这些都不是 prompt 能解决的。

所以项目里做了 idempotency_key、session 状态、SSE offset、replay buffer、工具超时重试、fallback 和知识库文件状态。

---

### Q279：如果面试官让你画源码链路，怎么讲？

**回答：**

可以按这条线讲：前端提交会话请求，后端保留 session 并派发后台任务；主 Agent 读取上下文和工具定义，决定是否调用 worker；worker 调用检索、知识库、地理解析或路线工具；工具结果写入 working memory；后端持续发 SSE 事件；前端根据事件更新进度、消息和地图；最终 session 存下历史和 artifacts。

这条链路既有 AI 决策，也有传统后端的确定性控制。

---

### Q280：如果面试官问“源码里哪里最能体现工程化”，怎么答？

**回答：**

我会选三点：第一，SSE 协议和 session 持久化，解决断线、刷新和重复提交；第二，工具注册和治理，解决工具扩展、校验、权限、超时、预算和追踪；第三，RAG 增量索引，解决文档上传后的状态、失败重试和索引成本。

这三点都不是单纯调用模型，而是把 AI 应用变成可维护系统。

---

## 20. 功能实现与 FastAPI 深挖追问

### Q281：FastAPI 在这个项目里承担什么角色？

**回答：**

FastAPI 是后端应用入口，负责暴露 HTTP API、SSE 流式接口、请求参数校验、依赖注入、生命周期初始化和统一响应模型。

更具体地说，普通接口负责机厅列表、详情、知识库、会话管理；SSE 接口负责把 Agent 执行过程推给前端；启动时会初始化数据源、RAG、工具注册表、MCP、会话存储和地图服务。

---

### Q282：FastAPI 项目启动时做了哪些初始化？

**回答：**

启动阶段会读取环境变量生成配置，然后构建一个应用级容器，把数据仓库、RAG 服务、工具注册表、LLM provider、地图服务、session store、replay buffer 和 orchestrator 都创建出来。

之后在 lifespan 启动钩子里刷新工具列表、预热 RAG 索引、检查数据源健康状态，并记录日志。这样请求进来时不用重复创建重对象。

---

### Q283：为什么要用应用级容器，而不是在每个接口里 new 服务？

**回答：**

因为很多服务是长生命周期对象，比如数据仓库、HTTP client、RAG 索引、工具注册表、session store、replay buffer。如果每次请求都创建，性能差，也容易造成状态不一致。

应用级容器把依赖统一构建好，再通过 FastAPI 的 Depends 注入到路由里。这样结构清楚，也方便测试时替换依赖。

---

### Q284：FastAPI 的 Depends 在项目里怎么用？

**回答：**

每个路由不直接 import 全局变量，而是通过 Depends 从 request.app.state 里取出共享容器。

好处是路由层只关心“我要用哪个服务”，不用关心服务怎么创建；测试时也可以通过创建新的 app 实例和环境变量来构造隔离的依赖。

---

### Q285：FastAPI 的 response_model 有什么用？

**回答：**

response_model 会把接口返回约束到 Pydantic 模型上，自动校验和序列化响应字段。

比如机厅列表、聊天会话、知识库状态、地图路线都有明确 DTO。这样前端拿到的数据结构更稳定，也能自动生成 OpenAPI 文档，方便联调。

---

### Q286：请求参数校验是怎么做的？

**回答：**

FastAPI 会结合 Query、Form、File 和 Pydantic 模型做校验。比如分页 page 必须大于等于 1，page_size 有上限，经纬度要在合法范围内，上传文件大小有限制，聊天请求必须有消息或附件。

这些校验放在接口入口处，可以尽早拦截错误请求，避免脏数据进入 Agent 或工具层。

---

### Q287：FastAPI 如何处理 JSON 请求和文件上传请求？

**回答：**

JSON 请求适合普通聊天、检索、会话查询；文件上传请求使用 multipart form，适合聊天附件和知识库文件上传。

项目里同一类能力通常有两个入口：纯 JSON 入口处理普通请求，upload 入口先解析文件、提取文本或图片信息，再组装成统一的聊天请求对象交给同一套执行链路。

---

### Q288：聊天接口为什么既有同步接口又有异步派发接口？

**回答：**

同步接口适合测试、简单调试或短任务；异步派发接口更适合真实前端体验。

Agent 执行可能涉及模型、工具、RAG、地图等多个慢操作，如果同步等待，浏览器体验会差。异步接口先返回 session_id，前端再通过 SSE 看进度，这是更接近产品形态的设计。

---

### Q289：FastAPI 怎么实现 SSE？

**回答：**

SSE 接口返回 StreamingResponse，media type 是 text/event-stream。后端用异步生成器不断读取某个 session 的事件，如果有新事件就按 SSE 格式 yield 给浏览器；如果暂时没有事件，就发送 keep-alive 注释保持连接。

事件格式包括 id、event 和 data。id 用于断线续播，event 是事件名，data 是 JSON payload。

---

### Q290：SSE 接口为什么要发 keep-alive？

**回答：**

长连接如果长时间没有数据，中间代理或浏览器可能认为连接空闲而断开。keep-alive 可以维持连接活性。

而且 Agent 某些阶段可能在等模型或外部工具，几秒内没有业务事件是正常的。keep-alive 能让前端知道连接还活着，不要误判为失败。

---

### Q291：SSE 什么时候结束？

**回答：**

当收到 assistant.completed 或 session.failed 这类终态事件时，SSE 生成器会结束。  

如果当前 session 已经是 completed 或 failed，也不会无限等待；如果 session 不是 running，并且超过最大等待时间还没有新事件，也会结束连接。

---

### Q292：Last-Event-ID 和 last_event_id 参数有什么区别？

**回答：**

Last-Event-ID 是浏览器 EventSource 标准里重连时可以带的请求头；last_event_id 是项目额外支持的查询参数。

两者目的相同，都是告诉后端“我已经收到哪个事件了，请从下一个事件开始补发”。支持两种方式能让前端实现更灵活。

---

### Q293：FastAPI 路由是怎么拆分模块的？

**回答：**

按业务能力拆分：健康检查、机厅检索、地区列表、定位逆地理、聊天会话、知识库管理、SSE 流。每类能力有独立 router，最后统一挂载到 app 上。

这样比把所有接口塞到一个文件或一个 router 里清晰，后续扩展新功能也不容易互相影响。

---

### Q294：健康检查接口为什么不只返回 status=ok？

**回答：**

因为这个项目依赖很多外部和内部组件，只返回 ok 信息量太少。健康检查会返回数据源状态、RAG 状态、工具 provider 状态、MCP 状态和环境信息。

这样排查问题时可以快速判断：是数据没加载、RAG 没启用、MCP 工具没发现，还是地图配置有问题。

---

### Q295：FastAPI 中间件在项目里做了什么？

**回答：**

项目里有请求日志中间件，会为每个请求生成或透传 trace id，记录方法、路径、状态码、耗时和客户端信息，并把 trace id 放回响应头。

这样前端报错时可以拿 trace id 对应后端日志，定位一次请求完整链路。

---

### Q296：为什么要配置 CORS？

**回答：**

前后端分离时，前端开发服务器和后端 API 往往不是同源，比如前端在 5173，后端在 8000。浏览器会触发跨域限制。

FastAPI 通过 CORS 中间件允许指定前端 origin 访问 API 和 SSE，否则浏览器会直接拦截请求。

---

### Q297：配置为什么放到 Settings 里统一读取？

**回答：**

因为项目配置很多，比如数据源、Supabase、LLM、RAG、FAISS、高德、MCP、SSE、会话存储、CORS 等。统一 Settings 可以让配置入口清晰，也方便不同环境切换。

本地可以用 JSONL 和本地索引，线上可以切 Supabase、真实 LLM、真实地图 key，而不用改业务代码。

---

### Q298：为什么有些路径配置要解析成项目路径？

**回答：**

运行环境可能不同，命令启动目录也可能不同。如果直接用相对路径，容易在 Docker、测试、部署环境里找不到文件。

所以配置层会把相对路径解析到项目根目录或已有路径，保证会话文件、事件日志、RAG 索引、知识库目录等位置稳定。

---

### Q299：机厅列表接口具体怎么实现筛选？

**回答：**

接口接收关键词、店名、机种、省市区、是否有机台、排序字段、分页、用户经纬度等参数。FastAPI 先校验参数，再把参数传给数据仓库。

数据仓库返回原始行和总数，后端再做 DTO 映射和地理信息补齐，最终返回 items、page、page_size、total、total_pages。

---

### Q300：距离排序是怎么做的？

**回答：**

前端或 Agent 会传入用户坐标和坐标系，后端检索时根据门店坐标计算距离，并按 distance 排序。

如果是 Agent 场景，工具参数准备阶段还会从 working memory 里读取用户定位，自动补齐 origin_lng、origin_lat 和 origin_coord_system，用户不需要每次手动输入坐标。

---

### Q301：地区列表接口有什么作用？

**回答：**

地区接口给前端筛选器使用，比如省、市、区三级筛选。它优先从数据或地区服务里拿可用地区，让筛选项和真实机厅数据一致。

这样用户不会选到一个没有任何机厅数据的地区，筛选体验更稳定。

---

### Q302：逆地理编码接口具体做什么？

**回答：**

前端拿到浏览器经纬度后，把 lng、lat、accuracy_m 发给后端。后端调用高德逆地理服务，把坐标转换成省、市、区、街道、adcode 和格式化地址。

返回结果再进入聊天请求的 location 字段，Agent 就能理解“附近”具体指哪个位置。

---

### Q303：为什么高德 REST key 不直接放前端？

**回答：**

REST key 更适合服务端使用，直接放前端容易泄露和被滥用。项目里浏览器侧只负责拿用户定位和加载 Web 地图，逆地理、路线规划这类服务端能力由后端统一调用。

这样更容易控制额度、日志、重试和错误降级。

---

### Q304：知识库上传接口怎么防止路径穿越？

**回答：**

上传文件时会取安全文件名，删除或重试文件时会校验相对路径：不能是绝对路径，不能包含 ..，最终解析后的路径必须仍然位于知识库目录下。

这样可以避免用户通过构造路径删除或覆盖知识库目录外的文件。

---

### Q305：知识库上传后为什么返回 rag status？

**回答：**

上传成功不代表索引完成。返回完整 RAG 状态可以让前端立即刷新文件列表、pending 数量、indexing 数量、失败数量和 active_job_id。

用户能看到“文件已上传，正在索引”，而不是误以为上传后马上可以检索。

---

### Q306：知识库删除为什么也要触发重建或增量索引？

**回答：**

文件删除后，旧 chunk 不能继续留在向量索引里，否则用户还能搜到已经删除的内容。

所以删除文件后要把对应路径加入索引任务，索引服务会根据当前文件快照移除旧记录，并刷新向量索引和缓存。

---

### Q307：聊天历史接口具体返回什么？

**回答：**

列表接口返回会话摘要，比如 session_id、标题、预览、意图、状态、turn 数、创建时间和更新时间。详情接口返回完整 turns、reply、shops、route、client_location、destination、view_payload、map_artifact、last_stream_offset 等。

这样列表页加载轻，点进会话后又能恢复完整聊天和地图状态。

---

### Q308：会话标题和预览怎么生成？

**回答：**

标题通常取第一条用户消息的前一段内容；预览取最近一条用户或助手消息的短文本。

这是一个实用型设计，不依赖模型额外总结标题，成本低且稳定。后续如果要更智能，可以异步生成会话标题。

---

### Q309：为什么删除正在运行的会话会返回冲突？

**回答：**

因为后台 Agent 还在写 session、事件流和 memory。如果这时删除，可能出现任务继续写入一个已删除会话的问题。

所以删除前会检查运行状态，running 状态下返回冲突，让用户等任务结束或后续实现取消任务。

---

### Q310：FastAPI 如何处理附件大小？

**回答：**

接口读取上传文件后会检查大小，比如单个聊天附件限制在 10MB 内。超过限制直接返回 400。

这能保护后端内存和模型上下文，避免用户上传超大文件导致服务阻塞。

---

### Q311：聊天附件里的图片怎么进入模型？

**回答：**

图片会被识别为 image 类型，编码成 data URL，并作为附件信息放入聊天请求。后续 provider 层如果支持视觉模型，就可以把图片作为模型消息的一部分。

同时会保留 preview_text，让前端和历史记录能显示“已附带图片”。

---

### Q312：聊天附件里的文档怎么进入模型？

**回答：**

文档会先提取文本，比如 txt、md、json 直接解码，docx 解析文档 XML，PDF 提取前几页文本。然后做长度裁剪，作为 extracted_text 放入本轮请求。

这和知识库索引不同，它是一次性上下文，不会长期进入 RAG 索引。

---

### Q313：FastAPI 中 async def 和普通 def 怎么选择？

**回答：**

I/O 型、需要 await 的接口用 async def，比如异步派发、文件上传读取、SSE 流。普通快速读取或同步服务调用可以用 def，FastAPI 会在线程池中处理。

关键是不要在 async def 里做长时间阻塞操作，否则会卡住事件循环。项目里耗时 Agent 执行会派发到后台任务，SSE 用异步生成器输出。

---

### Q314：后台任务为什么用 asyncio.create_task？

**回答：**

因为接口需要快速返回 202 Accepted，不应该等 Agent 执行完成。create_task 可以让当前事件循环调度一个后台协程继续执行。

同时项目会保存 task 引用和 session 运行状态，任务完成或失败后清理 active session，避免状态一直卡在 running。

---

### Q315：这个后台任务方案有什么局限？

**回答：**

它适合单实例或轻量部署，但多实例下会遇到问题：任务在某个进程内执行，另一个实例未必知道；进程重启会中断正在跑的任务。

更生产化的方案是把 run 状态、事件 offset 和任务队列放到 Redis、数据库或专门队列系统里，由 worker 消费。

---

### Q316：FastAPI 如何支持前后端协议生成？

**回答：**

FastAPI 自带 OpenAPI schema。只要请求和响应都用 Pydantic 模型声明，就可以从 OpenAPI 生成 TypeScript 类型。

这个项目的思路是后端模型作为协议源头，前端消费生成类型，减少手写类型漂移。SSE 事件也可以用类似方式生成事件名和 payload 类型。

---

### Q317：Pydantic 在项目里解决什么问题？

**回答：**

Pydantic 用来定义 DTO、校验入参、序列化响应和兼容旧字段。比如路线 payload 可以兼容旧的 polyline 结构，地图 view payload 可以把 version 转成 schema_version。

它让接口协议显式化，减少“字典里有什么字段靠记忆”的问题。

---

### Q318：接口错误码怎么设计？

**回答：**

参数错误返回 400，资源不存在返回 404，并发冲突返回 409，正常异步派发返回 202，删除成功返回 204。

这种状态码能让前端明确区分：是用户输入错、会话不存在、正在运行不能操作，还是任务已经被接受。

---

### Q319：FastAPI TestClient 在项目里测试什么？

**回答：**

测试会构造临时数据和临时环境变量，启动一个隔离的 FastAPI app，然后验证 health、机厅查询、聊天接口、文件上传、地区接口、知识库上传和会话恢复等核心路径。

这类集成测试能覆盖“路由、配置、依赖容器、DTO、业务服务”是否能一起工作。

---

### Q320：测试里为什么要用临时 JSONL 数据？

**回答：**

临时 JSONL 数据可控、轻量，不依赖外部数据库，也不会污染真实数据。测试可以写入一条固定机厅数据，然后验证搜索、详情、聊天是否按预期返回。

这让测试更稳定，也适合 CI 环境。

---

### Q321：如果面试官问 FastAPI 相比 Flask 的优势，怎么结合项目回答？

**回答：**

我会说 FastAPI 更适合这个项目，因为它天然支持类型标注、Pydantic 校验、OpenAPI 文档、异步接口和 StreamingResponse。

这个项目有 SSE、后台任务、复杂 DTO、文件上传、外部 API 调用和类型生成需求，所以 FastAPI 的收益比较明显。

---

### Q322：如果面试官问 FastAPI 相比 Django 的区别，怎么回答？

**回答：**

Django 更偏完整 Web 框架，自带 ORM、后台、模板和很多约定；FastAPI 更轻量，适合 API 服务和异步 I/O。

这个项目前后端分离，后端主要是 API、SSE、Agent 编排和工具调用，不需要 Django 的模板和后台系统，所以 FastAPI 更合适。

---

### Q323：SSE 长连接部署时要注意什么？

**回答：**

反向代理不能对 SSE 开启普通响应 buffering，否则前端收不到实时事件；读超时时间也要调长，否则连接会被代理断开。

另外要注意 worker 数、连接数和 keep-alive 频率，避免大量空闲 SSE 连接占满资源。

---

### Q324：接口如何避免暴露敏感信息？

**回答：**

API key 和 service role key 都放在服务端环境变量里，不返回给前端。错误信息也要控制长度，不把完整异常栈或密钥拼进响应。

前端只拿必要数据，比如地图 Web key 可以是浏览器侧 key，但高德 REST、Supabase service role、LLM key 不应该暴露。

---

### Q325：FastAPI 里怎么做统一日志和排查？

**回答：**

每个请求有 trace id，中间件记录请求耗时和状态码；Agent 和工具层也记录 session_id、tool、call_id、耗时、失败原因。

排查时可以从前端错误里的 trace id 找到后端请求日志，再沿着 session_id 查 Agent 事件和工具调用。

---

### Q326：如果一个接口变慢，你怎么定位？

**回答：**

先看 access log 的耗时和 trace id，确认是哪个路径慢。然后看业务日志：是数据查询慢、RAG embedding 慢、地图 API 慢、LLM 慢，还是 SSE 一直等待。

如果是工具慢，就看 tool duration_ms；如果是 RAG 慢，就看索引状态、缓存命中率、向量后端和 reranker；如果是外部 API 慢，就考虑超时、缓存和降级。

---

### Q327：FastAPI 的 OpenAPI 文档对前后端协作有什么价值？

**回答：**

它能把接口路径、参数、请求体、响应体自动文档化。前端不用靠口头约定字段，也可以生成 TypeScript 类型。

对多人协作来说，这能减少“后端改字段前端不知道”的问题。

---

### Q328：如果要把这个 FastAPI 后端部署到生产，你会注意什么？

**回答：**

会注意环境变量、密钥管理、CORS 白名单、日志、健康检查、反向代理 SSE 配置、进程数、后台任务持久化、数据备份、超时和限流。

尤其是 Agent 类应用，一定要考虑外部模型和地图 API 失败时的降级，以及用户重复提交带来的幂等问题。

---

### Q329：FastAPI 接口和 Agent 工具层的边界是什么？

**回答：**

FastAPI 接口负责接收请求、校验参数、管理会话和返回响应；Agent 工具层负责把业务能力封装成可被模型调用的工具。

比如机厅列表接口是给前端直接查的，db_query_tool 是给 Agent 查的。底层可以复用同一个数据仓库，但入口和职责不同。

---

### Q330：如果面试官问“你最熟 FastAPI 哪些点”，怎么答？

**回答：**

可以结合项目回答：我熟悉 APIRouter 拆分路由、Depends 做依赖注入、Pydantic 做请求响应模型、Query/Form/File 做参数校验、StreamingResponse 做 SSE、lifespan 做启动预热、中间件做日志和 trace id、TestClient 做集成测试。

再补一句：我不是只会写 CRUD，而是用 FastAPI 支撑了 Agent 的异步会话、流式输出、知识库管理和地图服务集成。

---

### Q331：JSONL 本地数据源具体怎么实现检索？

**回答：**

启动时会把 JSONL 逐行读取，每一行解析成一个机厅对象。解析过程中会过滤坏行、校验必要字段、规范化机种列表，并为每家店构建一个搜索文本。

查询时不再每次读文件，而是在内存列表里按关键词、店名、机种、省市区、是否有机台等条件过滤，再排序和分页返回。这种方式适合本地开发和中小数据量，速度快、依赖少。

---

### Q332：JSONL 中坏数据怎么处理？

**回答：**

读取时会统计总行数、成功加载数和坏行数。空行、JSON 解析失败、缺少必要字段、source_id 不合法的行都会被跳过。

健康检查会暴露这些统计信息，所以数据质量问题可以在启动后被发现，而不是等用户搜索时才报错。

---

### Q333：关键词检索为什么要构建 search blob？

**回答：**

因为用户可能搜店名、拼音、地址、交通、评论、省市区、机种名、版本等多个字段。如果每次查询都临时拼接这些字段，会重复计算。

启动时把这些字段提前合并成小写 search blob，查询时只要判断关键词是否都在 blob 里，逻辑简单，也减少运行时开销。

---

### Q334：机种筛选怎么实现？

**回答：**

每家店有 arcades 列表，里面包含 title_name、quantity、version 等字段。筛选时会把用户输入的机种名做归一化，比如一些常见别名映射到同一标准名，然后判断店铺机种列表里是否存在该机种。

这样可以避免用户写“舞萌”“maimai”等不同叫法时搜不到。

---

### Q335：按某个机种数量排序怎么实现？

**回答：**

先把目标机种名归一化，然后遍历每家店的机种列表，把匹配机种的 quantity 加总。排序时按这个数量、更新时间、source_id 做稳定排序。

这比只判断“有没有该机种”更细，因为音游玩家常常关心某家店有几台机器。

---

### Q336：地区名称为什么要做归一化？

**回答：**

用户可能输入“上海”“上海市”“浦东”“浦东新区”，数据里也可能带有省、市、区、县等后缀。如果直接字符串相等，容易漏匹配。

所以会去掉常见行政区后缀和空白，再比较归一化后的名称。这样地区筛选更接近用户习惯。

---

### Q337：分页为什么放在后端做？

**回答：**

后端分页可以减少返回数据量，也能保证 total、total_pages、当前 page 的语义一致。前端只负责展示，不需要拿全量数据再自己切。

尤其后续切到 Supabase 时，分页更应该在数据库或后端做，否则数据量大后前端压力会很高。

---

### Q338：机厅详情接口和列表接口有什么区别？

**回答：**

列表接口返回摘要信息，适合展示卡片和地图点位；详情接口按 source_id 查询单家店，返回更多字段，比如评论、图片、活动、机种详情、原始数据等。

这样列表轻，详情全，避免列表页一次加载过多无用信息。

---

### Q339：地理坐标补全模块具体怎么做？

**回答：**

优先使用数据源里已有的坐标。如果数据里已经有 GCJ02 或 WGS84 坐标，就直接返回。没有坐标时，再看本地缓存；缓存也没有，才调用高德地理编码。

调用成功后会把结果写入缓存，下次同一家店就不需要重复请求高德。

---

### Q340：为什么地理解析要有缓存？

**回答：**

地理编码是外部 API，存在额度、延迟和失败风险。如果每次列表展示都去请求，会慢，也容易超限。

缓存可以把店铺名称、地址、城市生成指纹，配合 source_id 和更新时间作为 key。店铺地址没变时复用旧坐标；地址更新后 key 变化，再重新解析。

---

### Q341：地理缓存如何避免写坏文件？

**回答：**

写缓存时先写临时文件，再替换正式文件。这样即使写入过程中进程异常，也不容易留下半截 JSON。

同时缓存写入有锁，避免多个线程同时写导致文件内容互相覆盖。

---

### Q342：为什么地理解析只同步补全一部分？

**回答：**

列表页可能一次返回很多店，如果全部都实时地理编码，会拖慢接口并消耗大量外部 API。项目里会限制同步解析数量，只对前几条或有限条做补全。

这是一种体验和成本的平衡：优先保证页面能快速返回，再逐步补齐坐标。

---

### Q343：地图点位没有精确坐标怎么办？

**回答：**

如果有地址但没坐标，可以尝试地理编码得到近似坐标，并标记 precision 为 approx。前端展示时可以用不同提示告诉用户这是估算位置。

如果完全无法解析，就仍然展示文字地址和跳转导航入口，不强行在地图上打一个错误点。

---

### Q344：LLM Provider 为什么同时兼容 Responses 和 Chat Completions？

**回答：**

不同 OpenAI-compatible 服务支持的接口不完全一样。有些更接近 Responses API，有些只兼容 Chat Completions。

适配层会优先尝试配置偏好的接口，失败后再尝试另一种，把结果统一成 text、tool_calls、reasoning_items、response_id 这种内部结构。这样上层 Agent 不需要关心底层模型供应商差异。

---

### Q345：模型工具调用结果如何做统一抽象？

**回答：**

无论底层模型返回的是 Responses 里的 function_call，还是 Chat Completions 里的 tool_calls，适配层都会解析成统一的工具调用对象：call_id、name、arguments。

后面的工具注册表只处理这个统一结构，不需要分别适配不同模型格式。

---

### Q346：为什么模型返回的 arguments 要安全解析？

**回答：**

工具参数通常是 JSON 字符串，但模型可能输出空字符串、非法 JSON 或不完整结构。如果直接 json.loads 抛异常，会中断整条链路。

项目里会做安全解析，解析失败就转成空对象，再交给 schema 校验。这样错误会变成可控的 validation_error，而不是系统异常。

---

### Q347：为什么要处理 DeepSeek 这类兼容差异？

**回答：**

OpenAI-compatible 不代表完全一致。一些模型服务对 tool message、thinking 参数、图片输入、tool_choice 等字段要求不同。

适配层对这些差异做兼容，可以让项目更容易切换模型，不会因为换模型就大改 Agent 主流程。

---

### Q348：视觉模型路由怎么实现？

**回答：**

请求上下文里如果包含图片，Provider Router 会优先选择配置好的视觉模型；如果没有图片，就走普通文本模型。

如果用户传了图片但当前模型不支持视觉输入，系统会返回明确提示，让用户知道需要配置视觉模型，而不是直接报一个底层 API 错误。

---

### Q349：上下文构建模块具体负责什么？

**回答：**

它负责把系统提示词、子 Agent 提示词、技能说明、用户位置、附件信息、session 状态、working memory、最近工具结果和历史对话组装成模型可读的输入。

可以理解为：Agent 决策前看到什么，不是随便拼字符串，而是由上下文构建层统一整理。

---

### Q350：为什么上下文里要放 runtime state JSON？

**回答：**

因为模型需要理解当前执行状态，比如当前 session、turn_index、active_subagent、用户位置、是否已有 shops、是否已有 route、最近错误、worker 执行记录等。

用 JSON 表达比纯自然语言更稳定，模型也更容易从固定字段里读取上下文。

---

### Q351：如何控制传给模型的历史长度？

**回答：**

上下文构建时只取最近若干轮历史，数量由配置控制。工具结果也会做压缩和截断，只保留关键信息。

这样既能保留多轮对话能力，又不会因为历史太长导致 token 成本过高或超过上下文限制。

---

### Q352：为什么 worker 只看自己的 scoped turns？

**回答：**

worker 负责完成具体任务，不需要读完整会话里所有无关内容。让 worker 只看自己的任务和相关工具结果，可以减少噪音，也降低模型误判概率。

主 Agent 负责全局调度，worker 负责局部执行，这样上下文边界更清楚。

---

### Q353：附件信息怎么进入上下文？

**回答：**

上传文件后，系统会生成附件 metadata，比如文件名、类型、大小、预览文本、提取文本或图片 data URL。上下文构建时会把这些信息加入当前请求。

文本附件用于模型阅读，图片附件用于视觉模型处理，同时历史记录里也能看到用户上传过什么。

---

### Q354：工具结果为什么要压缩后放回模型？

**回答：**

工具结果可能很大，比如机厅列表、RAG 命中、路线 polyline。如果原样塞给模型，会浪费 token，也可能影响模型关注重点。

所以会保留核心字段和摘要，比如店名、地址、距离、机种数量、路线距离、耗时、fallback reason。完整结构化结果仍可给前端展示。

---

### Q355：摘要工具在项目里有什么作用？

**回答：**

摘要工具负责把结构化结果转成用户能读懂的最终回复。比如检索到多家店后，摘要会说明推荐哪几家、为什么推荐、距离和路线如何。

这样模型不是凭空回答，而是基于工具结果做表达层总结。

---

### Q356：为什么最终回复和结构化地图结果要分开？

**回答：**

文本回复给人读，结构化地图结果给前端渲染。两者用途不同，不应该混在一段自然语言里。

分开后，前端可以稳定展示地图点位、路线和卡片；模型文本可以更自由地解释推荐理由。

---

### Q357：路线规划模块怎么选择起点和终点？

**回答：**

起点优先来自用户浏览器定位或已解析的位置；终点来自用户指定的店铺、上一次候选店铺、或者导航 worker 解析出的目标。

如果缺起点，就提示用户提供位置；如果缺终点，就先做检索或目标解析。不会在信息不完整时硬调用路线 API。

---

### Q358：用户说“去第二家”怎么实现？

**回答：**

上一轮检索结果会保存在 working memory 的 shops 里。用户追问“第二家怎么走”时，Agent 可以从 shops 中取第二个候选作为 destination，再调用路线规划。

这就是 working memory 的价值：把上一轮结构化结果保留下来，让自然语言追问能落到具体对象上。

---

### Q359：如果用户只说店名，怎么做路线？

**回答：**

系统会先通过店名检索或地理解析找到候选店铺。如果能唯一匹配，就把该店作为终点；如果有多个同名或相似结果，就返回候选让用户确认。

路线规划需要明确终点坐标，所以不能只靠一个模糊名字直接导航。

---

### Q360：附近推荐如何利用用户定位？

**回答：**

前端先拿浏览器定位，后端逆地理补充城市和区县，然后把 location 放入聊天请求。工具参数准备阶段会从 memory 里读取用户坐标，自动把检索排序切换为 distance。

最终返回的店铺会带 distance_m，前端和助手回复都可以展示距离信息。

---

### Q361：如果用户拒绝浏览器定位，附近推荐怎么处理？

**回答：**

可以退化为让用户输入城市、商圈或地标。系统会先做地区筛选或位置解析，再按该位置周边推荐。

如果完全没有位置信息，就不能准确做附近排序，只能给出通用搜索结果并提示需要位置。

---

### Q362：RAG 检索工具如何接入 Agent？

**回答：**

RAG 被封装成一个知识检索工具，工具 schema 规定 query 和 top_k 等参数。Agent 判断需要非结构化资料时调用它，结果会写入 working memory 的 knowledge_hits。

后续 summary 阶段可以引用这些命中片段，补充机厅评论、活动、攻略等数据库没有的内容。

---

### Q363：知识库 lookup 接口和 Agent RAG 工具有什么区别？

**回答：**

lookup 接口是给前端或调试直接查知识库用的，返回命中文档、摘要片段和可能的机厅候选。RAG 工具是给 Agent 调用的，参与多步推理和最终回复。

底层检索服务可以复用，但入口对象不同：一个服务前端交互，一个服务 Agent 决策。

---

### Q364：为什么知识库命中还要提取 arcade candidates？

**回答：**

因为有些知识文档本身描述了新店、活动店或非结构化门店信息。如果只返回 snippet，前端无法在地图上展示。

提取候选后，可以把知识库里的店名、地址、城市、交通和坐标转换成结构化候选，必要时再地理编码，从而接入地图展示。

---

### Q365：历史会话恢复时如何恢复地图？

**回答：**

会话详情里不仅返回 turns，还返回 shops、route、client_location、destination、view_payload 和 map_artifact。前端进入会话时用这些结构恢复地图点位或路线。

这样页面刷新后不需要重新跑 Agent，也能看到上次的地图结果。

---

### Q366：前端如何区分候选地图和路线地图？

**回答：**

map_artifact 里有 scene 字段，比如 agent_candidates 表示候选点位，agent_route 表示路线展示。前端根据 scene 选择不同渲染逻辑。

同时 schema_version 用来处理协议演进，旧结构可以走 adapter 兼容。

---

### Q367：API 层为什么要有 client_id？

**回答：**

client_id 用来区分不同浏览器或用户的会话访问范围。查询、流式连接、删除会话时都可以带 client_id，后端只返回属于这个客户端的 session。

这不是完整登录权限系统，但能避免匿名场景下不同客户端的会话混在一起。

---

### Q368：session store 为什么要做文件锁？

**回答：**

会话状态可能被多个请求或后台任务读写，比如 SSE 记录 offset、Agent 保存 turn、会话详情读取状态。文件锁可以降低并发读写同一个持久化文件时损坏数据的风险。

如果后续多实例部署，更适合把它换成 Redis 或数据库。

---

### Q369：replay buffer 为什么要限制每个 session 的事件数量？

**回答：**

如果不限制，长会话或异常循环会让内存和事件日志不断增长。限制每个 session 保留最近 N 条事件，可以控制资源。

终态结果仍然在 session store 里，replay buffer 主要负责断线后补最近过程事件。

---

### Q370：为什么上传知识库后索引任务要有 job_id？

**回答：**

job_id 可以把文件状态和后台索引任务关联起来。前端看到某个文件 pending 或 indexing 时，可以知道它属于哪次任务。

失败时也能追踪是哪次上传、重试或删除触发的索引失败。

---

### Q371：FastAPI 中如何处理长耗时操作的超时？

**回答：**

接口层避免直接阻塞等待长任务，长任务要么异步派发，要么设置请求超时。工具层也有单工具 timeout，外部 HTTP client 有自己的 timeout。

这样即使 LLM、RAG、地图或 Supabase 变慢，也不会无限挂住请求。

---

### Q372：如果 SSE 客户端断开，后端怎么感知？

**回答：**

SSE 生成器会检查 request.is_disconnected。如果浏览器关闭页面或网络断开，后端检测到后就退出生成器，不再继续向这个连接写数据。

后台 Agent 任务本身不会因为某个 SSE 连接断开而停止，用户重连后可以继续按 offset 续播。

---

### Q373：为什么异步派发接口返回 202？

**回答：**

202 Accepted 表示请求已被接受，但处理尚未完成。这个语义很适合后台 Agent run。

如果返回 200，容易让人误以为结果已经完成；202 能明确告诉前端：任务已开始，请去 SSE 或会话详情里看后续状态。

---

### Q374：为什么删除成功返回 204？

**回答：**

204 No Content 表示删除成功但没有响应体。删除会话或知识文件时，前端只需要知道成功与否，不需要额外 JSON。

这是比较标准的 REST 语义。

---

### Q375：FastAPI 里如何做接口分层？

**回答：**

路由层只负责 HTTP 相关的事情，比如参数、状态码、响应模型；服务层处理业务，比如 RAG、地图、地理解析；Agent 层处理编排和工具调用；数据层负责 JSONL 或 Supabase 查询。

这样的分层能避免路由函数变成“大杂烩”。

---

### Q376：如果面试官问“这个项目是不是 MVC”，怎么答？

**回答：**

它不是传统模板渲染型 MVC，更接近前后端分离的 API 分层架构。FastAPI 提供 Controller 层接口，服务和 Agent 负责业务逻辑，数据仓库负责 Model 层数据访问，前端 Vue/React 负责 View。

所以可以借 MVC 理解，但实际更偏 API + Service + Agent Runtime + Repository。

---

### Q377：如何避免路由层和 Agent 强耦合？

**回答：**

路由层只构造标准请求 DTO，然后调用 orchestrator。具体 Agent 怎么调 worker、怎么执行工具、怎么写 memory，路由层不关心。

这样未来即使 Agent Runtime 从 ReAct 换成图编排或任务队列，HTTP 接口也可以尽量不变。

---

### Q378：如果后续要加用户登录，FastAPI 怎么扩展？

**回答：**

可以加认证依赖，比如 JWT 校验依赖，解析出 user_id。然后在会话、知识库、工具权限里都带上 user_id 做数据隔离。

现有 client_id 可以看作匿名阶段的轻量隔离，真正登录后应升级为用户级权限和资源归属。

---

### Q379：如果后续要加限流，应该加在哪里？

**回答：**

可以分三层：API 层按 IP 或用户限流，Agent run 层限制并发会话数，工具层按工具组限制调用次数。

比如普通接口可以用中间件限流，LLM 和地图工具则要做更严格预算，避免单个用户刷爆成本。

---

### Q380：如果面试官让你现场讲“用户问附近机厅”的完整实现，怎么答？

**回答：**

我会这样讲：前端先尝试获取浏览器定位，然后调用后端逆地理接口补齐城市和区县；用户发起聊天后，FastAPI 创建异步 session 并返回 202；前端打开 SSE；主 Agent 判断是附近推荐，派发搜索 worker；检索工具从 memory 读取用户坐标，自动按距离排序查询机厅；结果写回 working memory；summary 阶段生成自然语言推荐；后端通过 SSE 发阶段事件、工具事件、token 和完成事件；前端同时展示回复、进度和地图点位。

这条链路能体现前端定位、FastAPI、Agent、工具、数据检索、SSE 和地图展示是怎么串起来的。

---

### Q381：地区级联筛选具体怎么实现？

**回答：**

前端先请求省份列表，用户选中省份后再请求城市列表，选中城市后请求区县列表。后端接口只返回 code 和 name，前端用它们渲染级联筛选器。

后端地区服务优先从高德行政区接口获取全国标准数据；如果高德不可用，就回退到本地机厅数据里已有的省市区索引，保证筛选器至少可用。

---

### Q382：为什么地区服务要有“高德优先、本地回退”？

**回答：**

高德行政区数据更完整，适合展示全国范围筛选；但它依赖外部服务和 key，可能失败。本地数据虽然不完整，但和当前机厅数据一定匹配。

所以高德优先保证完整性，本地回退保证可用性。这个设计体现了外部服务不可用时的降级思路。

---

### Q383：行政区 code 为什么要标准化？

**回答：**

不同来源可能返回 6 位 adcode，也可能项目内部使用 12 位区域码。如果不统一，前端选中的地区 code 和数据库里的 code 可能匹配不上。

标准化后，地区筛选、数据查询和前端级联选择才能稳定联动。

---

### Q384：路线规划工具具体怎么调用高德？

**回答：**

路线工具会根据出行方式选择高德路线接口，比如驾车或步行。请求里带起点、终点和 API key。高德返回后，后端解析距离、耗时和每一步 polyline，把它转换成统一的路线 DTO。

前端不直接依赖高德原始返回，而是渲染后端整理后的 route 结构。

---

### Q385：高德路线 polyline 怎么处理？

**回答：**

高德返回的 polyline 通常是字符串，多个坐标点用分号分隔，经纬度用逗号分隔。后端会把它解析成点数组，每个点带 lng、lat、coord_system、source 和 precision。

这样前端画路线时不用再解析高德私有格式，只处理统一坐标点数组。

---

### Q386：路线规划离线估算怎么做？

**回答：**

如果高德路线接口不可用，后端用 haversine 公式计算起终点直线距离，再按出行方式估算耗时，比如步行速度和驾车速度不同。

这种结果不如真实路线准确，所以会带 hint 或 fallback_reason，告诉前端和用户这是估算值。

---

### Q387：为什么路线 fallback 也要返回 polyline？

**回答：**

即使只是估算路线，前端也需要有线段可以画。fallback 会用起点和终点组成一条简单 polyline。

这样地图仍然能展示“从哪到哪”的方向感，同时通过提示告诉用户不是精确导航路线。

---

### Q388：工具 JSON Schema 为什么要支持默认值？

**回答：**

模型调用工具时不一定每个参数都填全。比如 page 默认 1，page_size 默认 5，provider 默认 amap。schema 默认值可以在校验前自动补齐这些字段。

这样模型只需要提供关键参数，工具层负责补齐合理默认值，减少无意义的调用失败。

---

### Q389：JSON Schema 校验失败怎么反馈？

**回答：**

校验失败会返回结构化错误，包括错误类型、字段路径、错误说明、输入值和触发的校验规则。

这样日志里能看清楚到底是缺字段、类型错、枚举值不合法，还是范围不对。比直接抛一个模糊异常更利于调试。

---

### Q390：为什么工具权限不仅靠 prompt 约束？

**回答：**

prompt 只能引导模型，但不能作为安全边界。模型仍可能输出不该调用的工具名。

所以后端会在执行前检查 allowed tools。只有当前 Agent 或 worker 被允许的工具才能执行，这才是可靠的权限控制。

---

### Q391：MCP 工具是怎么被发现和注册的？

**回答：**

后端会读取 MCP server 配置，支持 HTTP、SSE、stdio 等形式。配置里可以包含远端地址、超时时间、是否启用、路线工具名等。

启动或刷新时，MCP 网关去发现远端工具 schema，再把这些工具转换成本地统一的工具描述，注册到工具表里。

---

### Q392：MCP 配置为什么支持环境变量占位？

**回答：**

像高德 key 这类敏感信息不应该写死在配置文件里。配置里可以写占位符，运行时从环境变量展开。

这样代码仓库里不会泄露密钥，不同环境也可以使用不同 key。

---

### Q393：如果 MCP 远端工具名变了怎么办？

**回答：**

可以在配置里手动指定路线工具名，或者在发现阶段根据工具描述挑选最像路线规划的工具。

同时健康检查会暴露 MCP 发现状态、可用工具和错误信息，方便排查“为什么 Agent 没有拿到地图工具”。

---

### Q394：子 Agent 配置为什么用 YAML overlay？

**回答：**

基础代码里有默认 Agent 配置，YAML overlay 可以在不改代码的情况下调整 prompt、allowed tools、skill files 和输出 schema。

这对 Agent 项目很有用，因为 prompt 和工具权限经常需要迭代。配置化后改动更轻，也更适合做版本管理和回滚。

---

### Q395：allowed_tools_mode 的 merge 和 replace 有什么区别？

**回答：**

replace 表示完全用 YAML 里的工具列表覆盖默认工具，适合严格控制某个 worker 只能调用指定工具。merge 表示在默认工具基础上追加，适合小幅扩展。

例如搜索 worker 可以 replace 成只允许检索、位置解析和知识检索，避免它误调用路线规划。

---

### Q396：query rewrite 在项目里做了什么？

**回答：**

它不是让大模型重写，而是用可控规则提取用户问题里的关键信息，比如机种、城市、区县、店名、地标和知识库查询词。

比如用户说“魔都徐汇附近有没有舞萌”，它可以提取 city 为上海、county 为徐汇区、title 为 maimai。这样后续工具参数更稳定。

---

### Q397：为什么 query rewrite 用规则而不是全交给模型？

**回答：**

机种别名、城市别名、常见地标这些是高频确定性规则，用规则更稳定、成本更低，也不会受模型波动影响。

模型仍然负责复杂意图判断和表达，但确定性抽取优先交给规则，可以提升整体可靠性。

---

### Q398：知识检索工具怎么自动补 query？

**回答：**

如果模型调用知识检索工具时没有给 query，工具参数准备阶段会从当前用户消息或 query rewrite 结果里补一个知识检索 query。

这样即使模型漏填了 query，也有机会恢复成可执行请求，而不是直接失败。

---

### Q399：db_query_tool 怎么自动补参数？

**回答：**

参数准备阶段会读取 last_request、query_rewrite 和 client_location。如果用户问附近，就自动把排序切换成 distance，并补起点经纬度；如果用户提到机种、城市、区县，也会补 title_name、city_name、county_name 等字段。

这相当于在工具调用前做一层“参数水合”，让工具调用更稳。

---

### Q400：FastAPI 文件上传为什么要分聊天附件和知识库文件？

**回答：**

聊天附件是短期上下文，只影响当前会话；知识库文件是长期资料，会进入索引，后续所有查询都可能用到。

所以二者接口分开、处理链路也不同。聊天附件重在快速提取预览和本轮可读文本；知识库上传重在文件状态、后台索引和检索质量。

---

### Q401：FastAPI 里如何处理 multipart 表单里的复杂字段？

**回答：**

multipart 表单里除了文件，还会带 session_id、client_id、message、intent、location 等字段。像 location 这种复杂对象会以 JSON 字符串传入，后端解析后再用 Pydantic 校验成位置对象。

这样既能支持文件上传，又能复用原来的聊天请求结构。

---

### Q402：为什么聊天附件要做文本长度裁剪？

**回答：**

上传文档可能很长，如果全部塞给模型，会导致 token 超限、成本高、回答慢。裁剪可以保证单轮请求可控。

如果用户需要长期、大量文档问答，应该走知识库索引，而不是聊天附件。

---

### Q403：FastAPI 如何表达“资源不存在”和“无权限访问”？

**回答：**

在匿名 client_id 场景里，如果会话不属于当前 client，接口可以返回 404，而不是明确告诉对方这个 session 存在但无权访问。

这样既符合资源不可见的语义，也减少信息泄露。

---

### Q404：为什么会话删除要先 snapshot 再 delete？

**回答：**

先 snapshot 可以确认会话存在且当前 client 有访问权限，再判断是否 running。只有满足条件才真正删除。

这样删除操作不会绕过权限检查，也不会删除正在运行的会话。

---

### Q405：FastAPI 的路由函数为什么尽量短？

**回答：**

路由函数应该只做 HTTP 层工作，比如参数接收、调用服务、处理状态码。复杂逻辑放到服务层和 Agent 层。

这样接口更容易读，也方便单独测试服务逻辑。否则一个路由函数会同时处理校验、业务、外部 API、状态写入，后期很难维护。

---

### Q406：如何解释项目里的 repository 模式？

**回答：**

repository 把数据访问封装起来。上层只调用 list_shops、get_shop、list_regions 等方法，不关心底层是 JSONL 还是 Supabase。

这样切换数据源时，上层 Agent、FastAPI 路由和前端协议都不用大改。

---

### Q407：为什么 Supabase 查询也封装成 repository？

**回答：**

因为 Supabase 只是数据源实现之一。封装后，JSONL 和 Supabase 都实现相同的数据访问接口。

本地开发用 JSONL，线上用 Supabase，业务代码仍然面对同一种抽象。

---

### Q408：如何处理 Supabase 请求失败？

**回答：**

Supabase RPC 返回 4xx 或 5xx 时，repository 会把状态码和响应摘要包装成运行时错误。上层健康检查或接口可以暴露失败信息。

在产品层可以进一步做降级，比如读本地备份数据，或者提示数据服务暂不可用。

---

### Q409：为什么机厅 DTO 要区分 Summary 和 Detail？

**回答：**

Summary 用于列表和地图，只包含常用字段；Detail 用于详情页，包含评论、图片、活动、机种明细和 raw 数据。

这样可以减少列表接口负载，也让前端组件职责更清楚。

---

### Q410：地图 artifact 的 view_payload 有什么作用？

**回答：**

view_payload 是给地图视图的额外展示元信息，比如场景标题、渲染模式或未来扩展字段。它和 route、shops 不同，不代表业务实体，而是代表“前端该怎么展示”。

把它放进 versioned payload，可以给地图渲染留扩展空间。

---

### Q411：后端为什么不直接返回前端组件需要的格式？

**回答：**

后端应该返回稳定业务语义，比如店铺、路线、位置、场景；前端组件格式可能随着 UI 变化而变化。

如果后端强绑定前端组件格式，UI 改动就会牵动后端。现在用 adapter 在前端适配，更利于演进。

---

### Q412：FastAPI 中如何组织 API tag？

**回答：**

每类 router 都有 tag，比如 chat、knowledge、arcades、regions、location、stream、health。OpenAPI 文档里会按 tag 分组。

这对联调很有帮助，前端或面试官看接口文档时能快速找到对应模块。

---

### Q413：如何解释这个项目不是简单 CRUD？

**回答：**

CRUD 只是数据增删改查，而这个项目有自然语言意图识别、Agent 编排、工具调用、RAG、地图路线、SSE 流式过程、会话恢复和工具治理。

FastAPI 在这里不是只写接口，而是承载 AI 应用运行时和多个外部服务的编排入口。

---

### Q414：FastAPI 的同步路由会不会阻塞？

**回答：**

普通 def 路由会由 FastAPI 在线程池里运行，适合短同步逻辑。但如果里面做很长的阻塞操作，也会占用线程资源。

所以长耗时任务要么 async 化，要么后台派发，要么放到独立 worker。这个项目里 Agent 长任务就通过异步派发和 SSE 解耦。

---

### Q415：如果 RAG warmup 很慢，会影响启动吗？

**回答：**

启动预热可以让服务提前发现索引和配置问题，但如果数据量很大，预热可能拖慢启动。

生产上可以选择懒加载、后台 warmup、或者使用已持久化的 FAISS 索引快速恢复。关键是健康检查要能反映 index_ready，而不是让用户盲等。

---

### Q416：为什么 health 要暴露 RAG 的 pending/indexing/failed 计数？

**回答：**

因为 RAG 不是一个简单开关。它可能启用了，但还有文件在排队、索引中或失败。

暴露这些计数可以帮助前端展示知识库状态，也方便运维判断“为什么搜不到刚上传的文档”。

---

### Q417：如何解释“工具执行结果持久化为 tool turn”？

**回答：**

每次工具调用后，系统会把工具名、call_id、参数、结果、trace、耗时、失败原因等记录到会话 turns 里。

这样后续模型可以参考最近工具结果，开发者也能回放这次 Agent 到底做了什么。

---

### Q418：为什么 tool turn 不一定直接展示给用户？

**回答：**

tool turn 是系统内部执行记录，内容可能很长或偏技术细节。用户更关心最终回答和简洁进度。

前端可以用 SSE 事件展示“正在检索”“路线已生成”，而不是把原始 JSON 工具结果全部展示出来。

---

### Q419：如果模型没有输出文本也没有工具调用怎么办？

**回答：**

运行时会检查模型响应。如果没有文本也没有工具调用，会继续下一步或走 fallback reply，根据 working memory 里的结果生成保底回复。

这能避免模型偶发空响应时用户完全没有反馈。

---

### Q420：summary 阶段如何避免编造？

**回答：**

summary 应该只基于 working memory 里的 shops、route、knowledge_hits 和 tool results 组织回答。没有数据就说明没有找到，不能凭空编店名或路线。

同时可以把“数据来自结构化库还是知识库”说清楚，降低用户误解。

---

### Q421：FastAPI 和前端 SSE 如何配合展示工具进度？

**回答：**

后端在工具开始前发 tool.started，完成后发 tool.completed，失败发 tool.failed；前端把这些事件映射成用户可读的进度文案。

这样用户能看到系统正在检索、解析位置、规划路线，而不是只看到一个转圈。

---

### Q422：如果工具执行失败，最终回答一定失败吗？

**回答：**

不一定。比如地图路线失败，但机厅检索成功，系统仍然可以告诉用户找到了哪些店，只是路线暂不可用。

只有关键步骤全部失败，或者没有足够信息生成回答时，才应该让会话失败或给出明确错误。

---

### Q423：如何向面试官解释“可降级”？

**回答：**

可降级就是某个依赖失败时，系统不直接崩掉，而是退到较低能力但仍可用的方案。比如地图路线失败退到直线估算，地区接口失败退到本地地区，结构化检索为空退到 RAG，视觉模型没配就提示用户。

这体现的是线上应用思维。

---

### Q424：如果用户上传恶意文档诱导 Agent 泄露密钥怎么办？

**回答：**

首先密钥不进入模型上下文，只存在服务端环境变量。其次工具权限由后端控制，模型不能越权调用工具。再次，RAG 内容只作为参考资料，不应该覆盖系统指令。

可以在 prompt 中明确文档内容不可信，并在工具层做权限校验。

---

### Q425：FastAPI 如何支持后续权限系统？

**回答：**

可以增加一个鉴权依赖，从请求头解析 JWT 或 session cookie，得到 user_id，然后把 user_id 注入路由和服务层。

会话、知识库文件、工具调用预算都可以绑定 user_id。这样从匿名 client_id 平滑升级到正式用户体系。

---

### Q426：如何做接口版本化？

**回答：**

可以在路径上加版本，比如 /api/v1，也可以在 DTO 里加 schema_version。对于地图 artifact 和 SSE payload 这种前端强依赖结构的协议，schema_version 很重要。

版本化的核心是新旧客户端能共存，后端升级不应该立刻打坏旧前端。

---

### Q427：如果要把后台任务迁移到队列，会怎么改？

**回答：**

接口层仍然返回 session_id 和 202，但不再用进程内 create_task，而是把 run 任务写入 Redis、数据库或消息队列。Worker 消费任务后执行 Agent，并把事件写入事件表或 Redis stream。

SSE 接口从事件存储里读取并推送。这样多实例部署和进程重启会更稳。

---

### Q428：FastAPI 部署时 worker 数怎么考虑？

**回答：**

如果主要是 I/O 型任务，可以适当增加 worker 和并发；但 SSE 长连接会占用连接资源，Agent 后台任务也会消耗模型和工具调用额度。

所以不能只看 CPU 核数，还要考虑连接数、外部 API 限额、LLM 并发和会话状态共享方案。

---

### Q429：这个项目哪些接口适合缓存？

**回答：**

地区列表适合缓存，因为变化很少；地理编码结果适合缓存，因为地址短期内不变；RAG 查询可以短 TTL 缓存；机厅列表可以按查询条件短缓存，但要注意数据更新。

聊天和 SSE 不适合简单缓存，因为它们和 session 状态强相关。

---

### Q430：如果面试官让你讲一个 FastAPI 里最有亮点的实现，怎么答？

**回答：**

我会讲异步会话加 SSE。用户请求先进 FastAPI，接口用 202 返回 session_id；后台 Agent 持续执行；SSE 用 StreamingResponse 推送 session.started、tool.started、assistant.token、assistant.completed 等事件；事件有 id，支持断线后按 Last-Event-ID 续播。

这个点能同时体现 FastAPI、异步编程、状态管理、AI 应用体验和工程稳定性。

我通过异步会话、事件 offset、working memory、状态机和协议类型生成，把这些状态收束起来，再用回归测试验证关键链路。
