# Arcadegent 项目说明文档（演示版）

## 1. 项目一句话介绍

Arcadegent 是一个面向音游机厅场景的 Agent 应用，把自然语言对话、机厅检索、地图展示、路线规划和知识库问答整合在同一个工作台里，让用户可以直接用一句话完成“找店、看店、去店、查资料”。

## 2. 这个项目解决什么问题

传统的机厅查询体验通常是分散的：

- 想找某个机厅，要自己搜列表。
- 想确认路线，要再跳去地图。
- 想了解补充信息，要再翻文档或群消息。

Arcadegent 把这些步骤串成了一条连续链路：

- 用户直接输入自然语言需求。
- Agent 判断是“检索机厅”“推荐附近”“规划路线”还是“结合知识库回答”。
- 后端调用对应工具与子 Agent。
- 前端实时展示执行过程、地图结果和最终回复。

## 3. 核心能力概览

### 3.1 Agent 对话

- 支持自然语言提问。
- 支持多轮会话与历史恢复。
- 支持实时展示执行阶段。
- 现在还支持上传文件和图片，把附件一起带入当前对话。

### 3.2 机厅检索

- 支持按机厅名、省市区、机种、排序方式筛选。
- 支持分页浏览。
- 支持查看门店详情、机台信息和地图定位。

### 3.3 地图与路线

- 结合高德地图展示门店位置。
- 支持候选门店列表、地图卡片和路线卡片。
- 支持一键跳转高德查看或导航。

### 3.4 知识库

- 支持上传 Markdown、TXT、JSON、JSONL、PDF、DOCX、DOC。
- 支持前端增删与批量删除。
- 支持 RAG 检索，把知识库片段作为问答补充依据。

## 4. 演示时建议怎么讲

可以按下面这条顺序演示，最容易让别人快速理解项目价值。

### 第一步：打开首页

重点说明：

- 这是一个“Agent 工作台”，不是单纯聊天框。
- 左边是导航和历史会话，右边是当前工作区。
- 三个主能力分别是 Agent 对话、机厅检索、知识库管理。

### 第二步：演示 Agent 对话

建议输入：

- “帮我找找适合下班后去的机厅”
- “从当前位置到最近机厅怎么走”

重点说明：

- 会话不是等全部完成后一次性返回，而是通过 SSE 持续推送阶段事件。
- 前端能看到当前在执行哪个子阶段。
- 结果不只是文本，还会带地图卡片和路线卡片。

### 第三步：演示上传附件对话

建议操作：

- 上传一份文档或一张截图。
- 再输入一句问题，例如“结合我上传的内容帮我总结重点”。

重点说明：

- 上传的文件和图片会跟这轮对话一起提交。
- 后端会把文件内容抽取成文本上下文，注入当前会话。
- 这说明系统不只是“能上传”，而是“上传的内容真的参与了回答”。

### 第四步：演示机厅检索页

重点说明：

- 这里是结构化检索入口，适合明确筛选条件的场景。
- 结果列表、地图点位和详情面板是联动的。
- 当数据库没命中时，还会有知识库与高德候选的补充回退逻辑。

### 第五步：演示知识库页

重点说明：

- 文档可以直接从前端上传。
- 上传后会自动更新索引。
- 支持单删和批量删除，便于维护 RAG 数据源。

## 5. 技术亮点

## 5.1 Agent 不是“黑盒回复”

这个项目的 Agent 不是简单把用户问题丢给模型，而是做了完整编排：

- `main_agent` 负责理解需求与决策。
- `search_worker` 负责检索。
- `navigation_worker` 负责路线与导航。
- 工具结果会写进 working memory。
- 最终再由主流程把结构化结果组织成用户可读回复。

## 5.2 前后端是实时协同的

- 后端先异步派发会话。
- 再通过 SSE 实时推送 `session.started`、`subagent.changed`、`tool.*`、`assistant.token` 等事件。
- 前端同步展示阶段变化，而不是只等最终结果。

这对 Agent 产品体验非常关键，因为用户能看到系统正在“做事”。

## 5.3 地图不是外挂，而是主结果的一部分

- 路线、候选门店、终点信息都会变成地图 artifacts。
- 前端根据结构化 artifacts 渲染地图卡片和操作按钮。
- 这样地图是业务结果的一部分，不是单独拼接的第三方页面。

## 5.4 RAG 是可维护的

- 支持多格式知识文档。
- 支持前端增删与批量删除。
- 支持本地 embedding fallback。
- 已经具备继续扩展到 reranker、hybrid search、FAISS 等工程化路径的基础。

## 5.5 上传附件能力增强了对话边界

新补上的上传文件和图片能力说明这个系统已经不再局限于纯文本问答：

- 文件可以转成可检索上下文。
- 图片可以作为会话附件进入本轮请求。
- 用户问题可以从“问一句话”升级成“带材料来提问”。

这对真实业务演示会很加分。

## 7. 系统架构简述

可以用一句话概括：

“前端负责交互和实时展示，后端负责会话、Agent、工具和知识库，地图与知识能力通过统一上下文进入同一轮回答。”

更细一点的链路是：

1. 前端发起聊天请求或异步会话派发。
2. 后端创建/恢复 session。
3. Agent 识别意图并决定是否调用 worker。
4. worker 执行检索、路线规划、知识检索等工具。
5. 结果写入 working memory。
6. SSE 持续把执行过程推回前端。
7. 前端渲染文本、地图卡片、候选列表与详情。

## 8. 演示时可以强调的工程价值

- 它不是单一页面，而是完整的前后端协同系统。
- 它不是普通聊天机器人，而是“模型 + 工具 + 地图 + 知识库”的任务型 Agent。
- 它不仅能回答，还能展示过程、结构化结果和导航动作。
- 它支持知识库维护和附件对话，说明系统边界是可扩展的。

## 9. 适合什么场景

虽然当前是音游机厅场景，但这个架构可以迁移到很多“搜索 + 地图 + 知识 + 对话”的垂直领域，比如：

- 门店导览
- 展馆与景点推荐
- 园区服务助手
- 本地生活路线问答

所以它的价值不只在音游场景本身，也在于这套 Agent 工作台架构可以复用。

## 10. 演示结尾可以怎么总结

可以用下面这段话收尾：

> Arcadegent 的重点不是“做了一个聊天框”，而是把自然语言对话、结构化检索、地图路线、知识库维护和实时过程展示整合成了一个完整的 Agent 产品原型。它既能展示业务价值，也能体现前后端协作、Agent 编排、RAG 接入和交互体验设计能力。
