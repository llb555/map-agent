## 30 秒项目介绍

Arcadegent 是一个面向音游机厅检索、Agent 问答和路线建议的全栈应用。前端使用 React + TypeScript + Vite，后端使用 FastAPI，核心能力包括机厅检索、会话式问答、SSE 流式返回、地图展示、地理解析、路线规划，以及通过内建工具和 MCP 工具扩展 Agent 能力。数据层支持本地 JSONL 和 Supabase 两种读模型，适合本地开发和线上部署两种模式。

---

## 2 分钟架构介绍

可以按下面这个顺序讲：

1. 整体上是前后端分离架构，前端在 `apps/web`，后端在 `backend`。
2. 前端负责聊天 UI、机厅浏览器、地图渲染和会话状态管理，主要通过 REST API 和 SSE 跟后端交互。
3. 后端分成 API 层、容器与配置层、Agent 运行时、工具系统、数据仓储层、地理服务层。
4. Agent 运行时采用 ReAct 风格，有主 Agent 和 worker/subagent 的职责划分。
5. 工具系统支持内建工具，也支持启动时动态发现 MCP 工具，并统一走 JSON Schema 校验和权限控制。
6. 数据层抽象成仓储接口，默认读取 JSONL，也能切到 Supabase RPC。
7. 地图能力分三层：前端高德 Web 地图渲染，后端高德 REST 做逆地理和 geocode，MCP 则提供额外路线工具能力。
8. 会话通过本地 JSON 文件持久化，流式事件通过 replay buffer + SSE 回放，满足“异步派发 + 实时展示”的体验。

### 2. 为什么这个项目适合做前后端分离？

考察点：

- 前后端边界划分能力

- 后端承担 Agent 编排、工具执行、数据访问、地理能力和外部服务调用，逻辑复杂且需要保护密钥。
- 分离后前端可独立优化体验，后端可独立扩展模型、工具和数据源。

### 4. 聊天为什么用 SSE，而不是 WebSocket？

考察点：

- 实时通信协议理解
- 是否会根据业务形态做取舍

建议回答：

- 这里的核心是服务端单向推送事件，例如 `assistant.token`、`tool.*`、`session.failed`。
- SSE 天然适合“服务端持续输出，客户端只消费”的场景，实现和调试成本都比 WebSocket 低。
- 项目还支持基于事件 ID 的 replay，便于重连和继续显示进度。

深度追问：

- SSE 和长轮询的区别是什么？
- SSE 在代理层、超时、浏览器兼容性上有什么坑？
- 如果后续需要双向低延迟通信，会怎么演进？

相关知识点：

- SSE 协议格式

### 5. 前端的状态是怎么组织的？

考察点：

- 前端状态建模能力

- 流式回复又通过 `useStreamReply` 做细粒度控制，降低 token 逐字刷新的抖动感。

### 7. 这个项目的后端分层是怎样的？

### 8. Agent 运行时是怎么设计的？

考察点：

- 是否理解 ReAct 风格执行模型
- 是否能把 Agent 从“调用大模型”提升到“可编排系统”

建议回答：

- 核心运行时是 `ReactRuntime`，负责会话状态、意图识别、上下文构建、工具调用、事件输出和最终回复。
- 外层还有 `Orchestrator`，主要做同步/异步派发和会话并发保护。
- 会话执行过程中会把信息沉淀到 `working_memory`，并通过 replay buffer 以事件形式不断推送给前端。

相关知识点：

### 9. 主 Agent 和 subagent 的职责怎么划分？

考察点：

- 多 Agent 或多角色协作设计

- `search_worker` 聚焦机厅检索。
- `navigation_worker` 聚焦地理解析和路线规划。
- 每个 subagent 都有自己的 prompt、skill 和 allowed tools，降低上下文污染和误用工具的风险。

- 为什么不用一个大 prompt 全做？

相关知识点：

### 10. 工具系统是怎么设计的？

### 11. 为什么要支持 MCP？

考察点：

- 对 Agent 扩展生态的理解

建议回答：

- 这样项目不需要把所有工具都硬编码在代码里，可以在启动时动态发现和挂载。
- 对这个项目来说，高德路线工具就是很典型的接入对象。

## 二面常见深挖题

### 13. 数据层为什么要同时支持 JSONL 和 Supabase？

考察点：

- 数据访问层抽象
- 开发环境与生产环境的差异化设计

建议回答：

- JSONL 模式适合本地开发、离线运行、快速验证。
- Supabase 模式适合线上共享数据和更稳定的读模型。
- 两者都通过仓储接口抽象，对上层 API 和 Agent 基本透明。

深度追问：

- JSONL 方案的瓶颈是什么？
- Supabase 为什么走 RPC，而不是直接表查询？
- 这两种模式如何保证返回结构一致？

相关知识点：

- Repository Pattern
- 读模型设计
- RPC 封装

### 14. 本地 JSONL 读模型的优点和局限是什么？

考察点：

- 对轻量数据方案的工程判断

建议回答：

- 优点是简单、透明、零额外基础设施、便于本地调试。
- 局限是并发、索引、复杂查询、增量更新和多实例共享都比较弱。
- 这个项目把它定位成 read model，而不是事务型主数据库。

深度追问：

- 如果数据量涨到几十万条怎么办？
- 搜索和排序在 JSONL 模式下如何优化？
- 为什么不直接上 SQLite/Postgres？

相关知识点：

- 文件型数据源
- 内存读模型
- 索引与查询复杂度

### 15. Supabase 模式为什么选 RPC，而不是把查询逻辑放后端？

考察点：

- 数据下推能力

建议回答：

- 复杂筛选、排序、分页、地区过滤等逻辑下推到数据库更合适。
- RPC 还能把数据库侧逻辑版本化，降低后端拼 SQL 的复杂度。
- 后端主要做协议整合、DTO 映射、地理补充和 Agent 能力整合。

深度追问：

### 16. 会话状态为什么持久化到本地 JSON 文件？

考察点：

- 状态持久化方案判断

建议回答：

- 会话状态并不只是聊天文本，还包括 `working_memory`、active subagent、错误状态、时间戳等。
- 这个方案满足“可恢复历史会话”和“重启后保留上下文”的需求，但也明确存在横向扩展限制。

### 17. replay buffer 是干什么的？

考察点：

- 实时流式架构理解
- 可靠性设计

建议回答：

- replay buffer 用来缓存某个 session 最近的事件流。
- 客户端断开后可以按 `last_event_id` 继续拿到后续事件，避免 UI 状态丢失。
- 这比只靠“当前 SSE 连接”更稳，也让前端重连逻辑简单很多。

深度追问：

- replay buffer 和 session store 的职责区别是什么？
- 为什么不直接把事件也全持久化到数据库？

相关知识点：

- Event Replay
- 流式恢复
- 临时状态与持久状态划分

### 18. 地理解析链路是怎么做的？

考察点：

- 地图/定位相关工程能力
- 多数据来源融合能力

建议回答：

- 优先读取 catalog 中已有坐标。
- 如果没有，就查本地 geo cache。
- 再没有则调用高德 geocode，并把结果写回运行时缓存。
- 前端地图展示主要消费统一 DTO，而不是直接消费底层数据源原始字段。

相关知识点：

### 20. 为什么需要 payload mapper？

考察点：

- DTO 设计与边界意识

建议回答：

- 上游 store 或 Agent working memory 的数据结构不一定适合直接暴露给前端。
- `ArcadePayloadMapper` 负责把原始行数据、route payload、client location payload 统一映射成公共 DTO。
- 这样可以把数据源细节、坐标补充逻辑和 API 响应解耦。

深度追问：

相关知识点：

## 系统设计与架构追问题

### 21. 如果并发量上来，这个项目最先会卡在哪里？

建议回答：

- 当前最明显的瓶颈是单实例内存态和本地文件态，包括 session store、replay buffer、JSONL 读模型和本地缓存。
- 其次是外部服务依赖，比如模型接口、高德 geocode 和路线服务。
- 再往后是 SSE 长连接管理和单实例 CPU/内存压力。

深度追问：

- 你会先改哪三处？
- JSONL 和本地会话文件分别怎么迁移？
- 如何做限流和熔断？

相关知识点：

- 横向扩展
- 有状态服务拆分
- 外部依赖保护

### 22. 如果要支持多实例部署，你会怎么改？

建议回答：

- 会话状态迁到 Redis 或数据库。
- replay buffer 迁到 Redis Stream、Pub/Sub 或专门事件总线。
- JSONL 读模型改成数据库或搜索服务。
- 通过 sticky session 或共享状态消除单机绑定。

深度追问：

- SSE 在多实例下如何保证连到正确节点？
- Redis 适合存哪些状态，不适合存哪些？
- 要不要引入消息队列？

相关知识点：

- 分布式会话
- Pub/Sub
- Sticky Session

## 可直接背诵的高质量回答模板

### 模板 1：为什么这样选型

我通常会从三个维度判断选型：当前业务复杂度、未来演进空间、团队实现成本。这个项目里前端交互比较复杂，但体量还没大到需要很重的工程化框架，所以选了 React + Vite + Zustand；后端需要异步 API、类型化 DTO、SSE 和外部 HTTP 调用，所以选了 FastAPI。整体目标不是追求最重的架构，而是在可维护和可扩展的前提下，用最合适的复杂度把系统搭起来。

### 模板 2：为什么做这层抽象

做这层抽象的原因不是“为了优雅”，而是因为这里存在明确变化点。比如数据源既可能来自 JSONL，也可能来自 Supabase；工具既可能是内建工具，也可能是外部 MCP 工具；模型既可能走 Responses API，也可能走 Chat Completions。把变化点抽象出来后，上层业务逻辑就不需要跟着每次切换一起改。

### 模板 3：这个方案的缺点是什么

这个方案的缺点我很清楚。它在当前阶段简化了实现，但代价是单实例状态比较重，包括 session store、replay buffer 和本地缓存。如果业务继续增长，我会优先把这些状态外部化，同时把 JSONL 读模型迁到更适合并发查询的持久层。也就是说，这不是完美方案，而是阶段性最优解。
